  3. Remember your credentials — you'll use them to log in to EAS later.

Already have an Expo account? You're good — skip ahead.

Apple Developer Account Apple $99/year

Required to submit to the iOS App Store and sign builds.

Walkthrough: Enroll in the Apple Developer Program
  1. Go to developer.apple.com/programs/enroll and sign in with your Apple ID (or create one).
  2. Choose your enrollment type:
    • Individual / Sole Proprietor — apps are listed under your personal name. Simpler signup.
    • Organization — apps are listed under your company name and you can add team members. Requires a D-U-N-S Number (free from Dun & Bradstreet, but can take 5+ business days to issue).
  3. Follow the enrollment steps. You'll need to agree to the Apple Developer Agreement and pay the $99/year fee.
  4. Enrollment can take up to 48 hours to process, so do this early.
Nonprofits & education: Accredited educational institutions and eligible nonprofits may qualify for a fee waiver. Check Apple's fee waiver page before paying.
Walkthrough: Create your app in App Store Connect
  1. Go to appstoreconnect.apple.com/apps and sign in with your Apple Developer account.
  2. Click the blue + button next to "Apps", then select New App (not App Clip or App Bundle).
  3. Fill in the New App form:
    • Platforms — check iOS
    • Name — your app's display name on the App Store (you can change this later)
    • Primary Language — your app's main language
    • Bundle ID — select the bundle ID you registered in the Apple Developer Portal (from the App Identity card). If you haven't registered one yet, do that first and come back here.
    • SKU — an internal reference ID (not visible to users). Use something like your app slug, e.g. mycommunityapp
    • User Access — select Full Access unless you have specific team restrictions
  4. Click Create. Your app now exists in App Store Connect. You'll need the App Store Connect ID later (found under General → App Information → Apple ID).
Don't worry about filling in everything now. You only need the app to exist at this point. Store listing details (screenshots, description, pricing) can wait until you're ready to submit for review.
Google Play Console Google $25 one-time

Required to submit to the Google Play Store.

Walkthrough: Register a Google Play Developer account
  1. Go to play.google.com/console/signup and sign in with your Google account.
  2. Choose your account type:
    • Personal — uses your personal information. Ideal if you don't have a registered business.
    • Organization — requires official business details. Lets you add team members to manage apps.
  3. Accept the Developer Distribution Agreement and pay the $25 one-time registration fee.
  4. Complete your developer profile (name, address, contact info). Google may take a few days to verify your identity.
New account testing requirement: Google now requires new developer accounts to run a closed test with at least 20 testers for 14 consecutive days before you can submit to production. Register your account and start testing early to avoid delays at launch time.
Walkthrough: Create your app in Google Play Console
  1. Go to play.google.com/console and sign in with your Google Play Developer account.
  2. Click Create app on the All Apps page.
  3. Fill in the basics:
    • App name — your app's display name on the Play Store
    • Default language — your app's main language
    • App or Game — select App
    • Free or Paid — select Free (this cannot be changed later)
  4. Check the Developer Program Policies and US export laws declarations, then click Create app.
  5. Your app now exists in the Play Console. The package name gets locked when you upload your first build — make sure it matches the Android Package you set in the App Identity card.
Package name tip: Google Play doesn't ask for a package name during app creation — it's determined by your first uploaded AAB file. The package name comes from the Android Package field in the dashboard. Just make sure you don't change it after your first upload.

Optional Services

These are only needed if you use the features they power. The app works fine without them.

Sentry Sentry Free tier (5,000 errors/month)

Crash reporting — captures every native iOS/Android crash, JS exception, and React render error from any user's device and surfaces them in a web dashboard with full stack traces, breadcrumbs, device info, and OTA channel. You no longer need a Mac, Xcode, or physical device access to investigate iOS crashes. Skip if you're fine relying on App Store Connect / Google Play Console crash reports (which are slower, less detailed, and platform-specific).

Walkthrough: Create a Sentry account and project
  1. Go to sentry.io/signup and create a free account.
  2. Choose the Developer (Free) plan — 5,000 errors/month is plenty for most communities.
  3. When prompted to create a project, choose React Native as the platform.
  4. Sentry will show you a setup screen with installation commands — ignore those, the SDK is already integrated. You only need the DSN.
  5. After the project is created, go to Project Settings → Client Keys (DSN) and copy the full DSN URL (starts with https:// and contains @ and .ingest.sentry.io).
  6. In your WordPress admin, go to TBC Community App → Crash Reporting, paste the DSN, check Enable Crash Reporting, and save.
  7. Force-quit and relaunch the app on your test device twice (one-time bootstrap — explained in the Companion Plugin → Crash Reporting Tab section).
  8. Long-press the header logo for 2 seconds to open the dev debug menu, then tap Send test event to Sentry. Within 30 seconds the event should appear in your Sentry dashboard's Issues feed.
Completely optional. The Sentry SDK ships in every app build but stays dormant until you give it a DSN. With no DSN configured, zero data is sent and there is no performance cost. Source-map upload (for readable function names in production stack traces) is also optional — see setup/docs/logging.html#sentry for the one-time EAS secrets setup.

Set Up Your Project

Download & Extract

After purchasing, you'll receive a .zip archive (e.g. core-update-3.1.0.zip). Extract it to a folder on your computer and name it after your community — something like MyCommunityApp or FitnessTribeApp. This is your project root.

Open in VS Code

Open VS Code, go to File → Open Folder, and select your project folder. Open the built-in terminal (Terminal → New Terminal or Ctrl+`).

Claude Code — your AI co-pilot

If you installed the Claude Code extension, open it from the sidebar. It reads your project's CLAUDE.md and understands the full codebase — ask it to help with setup, build modules, or debug issues.

Launch the Dashboard

If you're using Claude Code, just ask it to "launch the dashboard". Otherwise, open a terminal in your project folder and run:

npm run dashboard

The dashboard opens http://localhost:3456 in your browser — your setup command center. It reads your actual config files, lets you edit everything in a browser UI, and validates your setup live. Work through each tab from left to right.

Keep the dashboard running while you work through setup. When you're done, click the stop button in the header, or press Ctrl+C in the terminal.
Install Dependencies

If node_modules is missing, the dashboard shows a yellow banner with an Install Dependencies button. Click it — this runs npm install for you. Once complete, the banner disappears and you're ready to configure.

Troubleshooting

npm run dashboard won't start

Port 3456 is likely still in use from a previous session. Run npm run kill-dashboard to free the port, then try again.

npm install fails

Make sure you're running Node.js 18+ (check with node -v). If the error persists, try deleting node_modules and package-lock.json, then run npm install again.

Dashboard opens but shows errors

Make sure you extracted the project files completely and didn't accidentally nest folders (e.g. MyApp/MyApp/). The dashboard expects package.json in the root of the folder you opened.

Config Tab

The Config tab is where you make the app yours. Work through each card top to bottom.

Save as you go. The floating Save Changes bar appears whenever you have unsaved changes. Click it any time — the dashboard writes your config to the right files (app.json, eas.json, app.config.ts, constants/config.ts, package.json) in one go. You don’t need to wait until the end — save after each card if you like.

Walkthrough: Set up the App Store Connect API Key (required for one-click submissions)

Without an API key, the dashboard's Submit to App Store button fails on a fresh setup with the error App Store Connect API Keys cannot be set up in --non-interactive mode. The API key lets EAS upload your IPA without prompting for an Apple password or 2FA code — this is what makes the dashboard's one-click submit work end-to-end.

Heads up — this is different from eas credentials --platform ios. That command sets up build credentials (signing cert + provisioning profile, used by eas build to compile and sign the IPA). The API key on this page is for submission (used by eas submit to upload the signed IPA to App Store Connect). You need both for a fresh setup, but they are completely separate Apple credentials.
Generate once, reuse everywhere. The .p8 file (and the Key ID + Issuer ID that go with it) is per Apple Developer team — not per app. If you have multiple apps under the same Apple account, you only ever need to generate one API key — then paste the same Key ID, the same Issuer ID, and drop the same .p8 file into each project's dashboard. One key rules them all. Save the original .p8 somewhere safe outside any project folder (e.g. 1Password or an encrypted USB) since Apple won't let you re-download it after the initial generation — if you lose it, you'd have to revoke and regenerate, then re-upload to every project.

Open App Store Connect API page →

  1. Sign in to App Store Connect → Users and Access → Integrations → App Store Connect API. (If the link drops you on the homepage, click your name → Users and Access → the Integrations tab → App Store Connect API in the left sidebar.) You must be the Account Holder or an Admin on the team to see this page.
  2. Copy the Issuer ID from the very top of the page — it's a UUID like 69a6de70-.... This is the same for every key your team ever generates, so once you have it, you have it forever.
  3. Click the  +  button next to Active to generate a new key.
  4. Name it something memorable like Setup Dashboard or EAS Submit.
  5. Set Access to Admin. (App Manager works too, but Admin is the safest bet for submission.) Click Generate.
  6. The new key appears in the table. Copy the Key ID — it's the 10-character string in the “Key ID” column (e.g. ABC123XYZ4).
  7. Click Download API Key on the right side of the row. This downloads a file named something like AuthKey_ABC123XYZ4.p8.
    Apple only lets you download the .p8 once. If you close the page or lose the file, you can't download it again — you'd have to revoke the key and generate a new one. Save it somewhere safe immediately (a password manager works great).
  8. Back in the dashboard → Config tab → App Store Submission:
    • Paste the Key ID into the App Store Connect API Key ID field.
    • Paste the Issuer ID into the App Store Connect Issuer ID field.
    • Drag the .p8 file onto the drop zone (or click and pick it). The dashboard saves it to ./asc-api-key.p8 at your project root and writes the path into eas.json automatically.
  9. The status row flips from red “No API key” to green “API key uploaded”. You're done.
  10. Test it. Go to the Builds tab, find a finished iOS production build, and click Submit to App Store. The submission should run end-to-end with no prompts and land in TestFlight within a few minutes.
Where it lives: The .p8 file sits at the root of your project as asc-api-key.p8. It's automatically excluded from git (the *.p8 rule in .gitignore) and from EAS uploads, so it never leaves your machine. If you ever rotate the key, just upload a new .p8 from the same drop zone — the old one is overwritten. Remember: you can drop the same file into other projects under the same Apple team (see “Generate once, reuse everywhere” at the top of this walkthrough).
Walkthrough: Set up iOS Build Credentials with eas credentials (first iOS build only)

Before EAS can build for iOS, your Apple Developer account needs a distribution certificate and a provisioning profile for your app's bundle ID. EAS handles both through one interactive command — you only need to run it once per app.

This is different from the App Store Connect API Key above. Build credentials sign the IPA so iOS will install it; the API key authenticates the upload to App Store Connect. You need both for a fresh setup, but they're separate Apple credentials with separate setup flows.

How you'll know you need this: If you click an iOS build button in the dashboard and the build fails with “iOS credentials aren't set up yet,” that's the trigger. The iOS Credentials field in the App Store Submission card has a one-click Copy button for the command below.

  1. Open a terminal in your project root and run: 
    eas credentials --platform ios
  2. Pick “Build Credentials” from the menu (not Push Notifications or App Store Connect — Build Credentials is the one that sets up signing for eas build).
  3. Pick “All” when asked what to do — this sets up the distribution certificate AND the provisioning profile in one pass.
  4. Sign in to Apple when prompted — your Apple ID email, password, and a 2FA code from your phone or trusted device.
  5. Distribution certificate: if you already have other apps under the same Apple Developer account, EAS will offer to reuse an existing certificate. Always pick yes — Apple limits you to 3 distribution certs at a time, so reusing is the right move. If this is your first app, EAS will generate a fresh one for you.
  6. Provisioning profile: pick yes to generate a new one. Profiles are scoped per-bundle-ID, so this doesn't affect any other apps you have on the same account.
  7. Device registration: only matters for development and preview profiles (those install directly to phones, not via the App Store). If you have your iPhone handy, pick yes — EAS gives you a URL to open on your phone that auto-registers it. Otherwise pick no, and either:
    • Run eas device:create later when you have a device to register, or
    • Use the production profile instead — production builds go to the App Store, not directly to phones, so they don't need any registered devices.
  8. Done. EAS prints “All credentials are ready to build your project.” Pick “Go back” then “Exit” to leave the menu.
  9. Back in the dashboard, click the iOS build button again — this time it should succeed and the build will queue on EAS.
Multi-app accounts: If you build multiple apps under the same Apple Developer account, each app needs its own provisioning profile (scoped per bundle ID), but they can all share the same distribution certificate. EAS will detect existing certs and offer to reuse them — always pick reuse to avoid burning your 3-cert limit.
Where the credentials live: Unlike the .p8 API key, build credentials are stored on EAS's servers, not in your project folder. There's no local file to back up — if you switch machines, just run eas login on the new machine and EAS pulls them automatically.
App-Specific Password (legacy fallback): If you skip the API key setup above, EAS falls back to prompting for an app-specific password on every submit, or you can set EXPO_APPLE_APP_SPECIFIC_PASSWORD as an environment variable. The API key approach is strongly recommended — it's what unlocks the dashboard's one-click submit.

Push Notifications

Requires: Firebase

Push notifications require a Firebase project with two config files — one for Android and one for iOS. The dashboard shows a red indicator for each missing file and a green checkmark once uploaded.

1. Create a Firebase project
  1. Go to console.firebase.google.com and sign in with your Google account.
  2. Click Create a project (or Add project).
  3. Enter a project name — use your app name (e.g. “My Community App”). Firebase will generate a project ID below the name field automatically.
  4. On the Google Analytics step, disable the toggle. The app only uses Firebase for push notifications — Analytics is not needed and adds unnecessary complexity.
  5. Click Create project and wait for it to finish provisioning.
2. Add the Android app
  1. From your Firebase project dashboard, click Add app and select the Android icon.
  2. For Android package name, enter the same package name you set in the App Identity card (e.g. com.company.appname). This must match exactly.
  3. For App nickname, enter your app name followed by “Android” (e.g. “My Community App Android”). This helps you tell the two apps apart in Firebase later.
  4. Click Register app.
  5. On the next step (Download and then add config file), click Download google-services.json. Save this file somewhere you can find it.
  6. Click Next to skip through the remaining steps (Add Firebase SDK and Next steps), then click Continue to console. The app already has the Firebase SDK built in — you do not need to edit any Gradle files or add any code.
3. Add the iOS app
  1. Back on the project dashboard, click Add app again and select the iOS icon.
  2. For Apple bundle ID, enter your iOS bundle identifier from the App Identity card (e.g. com.company.appname). This must match exactly.
  3. For App nickname, enter your app name followed by “iOS” (e.g. “My Community App iOS”).
  4. For App Store ID, enter your numeric Apple ID if you already have one — it’s the same App Store Connect ID from the previous step. Find it in App Store Connect → your app → General → App Information → Apple ID. If you haven’t created your app in App Store Connect yet, leave this blank — you can add it later.
  5. Click Register app.
  6. On the next step, click Download GoogleService-Info.plist. Save this file.
  7. Click Next to skip through the remaining steps, then click Continue to console. Same as Android — the SDK is already built into the app.
4. Upload to the dashboard
  1. Back in the setup dashboard, find the Push Notifications card.
  2. Drag and drop google-services.json into the Android upload area. The indicator will turn green with a checkmark.
  3. Drag and drop GoogleService-Info.plist into the iOS upload area. The indicator will turn green with a checkmark.
5. Enable Cloud Messaging
  1. In the Firebase console, click the gear icon next to “Project Overview” and select Project settings.
  2. Click the Cloud Messaging tab.
  3. Look for Firebase Cloud Messaging API (V1) — it should show Enabled with a green checkmark. If it’s already enabled, you’re done.
  4. If it shows Disabled, click the three-dot menu (⋮) on the right side of that row, then click Enable. This may take you to the Google Cloud Console — click Enable there and return to Firebase.
What about the rest of this page? You’ll see sections for APNs Authentication Keys, APNs Certificates, and Web Push certificates — leave these all blank. The app sends push notifications through Expo’s push service, which handles the APNs (Apple) delivery using its own credentials. Firebase is only used for Android delivery via FCM. You also don’t need the Cloud Messaging API (Legacy) — that’s the old deprecated API.
Both config files are required. The app needs google-services.json (Android) and GoogleService-Info.plist (iOS) for push notifications to work on both platforms.

Troubleshooting

EAS login keeps failing

Try running eas logout first, then eas login again. VPNs can sometimes interfere with Expo's auth — try disconnecting temporarily. Make sure you're using your Expo account credentials (not Apple or Google).

Can't find my App Store Connect ID

In App Store Connect, go to your app → General → App Information. The Apple ID field (a numeric string like 1234567890) is your App Store Connect ID.

Push notifications not working after setup

Both Firebase config files are required — GoogleService-Info.plist (iOS) and google-services.json (Android). Also verify that Firebase Cloud Messaging API (V1) is enabled in your Google Cloud Console. Check the Companion Plugin → Push Log tab for error details.

Save button doesn't appear or changes don't stick

The floating save bar only appears when you make changes. If changes aren't persisting, check your browser console for errors. Try a hard refresh (Ctrl+Shift+R) or a different browser.

Crash Reporting Tab

The Crash Reporting tab connects the mobile app to your Sentry account. When enabled, every crash, JS exception, and React render error from any user's device flows into your Sentry dashboard automatically — with full stack traces, breadcrumbs, device info, app version, and OTA channel attached. You no longer need physical device access or App Store Connect crash logs to investigate iOS-specific bugs.

FieldDescription
Enable Crash ReportingMaster switch. When unchecked, the Sentry SDK stays dormant — zero data sent, zero performance cost, even though the SDK ships in every app build.
Sentry DSNYour project's write-only public key. Format: https://<publicKey>@<host>/<projectId>. The plugin validates the format on save. Leave blank to disable. The DSN is sent to the app over the existing /app-config endpoint and cached in MMKV — there is no per-call configuration.
Free tier is plenty. Sentry's free plan gives you 5,000 errors/month, 1 user seat, 30-day retention, email alerts, and source-map upload — enough for most communities indefinitely. Sign up at sentry.io, create a "React Native" project, and copy the DSN from Project Settings → Client Keys (DSN).

Two-relaunch bootstrap (one-time)

Sentry must initialize at module-load time, before React mounts, so that errors during app startup are captured. The init reads the DSN synchronously from a local cache (MMKV). The first launch after saving a new DSN fetches the config and writes it to the cache; only the second launch initializes Sentry from that cache. After this one-time bootstrap, Sentry initializes instantly on every cold start. To verify, long-press the header logo for 2 seconds to open the dev debug menu — the "Crash Reporting (Sentry)" section shows whether the SDK initialized this session, plus a "Send test event to Sentry" button.

Privacy

Sentry receives crash stack traces, device info (model, OS version), app version, and recent log breadcrumbs. It does not see user content, messages, passwords, or auth tokens — those are stripped automatically before transmission. Source-map upload for readable stack traces in production is optional — see setup/docs/logging.html#sentry for the EAS secrets setup.

To disable: uncheck Enable Crash Reporting or clear the DSN field. Either disables Sentry on the next app launch with zero performance cost — the SDK stays in the binary but never initializes.

Troubleshooting

Deep links not working

Verify the .well-known verification files are accessible over HTTPS without redirects. Check that your CDN or caching plugin isn't blocking them. After changing deep linking settings, users may need to reinstall the app for OS-level link association to update.

Maintenance mode not blocking users

Check the Bypass Roles setting — the user's role may be in the bypass list. Also check that the feature is enabled and settings are saved. The app reads maintenance status on each launch.

Smart App Banner not appearing

Requires Deep Linking to be configured with at least the App Name, Bundle ID (iOS), URL Scheme, and App Store ID filled in. The banner only shows on community pages served by Fluent Community.

Push notifications not sending

Check the Push Log tab for errors. Verify Firebase is configured correctly (both platform config files uploaded). Make sure the notification type is enabled on the Notifications tab. Users must have the app installed with push permissions granted.

Feature changes not taking effect

Feature flags are fetched on app launch. Users need to fully close and reopen the app to pick up changes. If you just saved settings, wait a moment for any server-side caching to clear.

Builds Tab

The Builds tab is where you build, submit, and publish your app. It's a thin wrapper around EAS (Expo Application Services) — the tab reads your version, lets you bump it, kicks off native builds for iOS and Android, and lists every build EAS has ever made for this project with one-click actions (download, install, submit).

Everything configured in the Config tab (app identity, EAS project ID, Apple/Google submission keys, Firebase, branding) feeds into the Builds tab. If a build fails because something's missing, the error will point you back to the Config card that needs fixing.

1. Bump the Version

The top of the New Build card shows your current App Version with three bump buttons: Major, Minor, and Patch. Clicking any of them updates four files at once:

  • package.json → version
  • app.json → expo.version
  • app.json → expo.ios.buildNumber (iOS requires this to be unique per upload)
  • app.json → expo.android.versionCode (Play Store requires this to increase every upload)
Always bump before a new production build. Both stores reject uploads with a duplicate version number. If you built 3.5.0 yesterday and try to submit another 3.5.0 today, the store will error out with “version already exists.” Bump first, build second, submit third.

Which bump to pick? Patch for bug fixes (3.5.0 → 3.5.1). Minor for new features that don't break anything (3.5.0 → 3.6.0). Major for breaking changes or a big redesign (3.5.0 → 4.0.0). When in doubt, patch.

2. Pick a Profile and Build

Below the version bumper, the New Build card splits into two platforms — iOS (Build for iPhone / iPad) and Android (Build for Android devices) — each with profile buttons. 4. Submit to the Store

Once a production build finishes, the Submit to App Store / Submit to Play Store button appears on that row. Clicking it uploads the binary to the store without another build — no re-compilation, no queue wait.

iOS — Submit to App Store walkthrough

Prerequisite: the App Store Submission card in the Config tab must have the App Store Connect API Key ID, Issuer ID, and .p8 file set. Without those the button will show Submit (not configured).

  1. Find a finished iOS production row in Recent Builds and click Submit to App Store.
  2. EAS uploads the IPA directly to App Store Connect using the API key — no Apple ID password prompt, no 2FA. This usually takes 2–5 minutes.
  3. The row updates to Submitted to App Store · [timestamp] when it's done. The build is now in App Store Connect — head to appstoreconnect.apple.com to continue.
  4. TestFlight (first stop). The build shows up in TestFlight → Builds within a couple minutes — usually with a yellow “Missing Compliance” banner. Click the build, answer the Export Compliance question (the app uses HTTPS only — select Yes → exempt), and Apple will process it for TestFlight.
  5. Add internal testers. Under TestFlight → Internal Testing, add yourself and any team members. They get an email invite — install the TestFlight app from the App Store, accept the invite, and the build downloads immediately (no Beta App Review for internal testers).
  6. Test thoroughly. At minimum: cold-launch, log in, post something, receive a push notification. When satisfied, move on to submission.
  7. Submit for App Store review. Under the App Store tab, create a new version (matching the version number you just built), attach the TestFlight build, and fill in the required fields (see Publishing Checklist below). Click Add for Review → Submit for Review.
  8. Wait for review. Apple review takes 24–48 hours typically. You'll get an email when it's approved (or rejected — see Troubleshooting).
  9. Release. Once approved, the app goes live automatically (if you chose Automatically release) or stays in Pending Developer Release until you hit the release button manually.
External testing (optional). To let non-team-members test via a public link, add them as External testers in TestFlight. External builds require a Beta App Review (~24 hours) on the first build, but subsequent builds in the same major version skip review.
Android — Submit to Play Store walkthrough

Prerequisite: the App Store Submission card must have the Google Play service account key (JSON) uploaded, and the track (internal or production) set. Without those the button will show Submit (not configured).

  1. Find a finished Android production row in Recent Builds and click Submit to Play Store.
  2. EAS uploads the .aab bundle to Google Play Console via the service account key — no login prompt. Usually takes 1–3 minutes.
  3. The build lands in Google Play Console on whichever track you configured.
  4. Internal testing first. Under Testing → Internal testing, create a new release from the uploaded build, add tester email addresses (up to 100), and share the opt-in link. Testers accept the invite and install via Google Play — no review delay.
  5. Closed testing (if this is a new developer account). Before you can create a production release, Google requires at least 20 opted-in testers for 14 consecutive days in closed testing. Start this early — it's the #1 cause of launch delays for new accounts.
  6. Promote to production. Once testing is done, go to Production → Create new release, promote the tested build, fill in the release notes, and click Review release → Start rollout to production. Google review typically takes 1–3 days on a first release.
  7. Once approved, the app appears on the Play Store. You can optionally roll out gradually (e.g. 20% → 50% → 100%) from the same release page.
New Play Console accounts must complete closed testing first. The 20-tester / 14-day requirement applies to any developer account created after late 2023. If you're launching under an established account (existing apps in production), this doesn't apply — you can go straight to production.
CLI equivalents: the Submit buttons just run eas submit --platform ios --latest or eas submit --platform android --latest under the hood. You can run these from the terminal if you ever need to.
5. Publish — Store Listing Checklist

eas submit uploads the binary, but each store still needs a listing (description, screenshots, privacy policy, content ratings) before the app goes live. Complete these before you submit your first production build — they take longer than the build itself.

Apple App Store Connect checklist

In App Store Connect, open your app and fill in the required sections under the App Store tab before submitting for review.

App information

  • Privacy policy URL — required. Host it on your website.
  • App category — select Social Networking (primary) and optionally a secondary category.
  • Age rating — complete the questionnaire. A community app with user-generated content typically gets 12+ or 17+.
  • App privacy (nutrition labels) — declare data types collected: contact info (email, name), user content (posts, messages, photos), identifiers (push token). Select Data not linked to user for push token. No data is used for tracking.

Store listing

  • Screenshots — required for 6.7" and 5.5" iPhone sizes (at minimum). iPad screenshots if you support tablets.
  • Description — your app's full description.
  • Keywords — up to 100 characters, comma-separated, for search.
  • Support URL — a link to your support/contact page.

Review information

  • Sign-in required — check this box and provide demo account credentials so Apple reviewers can log in. The #1 cause of iOS rejection is reviewers unable to sign in.
  • Contact information — a phone number and email Apple can reach you at during review.
  • Notes — briefly explain what the app is (e.g. “Private community app for [your org]. Requires an account on our website to log in.”).
Google Play Console checklist

Google Play requires several sections completed before you can create a production release. Find them under Grow → Store presence and Policy → App content.

App content

  • Privacy policy — URL to your privacy policy page.
  • App access — select All or some functionality is restricted and provide demo credentials.
  • Ads — select No (the app has no ads).
  • Content rating — complete the IARC questionnaire (typically results in Teen).
  • Target audience — select 18 and over unless you comply with COPPA.
  • Data safety — declare: account info, user-generated content, push token. Not shared with third parties for advertising.
  • Government apps / Financial features — select No unless applicable.

Store listing

  • App name, short description, full description.
  • App icon — 512×512 PNG, no transparency.
  • Feature graphic — 1024×500 JPG or PNG.
  • Screenshots — at least 2 phone screenshots (4–8 recommended).
  • App category — Social or closest fit.
  • Contact details — support email required.
Both stores require a publicly accessible privacy policy URL. It must describe what data the app collects (account info, user content, push tokens) and how it's used. Write this early — neither store will approve your app without it.
Local Development

This app uses an Expo dev client (not Expo Go). The dev client is a custom build that includes all native modules.

Dev client setup
  1. Build the dev client (one-time, or after adding native modules): eas build --profile development --platform [ios|android]
  2. Install the dev client on your device or simulator from the EAS build link
  3. Start the dev server: npx expo start
  4. Open the dev client on your device — it connects automatically (same WiFi network required)
Dev client vs Expo Go: Expo Go doesn't include the native modules this app needs. You only need to rebuild the dev client when adding or removing native packages — code changes hot-reload instantly.
Troubleshooting

Build fails immediately

Run npx expo install --check to detect outdated or incompatible packages, then npx expo install --fix to resolve them. Also check app.json for syntax errors (missing commas, trailing commas, etc.).

eas submit fails for iOS

Verify your Apple Account email and App Store Connect ID are set correctly in the Config tab. You also need an App-Specific Password — generate one at appleid.apple.com under Sign-In and Security → App-Specific Passwords.

eas submit fails for Android

Verify your Service Account Key (.json file) is uploaded in the Config tab. The service account must have the correct permissions in Google Play Console (Users and permissions → Invite user with Admin access). Also ensure the app exists in Google Play Console and you've uploaded at least one build to any testing track.

Build succeeds but app crashes on launch

This usually means a native module mismatch. Check that your app.json plugins array hasn't changed unexpectedly. If you recently added or removed a package, make sure it was a full native build (not an OTA update). Review the crash logs in your Sentry dashboard if you have crash reporting enabled (see the Companion Plugin → Crash Reporting tab), or fall back to App Store Connect (Crashes) / Google Play Console (Crashes and ANRs).

App rejected by store review

The most common reasons: (1) the reviewer can't log in — make sure you provided working demo credentials in the review notes, (2) the app crashes on launch, (3) missing privacy policy URL. See the Store Listing Checklist for the full required fields per store.

Pushing an OTA Update

Use the dashboard OTA tab or the CLI:

eas update --channel production --message "Brief description of changes"

Updates are delivered the next time users open the app. The app checks for updates on launch.

Reading the Results

Each check shows one of three states:

  • Pass — configured correctly
  • Fail — missing or placeholder value, needs to be set
  • Warn — optional but recommended (e.g., Apple submission fields, Google Play service account key)
Goal: All checks passing before your first production build. Warnings are OK if you haven't set up iOS submission yet.

Updates Tab

The Updates tab in the setup dashboard handles license activation, version checking, applying updates, and backup/restore. This is the recommended way to keep your app up to date.

License Activation

When you purchase the app from twobirdscode.com, you receive a license key.

  • Open the Updates tab in the dashboard
  • Enter your license key and click Activate
  • The license is automatically activated and linked to your site URL
  • Once activated, the tab displays your plan type and expiration date
  • Your license key is stored locally in setup/.license — excluded from builds and version control

Checking for Updates

  • The Updates tab automatically checks for available updates when opened (requires an active license)
  • Shows your current version vs the latest available version
  • Displays the changelog for the new version so you can see what changed
  • Click Check Now to manually recheck at any time

Applying Updates

  1. Click Backup & Update when an update is available
  2. The dashboard automatically creates a backup of your current state before applying anything
  3. The update package is downloaded from the license server and applied
  4. Only core files are updated — your customizations are never overwritten

Protected files that are never overwritten by an update:

  • constants/config.ts — your app name, site URL
  • app.json — your bundle IDs, app name, icons, EAS config
  • eas.json — your build profiles and site URL
  • app.config.ts — your fallback values and deep link paths
  • package.json — your package name and any added dependencies
  • assets/images/ — your branding assets
  • modules/ — your custom modules
  • google-services.json / GoogleService-Info.plist — your Firebase configs
After updating: Run npm install to pick up any new or updated dependencies. If the dashboard itself was updated, restart it with npm run dashboard.

Manual Upload

If you received an update package directly (e.g., via email or download), you can apply it manually:

  • Click Upload Package in the Updates tab
  • Select a .zip update file (e.g. core-update-3.5.0.zip)
  • The same backup and protected-file logic applies — your customizations are safe

Backup & Restore

  • An automatic backup is created before every update, stored in setup/.backups/
  • The dashboard lists available backups with dates and sizes
  • Click Restore on any backup to roll back to that state
  • The last 3 backups are kept; older ones are auto-pruned

manifest.json

The manifest.json file in the project root contains metadata used by the update system:

  • Product version number
  • Update server URL
  • List of protected paths (files that are never overwritten)
  • Core companion plugins list
Do not edit manifest.json manually. It is updated automatically when an update is applied. Editing it could break the update process.
