1. How It Works

Thread List (/messages) ├─ Fetches all threads + unread IDs in parallel ├─ Client-side search (name, username, preview) ├─ Swipe-to-delete threads ├─ Pusher updates thread previews in real-time ├─ New Message modal — search users, tap to start conversation └─ Tap thread → Chat screen Chat Screen (/messages/user/[userId]) ├─ Thread resolution: find existing thread or show "new conversation" ├─ Cursor-based message pagination (load older by scrolling up) ├─ Send text + image attachments + reply quotes ├─ Real-time: Pusher appends new messages instantly ├─ Reactions: emoji picker, optimistic toggle, Pusher sync ├─ Context menu: reply, delete (own messages) ├─ Image lightbox via MediaViewer └─ Block/unblock thread participants

The messaging system is built on Fluent Community’s Chat API (v2.2.0). Threads represent conversations between users. Messages are loaded via cursor-based pagination and updated in real-time via Pusher.

Navigation Into Chat

The chat screen is always navigated to by user ID, not thread ID. This supports both:

  • Existing conversations — the hook resolves the user ID to an existing thread
  • New conversations — when tapping “Message” on a profile or selecting a user in the New Message modal, no thread exists yet. The hook shows the user’s info and creates a thread on first message send.
// Navigate to chat with a user
router.push(`/messages/user/${userId}`);

// From thread list (uses userId from thread.info)
const userId = getThreadUserId(thread);
router.push(`/messages/user/${userId}`);

// From New Message modal (passes display info as params)
router.push({
  pathname: '/messages/user/[userId]',
  params: {
    userId: String(member.user_id),
    displayName: member.display_name,
    avatar: member.avatar || '',
  },
});

3. Thread Resolution

The useChatMessages hook handles finding the right thread for a given user ID. This is the most complex part of the messaging system.

useChatMessages({ targetUserId, knownThreadId, currentUserId, listRef }) ↓ knownThreadId provided? ├─ Yes: getMessages(threadId) directly └─ No: getThreadsForUser(userId) ↓ selected_thread returned? (existing conversation) ├─ Yes: load messages from that thread └─ No: intended_object returned (user info, no thread yet) ↓ Show user info + empty chat Thread created on first message send via startChatWithUser()

5. Cursor-Based Pagination

Messages use cursor-based pagination (not page numbers). The cursor is the id of the oldest loaded message.

// First load — newest messages
const response = await messagesApi.getMessages(threadId);
// response.data.messages = [newest, ..., oldest]
// response.data.has_more = true/false
// response.data.threadDetails = { id, title, info, blocked_thread, ... }

// Load older messages
const oldestId = messages[0].id;
const older = await messagesApi.getMessages(threadId, oldestId);
// Prepend older.data.messages to existing array

Message Order

The API returns messages in descending order (newest first). The useChatMessages hook reverses them to ascending (oldest first) for chronological display in a bottom-anchored FlashList:

const fetchedMessages = (response.data.messages || []).slice().reverse();

6. Sending Messages

Existing Thread

await messagesApi.sendMessage(threadId, text, attachments, replyData);

New Conversation (No Thread Yet)

// Creates thread + sends first message in a single API call
const response = await messagesApi.startChatWithUser(userId, text);
// response.data.thread — the new thread
// response.data.message — the sent message (optional)

Attachments

Image attachments are uploaded first via mediaApi.uploadMedia() with the 'chat' context, then their URLs are passed as ChatInputAttachment objects. Internally, sendMessage maps them to URL strings in the mediaItems body field:

// ChatInputAttachment shape
interface ChatInputAttachment {
  url: string;
  type: 'image';
  width?: number;
  height?: number;
}

// Usage
await messagesApi.sendMessage(threadId, text, [
  { url: 'https://...uploaded-image.jpg', type: 'image' },
]);

// Internally sent as: { text, mediaItems: ['https://...uploaded-image.jpg'] }

The ChatInput component limits attachments to 4 images per message (selectionLimit: 4 - attachments.length).

Reply Quotes

To reply to a specific message, pass reply_to (message ID) and reply_text (quoted text):

await messagesApi.sendMessage(threadId, text, undefined, {
  reply_to: originalMessageId,
  reply_text: 'The text being replied to',
});

7. Real-Time (Pusher)

New messages and reactions arrive via Pusher on the user’s private channel. See Pusher doc for connection details.

Hooks

HookSourcePurpose
useNewMessageListener(handler)contexts/PusherContext.tsxSubscribe to incoming messages
useReactionListener(handler)contexts/PusherContext.tsxSubscribe to reaction events

Both hooks use useEffectEvent internally to keep the handler reference stable without re-subscribing on every render.

Message Events

// In useChatMessages.ts — uses useEffectEvent for stable handler
const handleNewMessage = useEffectEvent((data: PusherMessage) => {
  const msgThreadId = data.thread_id || data.message.thread_id;
  if (thread && String(msgThreadId) === String(thread.id)) {
    const newMessage: ChatMessage = data.message;

    setMessages(prev => {
      // Deduplicate — message may already exist from sendMessage response
      const exists = prev.some(m => m.id === newMessage.id);
      if (exists) return prev;
      return [...prev, newMessage];
    });

    // Auto-scroll to bottom
    setTimeout(() => { listRef.current?.scrollToEnd({ animated: true }); }, 100);
  }
});

const handleReaction = useEffectEvent((data: PusherReaction) => {
  if (thread && String(data.thread_id) === String(thread.id)) {
    setMessages(prev => prev.map(m => {
      if (m.id === data.message_id) {
        return { ...m, meta: { ...m.meta, reactions: data.reactions } };
      }
      return m;
    }));
  }
});

useNewMessageListener(handleNewMessage);
useReactionListener(handleReaction);

Thread List Updates

The thread list screen also subscribes to useNewMessageListener. When a new message arrives, it updates the thread’s preview message and moves it to the top of the list.

Badge Counts

Unread message counts are tracked in UnreadCountsContext. The count updates via:

  • Startup batch — initial count via getMessageUnreadCount()
  • Response headers — X-TBC-Unread-Messages on every API call
  • Pusher — bumps count on incoming message event

8. Message Reactions

hooks/useChatReactions.tsx

Chat message reactions use the same emoji set as feed reactions (configured via useReactionConfig) but with a different API endpoint. Reactions are stored in the message’s meta.reactions field.

Data Shape

// meta.reactions is a map of emoji → array of user IDs
{
  "reactions": {
    "👍": [1, 42],     // user IDs who reacted with thumbs up
    "❤️": [1],         // user IDs who reacted with heart
  }
}

Interaction Modes

  • Tap smiley button — toggles the default reaction (first in config). If user already has a reaction, it toggles that one instead.
  • Long-press smiley button — opens the reaction picker (positioned at button anchor).
  • Tap existing reaction pill — toggles that specific reaction for the current user.

Optimistic Updates

All reactions use optimisticUpdate() from utils/optimisticUpdate. The UI updates immediately; if the API call fails, the change is reverted. Pusher also delivers the update, which reconciles the server state. One reaction per user — adding a new reaction removes the previous one.

API

// Toggle reaction (add if not present, remove if present)
await messagesApi.toggleReaction(messageId, '👍');

9. Block & Unblock

Users can block conversations to prevent receiving messages from a specific user.

// Block
await messagesApi.blockThread(threadId);

// Unblock
await messagesApi.unblockThread(threadId);

When a thread is blocked:

  • The chat input is disabled
  • A blocked-state indicator is shown
  • The user can unblock from the chat menu or the blocked state UI

Block state is read from threadDetails.blocked_thread in the messages response. The useChatMessages hook tracks it via the isBlocked state variable and provides handleBlockPress(displayName) and handleUnblockPress() handlers that show confirmation alerts.

13. Rules

  1. Navigate to chat by user ID, not thread ID. The useChatMessages hook handles thread resolution. This supports both existing and new conversations.
  2. Use messagesApi for all chat API calls. Don’t call endpoints directly. The service handles request building and type safety.
  3. Use the helper functions for thread data. Don’t access thread.info.title directly — use getThreadDisplayName(thread). The helpers handle null checks and fallbacks.
  4. Strip HTML from message text before display. Use getMessageText() for full text or getMessagePreview() for truncated previews. Never render raw message.text as plain text.
  5. Use useNewMessageListener and useReactionListener for real-time updates. Don’t subscribe to Pusher channels directly for messaging.
  6. Mark threads read on screen mount. The useChatMessages hook does this automatically after loading messages.
  7. Check features.messaging (via useFeatures()) before showing chat UI. Sites without Fluent Messaging installed should hide all messaging UI.
  8. Reactions use the same meta.reactions shape as feeds — a map of emoji string to array of user IDs. Toggle (not add/remove) is the API contract. One reaction per user.
  9. Deduplicate Pusher messages. The send response already adds the message to state; when Pusher delivers the same message, skip it (check prev.some(m => m.id === newMessage.id)).
  10. Use optimisticUpdate() for reactions. Update the UI immediately, call the API, revert on failure. This is handled by useChatReactions.
