1. Overview

The app uses a centralized tagged logger that serves two roles:

  • In dev (__DEV__) — every log line is printed to the Metro console prefixed with a [Tag] so you can filter by subsystem.
  • In production — info/warn become Sentry breadcrumbs, error becomes a captured exception. debug is dropped entirely. When a crash is captured, the breadcrumb trail shows you exactly what the user was doing in the seconds before it happened.
Sentry-aware. Once a Sentry DSN is configured in WP admin (TBC Community App → Crash Reporting), log.info, log.warn, and log.error automatically feed Sentry. No per-call-site changes needed.

How it works

// utils/logger.ts — public API

export interface Logger {
  /** Verbose dev signal. Console only in __DEV__, dropped in production. */
  debug(message: string, data?: Record<string, unknown>): void;

  /** Notable event. Console (dev) + Sentry breadcrumb (prod). */
  info(message: string, data?: Record<string, unknown>): void;

  /** Warning. Console (dev) + Sentry breadcrumb level=warning (prod). */
  warn(message: string, data?: Record<string, unknown>): void;

  /**
   * Captured error. First arg MUST be the error (Error instance preferred
   * for stack traces; strings/unknowns are captured as messages).
   * Second arg is an optional human description of what was happening.
   * Third arg is optional structured context.
   */
  error(
    error: unknown,
    message?: string,
    context?: Record<string, unknown>,
  ): void;

  /** Returns a child logger with persistent context merged into every call. */
  withContext(context: Record<string, unknown>): Logger;
}

export function createLogger(tag: string): Logger;

2. Basic Usage

import { createLogger } from '@/utils/logger';

const log = createLogger('MyFeature');

// Verbose dev tracing — dropped in production
log.debug('initialized', { itemCount: items.length });

// Notable event — becomes a Sentry breadcrumb in production
log.info('checkout started', { total, currency });

// Warning — handled but unexpected
log.warn('coupon expired', { code });

// Error — captured by Sentry as an exception
log.error(err, 'Failed to remove cart item', { itemId });

// Child logger with persistent context
const scoped = log.withContext({ userId: 42 });
scoped.info('clicked checkout');  // ctx merged automatically
Always declare the logger at module level (top of file), not inside functions or components.

3. Methods

MethodDev behaviorProduction behaviorUse for
log.debug(msg, data?) console.log dropped (no-op) Verbose tracing — request details, cache hits, internal state. Safe to leave in.
log.info(msg, data?) console.log Sentry breadcrumb (info) Notable events — auth state, navigation, write operations, lifecycle.
log.warn(msg, data?) console.warn Sentry breadcrumb (warning) Handled but unexpected — retries, fallbacks, validation failures.
log.error(err, msg?, ctx?) console.error Sentry.captureException Real failures — caught exceptions, broken invariants, 5xx responses. Error first, then optional human description, then optional structured context.
log.withContext(ctx) returns child logger returns child logger Scope a logger to a request, screen, or user — context auto-merged into every call.
The first arg to log.error is the error itself, not a string label. Pass Error instances for full stack traces. The optional second arg is a short human description ("Failed to load space"), and the optional third arg is structured context. This argument order is the opposite of log.info / log.warn — Sentry needs the error first. 
// Just the error
log.error(err);

// Error + description
log.error(err, 'Failed to load space');

// Error + description + structured context
log.error(err, 'Checkout failed', { cartId: 42, total: 99.99 });

4. When to Log

DO log

  • Write operations — creating, updating, deleting (posts, messages, profiles, threads)
  • Auth events — login, logout, token refresh, credential storage
  • Errors and failures — API errors, batch failures, unexpected states
  • Connection events — Pusher connect/disconnect, push notification registration
  • Batch operations — what's being batched, which sub-requests failed
  • Cache events — cache hits, misses, clears (when debugging data freshness)

DO NOT log

  • Every read/GET request — too noisy, the API client already logs at request level
  • Render cycles — never log inside render functions or on every re-render
  • Sensitive data — never log tokens, passwords, or full user credentials
  • UI state changes — tab switches, scroll positions, animation frames

Pattern: log on mutations, not on reads

// GOOD — log the write operation as info (becomes a breadcrumb)
export async function createThread(data: CreateThreadRequest) {
  log.info('createThread', { intentType: data.intent_type, intentId: data.intent_id });
  return post<CreateThreadResponse>(ENDPOINTS.CHAT_THREADS, data);
}

// GOOD — log caught errors with the Error first
try {
  await batchRequest(...);
} catch (err) {
  log.error(err, 'Batch request failed', { phase: 'batch' });
  throw err;
}

// BAD — don't use log.info for verbose tracing; use log.debug instead
export async function getThreads() {
  log.debug('fetching threads');  // dev only, dropped in prod
  return get<ThreadsResponse>(ENDPOINTS.CHAT_THREADS);
}

7. Adding Logging to a New File

Two lines at the top of your file — that's it:

import { createLogger } from '@/utils/logger';

const log = createLogger('YourTag');

Then use log.debug(), log.info(), log.warn(), and log.error() throughout the file.

Step-by-step for an API service

// services/api/widgets.ts

import { ENDPOINTS } from '@/constants/config';
import { get, post } from './client';
import { createLogger } from '@/utils/logger';

const log = createLogger('WidgetsAPI');

export async function getWidgets() {
  // No logging needed — it's a read
  return get('/widgets');
}

export async function createWidget(data: CreateWidgetData) {
  // Log the write operation as info (becomes a Sentry breadcrumb)
  log.info('createWidget', { name: data.name });
  return post('/widgets', data);
}

export async function deleteWidget(id: number) {
  // Log destructive operations as info
  log.info('deleteWidget', { id });
  return post(`/widgets/${id}/delete`, {});
}

Step-by-step for a module

// modules/mymodule/services/myApi.ts

import { createLogger } from '@/utils/logger';

const log = createLogger('MyModuleAPI');

// Follow the same pattern — log writes, skip reads

9. Rules

Never use raw console.log / console.warn / console.error.
Always use createLogger. The only exception is the module registry's __DEV__-wrapped validation warnings and the ErrorBoundary component.
  1. Import from @/utils/logger — the only logging utility in the project.
  2. Declare at module level — const log = createLogger('Tag') at the top of the file, not inside functions.
  3. Log writes, not reads — mutations, creates, deletes, state transitions. Skip GET requests.
  4. Never log sensitive data — no tokens, passwords, full credentials, or personal data.
  5. Keep tags short and unique — check the tag list before picking a new one.
  6. Use the right level — log.debug() for verbose dev tracing (dropped in prod), log.info() for breadcrumb-worthy events, log.warn() for handled issues, log.error(err, msg?, ctx?) for failures (passes err first).
  7. Production routes through Sentry — info/warn/error reach Sentry as breadcrumbs/exceptions when a DSN is configured. Debug never does.
  8. Don't create a new logging utility — createLogger is the only one. Don't add dependencies like winston, pino, etc.
