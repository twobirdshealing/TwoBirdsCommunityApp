State Management

React Context providers, nesting order, data flow, and service ↔ React bridges

Source directory: contexts/
The app uses React Context for all global state. No external state library (Redux, Zustand, etc.). Screen-level data uses useCachedData. Contexts handle auth, config, theme, real-time, badges, and UI coordination.

1. Provider Nesting Order

Providers are nested in app/_layout.tsx. Order matters — each provider can only access contexts above it.

GestureHandlerRootView └─ ErrorBoundary └─ ThemeProvider ← Colors, dark mode, maintenance, update, branding └─ AuthProvider ← Login/logout, user state, profile completion └─ AppConfigProvider ← Visibility flags, portal slug, socket config └─ UnreadCountsProvider ← Badge counts (notifications, messages, cart) └─ PusherProvider ← Real-time WebSocket (needs Auth + AppConfig) └─ ModuleProviders ← Dynamic providers from module registry └─ KeyboardProvider └─ BottomSheetModalProvider └─ RootLayoutNav (screens)

Additional providers are nested inside tab screens, not in the root layout:

ProviderLocationWhy Not Root
TabBarProviderapp/(tabs)/_layout.tsxOnly tab screens need scroll-hide behavior
BottomOffsetProviderapp/(tabs)/_layout.tsxOnly tab screens need addon height offset
FeedModalsProvidercomponents/feed/FeedList.tsxWraps each FeedList instance so modals are scoped to the list

2. AuthContext

contexts/AuthContext.tsx

State

FieldTypePurpose
isAuthenticatedbooleanWhether user has a valid JWT
isLoadingbooleanTrue during initial auth check on mount
userAuthUser | nullCurrent user profile (from SecureStore)
needsProfileCompletionbooleanSet by response headers when profile is incomplete

Key Methods

const { login, logout, refreshAuth, updateUser, registerAndLogin, markProfileComplete } = useAuth();
MethodPurpose
login(username, password)Authenticate → store tokens → register push token
logout()Revoke session → clear caches → unregister push → clear SecureStore
registerAndLogin(access, refresh, user)Store tokens directly after registration (no login call needed)
updateUser(updates)Merge partial updates into user state + SecureStore
refreshAuth()Re-read user from SecureStore (after external changes)

Startup Flow

  1. Check hasStoredAuth() — is there a JWT in SecureStore?
  2. If yes: load stored user, set isAuthenticated = true, re-register push token
  3. If no: set isLoading = false (shows login screen)
  4. Register onAuthCleared callback so API client can force logout on refresh failure
Auth is checked client-side only. On mount, the app checks if a JWT exists in SecureStore — it doesn’t validate it with the server. If the token is expired, the first API call triggers silent refresh.

3. ThemeContext

contexts/ThemeContext.tsx

Manages the app’s visual theme and also holds server-synced config data (maintenance mode, update prompts, branding).

State

FieldTypePurpose
theme'light' | 'dark'Current theme mode
isDarkbooleanConvenience: theme === 'dark'
colorsColorThemeResolved color tokens (base + Fluent overrides)
updateUpdateConfig | nullApp update info from server (version, URL, force flag)
maintenanceMaintenanceConfig | nullMaintenance mode (message, bypass flag)
brandingBrandingConfig | nullServer-synced logos and brand colors

Usage

const { colors, isDark, theme, setTheme, update, maintenance, branding } = useTheme();

// Theme colors (most common usage)
<View style={{ backgroundColor: colors.background }}>
  <Text style={{ color: colors.text }}>Hello</Text>
</View>

// Check maintenance mode
if (maintenance?.enabled) { /* show maintenance screen */ }

// Check for updates
if (update?.force) { /* show force update screen */ }

Color Resolution

Colors are resolved by merging the base palette with Fluent Community’s server-sent overrides:

// Base colors (constants/colors.ts)
const base = isDark ? darkColors : lightColors;

// Server overrides (from /app-config → theme_data)
const overrides = fluentOverrides?.[isDark ? 'dark' : 'light'] || {};

// Merged result
const colors = { ...base, ...overrides };

See Theme System doc for full token reference and usage rules.

Data Flow

ThemeContext receives config data from two sources:

  1. Startup batch — _layout.tsx calls setFromBatch(data) with pre-fetched config
  2. Background refresh — refreshAppConfig() fetches /tbc-ca/v1/app-config directly

Both paths extract theme colors, maintenance status, update info, branding, and social providers from the same AppConfigResponse.

4. AppConfigContext

contexts/AppConfigContext.tsx

Holds feature visibility flags, the portal slug, and socket provider configuration. This context is a passive state holder — it doesn’t fetch data itself. All data is fed in from _layout.tsx.

State

FieldTypePurpose
visibilityVisibilityConfig | nullFeature visibility from server (hide_menu[] array)
portalSlugstringFluent Community portal slug (used in deep link URL mapping)
socketConfigSocketConfig | nullPusher/Soketi connection config (api_key, cluster, wsHost)

Usage

const { visibility, portalSlug, socketConfig } = useAppConfig();

// Check if a menu item should be hidden
const showCourses = !visibility?.hide_menu?.includes('courses');

// Portal slug for deep links
const feedUrl = `${SITE_URL}/${portalSlug}/feeds/${slug}`;

Cache Persistence

Visibility and socket config are cached to AsyncStorage so the app has correct state on cold start before the batch response arrives:

  • tbc_app_visibility_cache — survives logout (site-level, not user-specific)
  • tbc_socket_config_cache — survives logout

Socket Config Stability

Socket config uses shallow comparison on key fields (api_key, cluster, wsHost) before updating state. This prevents unnecessary Pusher reconnections when the config hasn’t actually changed.

5. UnreadCountsContext

contexts/UnreadCountsContext.tsx

Single source of truth for badge counts displayed on notification bell, chat icon, and cart icon.

State

FieldTypeWritten By
unreadNotificationsnumberStartup batch, response headers, push listener
unreadMessagesnumberStartup batch, response headers, Pusher real-time
cartCountnumberStartup batch, response headers

How Counts Stay Updated

  1. Startup batch — populates initial counts from a single API call
  2. Response headers — every authenticated API response piggybacks current counts via X-TBC-* headers (see API Client → Response Headers)
  3. Real-time — Pusher events bump message count; push notifications bump notification count

Usage

const { unreadNotifications, unreadMessages, cartCount, resetCounts } = useUnreadCounts();

// Display badge
<BadgeIcon count={unreadNotifications} />

// On logout
resetCounts();

6. PusherContext

contexts/PusherContext.tsx

Manages the Pusher WebSocket connection for real-time messaging. See Pusher doc for full details.

Dependencies

Requires both AuthContext (for user ID + auth state) and AppConfigContext (for socket config). Connects when both are available, disconnects on logout.

Key Behavior

  • Subscribes to private-chat_user_{id} for direct messages
  • Reconnects on app focus (via useAppFocus)
  • Falls back to polling via useTickerPolling when WebSocket fails
  • Bumps unreadMessages count on incoming message events

7. TabBarContext

contexts/TabBarContext.tsx

Controls scroll-aware tab bar hiding. Only available inside app/(tabs)/_layout.tsx.

How It Works

Tab bar visibility is driven by a Reanimated SharedValue. Scroll direction is detected via plain JS onScroll (FlashList compatible), and the animation runs on the UI thread.

const { handleScroll, showTabBar, setLocked } = useTabBar();

// Attach to any ScrollView or FlashList
<FlashList onScroll={handleScroll} />

// Force visible (e.g., on tab switch)
showTabBar();

// Lock visible during edit mode
setLocked(true);

Scroll Logic

  • Scroll down past threshold (10px) → hide tab bar
  • Scroll up past threshold → show tab bar
  • Near top of list → always show
  • Locked → always show, ignore scroll events

8. BottomOffsetContext

contexts/BottomOffsetContext.tsx

Tracks extra height above the tab bar from module addons (e.g., mini audio player, cart bar). Tab content screens use this to set correct bottom padding.

Usage

import { useTabContentPadding } from '@/contexts/BottomOffsetContext';

// Returns: tab bar height + safe area + addon height + spacing
const bottomPadding = useTabContentPadding();

<FlatList contentContainerStyle={{ paddingBottom: bottomPadding }} />

How Addon Height Is Set

The tab bar layout measures the addon area via onLayout and calls setAddonHeight(). When no addon is rendered, height is 0.

9. FeedModalsContext

contexts/FeedModalsContext.tsx

Hoists all feed card modals (menu, reaction picker, reaction breakdown, media viewer, report) into a single provider. Without this, every FeedCard in a FlashList would render its own 5 modals.

Where It Lives

FeedModalsProvider wraps FeedList directly (not in the root layout). Each feed list has its own provider instance. It’s also used in app/feed/[id].tsx for the single-feed detail view.

Usage (from FeedCard)

const { openMenu, openReactionPicker, openMediaViewer, openReport } = useFeedModals();

// Open the 3-dot menu
openMenu({
  feed,
  anchor: { top: y, right: x },
  isOwner: feed.user_id === currentUser.id,
  canEditOrDelete: true,
  canPin: false,
  isSticky: feed.is_sticky,
  onEdit: () => handleEdit(feed),
  onDelete: () => handleDelete(feed),
});

10. Module Providers

Modules can register their own context providers via the module manifest. These are dynamically nested inside ModuleProviders in app/_layout.tsx.

// modules/bookclub/module.ts
export const bookclubModule: ModuleManifest = {
  providers: [{ component: AudioPlayerProvider }],
  // ...
};

Providers are nested from outside-in based on registration order using reduceRight. Module providers have access to all core contexts (Auth, Theme, AppConfig, etc.) since they’re nested inside them.

Module providers are automatically included. Register them in the module manifest and they’ll be nested in the root layout. No need to edit _layout.tsx.

11. Service ↔ React Bridges

Several non-React services need to communicate with React contexts. Since services can’t call hooks, they use a callback registration pattern:

ServiceRegistration FunctionConsumerPurpose
services/auth.tssetOnAuthCleared(callback)AuthContextForce logout when silent refresh fails
services/api/client.tssetOnResponseHeaders(callback)_layout.tsxExtract badge counts + maintenance from response headers

Pattern

// In the service (non-React):
let callback: ((data: SomeData) => void) | null = null;

export function setOnSomething(cb: typeof callback): void {
  callback = cb;
}

// Inside the service, when an event occurs:
if (callback) callback(data);

// In the React component (registers on mount, cleans up on unmount):
useEffect(() => {
  setOnSomething((data) => {
    setContextState(data);
  });
  return () => setOnSomething(null);
}, []);
Why not events or observables? The callback pattern is the simplest approach for 1:1 communication between a singleton service and a single React consumer. No event system, no subscription management, no memory leaks from forgotten listeners.

12. Rules

  1. Use useCachedData for screen data, contexts for global state. Don’t create a context to hold data that only one screen uses. Contexts are for state shared across multiple screens or the entire app.
  2. Don’t add new contexts without checking existing ones. The current set covers auth, config, theme, badges, real-time, and UI coordination. Most new features don’t need a new context.
  3. Module providers go in the module manifest. Don’t edit _layout.tsx to add module providers. Register them via providers: [{ component: MyProvider }] in the module’s module.ts.
  4. Use the callback bridge pattern for service → React communication. Don’t import React contexts into plain services. Register a callback from the React side.
  5. Always throw in useMyContext() hooks. All context hooks should throw an error if used outside their provider. This prevents silent failures.
  6. Memoize context values. Wrap the value prop in useMemo to prevent unnecessary re-renders in consumers. All existing contexts follow this pattern.
  7. Keep contexts in contexts/. Module-specific contexts live in modules/{name}/contexts/. Don’t scatter context files across the project.
  8. Never import lightColors/darkColors directly. Always use const { colors } = useTheme(). Direct imports bypass Fluent overrides. See Theme System rules.
  9. Use useUnreadCounts() for badge counts. Don’t fetch notification counts independently. The context is the single source of truth, updated by startup batch + response headers + real-time events.
  10. Use useTabContentPadding() for bottom padding in tab screens. Don’t manually calculate tab bar + safe area + addon height. The hook handles it.
