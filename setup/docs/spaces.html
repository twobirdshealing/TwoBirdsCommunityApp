Spaces (Groups)

Discovery, membership, feeds, lock screens, and space groups

Key files: services/api/spaces.ts (API), app/(tabs)/spaces.tsx (discovery tab), app/space/[slug]/index.tsx (detail screen)
Spaces are Fluent Community's groups. Users discover them on the Spaces tab, join/leave, and view per-space feeds. Spaces are organized into Space Groups (categories) and support public, private, and secret privacy levels.

1. Overview

Spaces are the main organizational unit in Fluent Community. They function like groups or channels — each space has its own feed, member list, and privacy settings.

Key Concepts

ConceptDescription
SpaceA group with its own feed, members, and settings
Space GroupA category that organizes multiple spaces (displayed as tabs)
PrivacyPublic (open), Private (request to join), Secret (invite only)
RoleAdmin, Moderator, or Member within a space
Lock ScreenCustom content shown to non-members of private/secret spaces

  role?: 'member' | 'moderator' | 'admin';

  // User's relationship (profile endpoint)

  // User's relationship (discover endpoint)
  space_pivot?: { id, space_id, user_id, status, role, meta, created_at, updated_at } | null;

  creator?: { id, username, display_name };

  permissions?: Record<string, boolean>;
  membership?: { ID, display_name, pivot: { role, status, ... } };
  topics?: Record<string, unknown>[];
  header_links?: { title, route: { name } }[];
  lockscreen_config?: LockScreenConfig | null;
}

4. Discovery & Browsing

app/(tabs)/spaces.tsx is the Spaces tab. It loads all discoverable spaces and organizes them by group.

Data Loading

Uses useAppQuery with CACHE_EVENTS.SPACES for invalidation. Two parallel fetches inside the fetcher:

  1. spacesApi.getSpaceGroups({ options_only: true }) — lightweight group list (id + title) for tabs
  2. spacesApi.discoverSpaces({ per_page: 100 }) — all spaces with membership info

Group Tabs

Spaces are organized into horizontal tabs by group. Each tab shows its group name and space count badge. Spaces without a group appear under "Other Spaces". Within each group, spaces are sorted by their serial field. Tabs only appear when there are multiple groups.

Space Cards

Each space renders as a SpaceCard with:

  • 16:9 cover image (prefers og_image, falls back to cover_photo, filters out Fluent default placeholders) with gradient overlay
  • Space logo or emoji icon on the hero overlay
  • Title + privacy label + member count on the gradient overlay
  • Description snippet below the hero (2 lines max, HTML stripped)
  • Membership badge (top-right overlay):
    • Admin/Mod (primary color background) — user is admin or moderator
    • Awaiting (glass/semi-transparent) — join request pending
    • Joined (glass/semi-transparent) — regular member
    • Join / Request button (bordered glass) — non-member; shows lock-open icon for private spaces, plus icon for public

Search

Full-text search filters by title, description, and slug across all spaces in the current tab. Shows a result count when searching.

5. Space Detail Screen

app/space/[slug]/index.tsx shows an individual space:

  • PageHeader — back button, space title, SpaceMenu (settings icon) on the right
  • SpaceInfoHeader — 16:9 cover photo with gradient, logo, title, privacy label, description
  • QuickPostBox — shown for members ("Post in [Space]..."), or "Join Space to Post" button for non-members of public spaces
  • FeedList — space-specific feed with full feed actions (react, comment, bookmark, edit, delete, pin)
  • SpaceLockScreen — shown for non-members of private/secret spaces instead of the feed

The space is fetched by slug via spacesApi.getSpaceBySlug(slug). The detail response includes permissions, membership, and lockscreen_config not present in the discovery list. Feeds are fetched separately using useAppQuery with CACHE_EVENTS.FEEDS invalidation.

Pin/Sticky Support

Admins and moderators can pin posts within a space. The permission check uses membership.pivot.role and the permissions object (community_admin, community_moderator, edit_any_feed). Pinned posts appear at the top of the feed.

6. Join & Leave

Join Flow

Space PrivacyBehavior
PublicInstant membership. Feed access immediately. Optimistic UI update shows post box right away.
PrivateJoin request sent. Status becomes "pending". Admin must approve.
SecretInvite-only. No join button shown. Not in discovery.
// Join a space
const response = await spacesApi.joinSpace('general');
// Response includes data.data.status:
//   undefined/absent = auto-approved (active)
//   'pending' = awaiting admin approval

After joining, CACHE_EVENTS.SPACES is emitted to refresh the discovery tab and update badges.

Leave Flow

  1. User taps "Leave Space" in SpaceMenu
  2. Confirmation alert shown ("Leave" / "Cancel")
  3. spacesApi.leaveSpace(slug) called
  4. Cache invalidated via CACHE_EVENTS.SPACES
  5. Navigates back via router.back()

7. Member List

app/space/[slug]/members.tsx shows all members of a space:

  • Sorting: Admins and moderators first, then members, alphabetically within each group
  • Section headers: "Leadership" for admins/mods, "Members" for the rest
  • Per member: Uses MemberCard component with avatar, display name, role badge, bio, last active
  • Actions: Follow/notify toggle and message button (hidden for current user)
  • Pagination: 20 per page with infinite scroll (FlashList onEndReached)
  • Follow state: Tracks current_user_follows from API response via useFollowToggle

Lock Screen System

components/space/SpaceLockScreen.tsx renders for non-members of private/secret spaces. Three rendering modes inside the component:

  1. Pending — "You're all set! A team member will review and let you in soon." (time icon)
  2. Custom — WordPress-configured blocks with images, text, overlays, and buttons. Blocks with hidden: true are filtered out. Includes optional "Ask to Join" button.
  3. Default — "This is a private space" with lock icon and "Request Access" button

Redirect mode is handled by the parent SpacePage before SpaceLockScreen renders — if lockscreen_config.redirect_url is set, the page navigates to a WebView and does not mount the lock screen component.

Lock Screen Config

9. Space Feeds

Each space has its own feed, filtered by space slug:

// Fetch space feed
feedsApi.getFeeds({ space: 'general', per_page: 20 });

// Create post in space
feedsApi.createFeed({ space: 'general', message: '...', content_type: 'markdown' });
Always use space slug, not ID. The space parameter in feed API calls is the space's URL slug (e.g., 'general'), not its numeric database ID. This matches the Fluent Community web behavior.

Sticky Posts

Space admins and moderators can pin posts. The space detail screen fetches sticky posts from a separate response.data.sticky field (FC 2.3.0+), merges them at the top of the feed with is_sticky: true, and removes duplicates from the regular feed list.

Quick Post Box

The space detail screen shows a QuickPostBox when the user is a member. Tapping opens the composer with the space pre-selected. Non-members of public spaces see a "Join Space to Post" button instead. Non-members of private/secret spaces see the lock screen.

10. Space Groups

Space Groups are categories that organize multiple spaces. They're managed in WordPress admin, not the app.

Hierarchy

SpaceGroup (e.g., "Discussion")
  ├── Space: General Chat (parent_id → group.id)
  ├── Space: Introductions
  └── Space: Off Topic

SpaceGroup (e.g., "Projects")
  ├── Space: Feature Requests
  └── Space: Bug Reports

Ungrouped → "Other Spaces" tab
  • Groups are sorted by serial field
  • Spaces within groups are also sorted by serial
  • parent_id on a Space links it to a SpaceGroup's id
  • Groups can be collapsible with settings.is_collapsible and settings.default_collapsed
  • The Spaces tab fetches groups via options_only: true for a lightweight list (id + title only)

13. Rules

  1. Always use space slug, not numeric ID. API calls use the slug (e.g., 'general') for join, leave, members, and feed filtering. The only exception is getSpaceById() which exists for legacy compatibility.
  2. Use discoverSpaces() for the Spaces tab, getSpaces() for the user's joined spaces. The discover endpoint returns all visible spaces with membership info. The regular endpoint only returns spaces the user has joined.
  3. Space Groups are categories, not nested spaces. parent_id on a Space points to a SpaceGroup, not another Space. There's no space-within-space nesting. Groups are purely organizational for the discovery UI.
  4. Join behavior depends on privacy. Public = instant, Private = pending approval, Secret = not shown. Check the response data.data.status for 'pending' to show the correct UI after joining.
  5. Emit CACHE_EVENTS.SPACES after join/leave. This refreshes the discovery tab and updates membership badges. Don't rely on local state mutations alone.
  6. Lock screen config comes from the detail endpoint only. The discovery list doesn't include lockscreen_config. Always fetch via getSpaceBySlug() before rendering lock screen content.
  7. Space feeds use the same feed API with a slug filter. Don't build separate feed fetching for spaces. Use feedsApi.getFeeds({ space: slug }). The feed system handles everything else (reactions, comments, sticky posts).
  8. Member list sorts leadership first. Admins and moderators appear under "Leadership" section before regular "Members". The sort is by role priority then alphabetical within each group.
  9. Privacy icons on SpaceCard. globe-outline = public, lock-closed-outline = private, eye-off-outline = secret. These help users understand access level before tapping into a space.
  10. Server enforces all access control. The app doesn't pre-check permissions. If a non-member tries to access a private space's feed, the server returns the lock screen config. Don't add client-side access checks that could get out of sync.
  11. Membership comes from different fields per endpoint. Discover uses space_pivot, profile uses pivot, detail uses membership.pivot. SpaceCard merges them: const pivot = space.space_pivot ?? space.pivot.
