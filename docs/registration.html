2. The 5-Step Wizard

app/register.tsx renders a multi-step form. Steps 3–5 are conditional based on server config.

StepComponentPurposeConditional?
1Inline (renderStep1)Name, email, username, passwordAlways shown
2Inline (renderStep2)Custom profile fields + terms checkboxSkipped if no custom fields
3EmailVerifyStep6-digit email verification codeOnly if email_verification_required: true
4+Module-injected stepsVerification steps from add-on modulesOnly if module's shouldActivate() returns true
LastPost-login stepsProfile completion, onboarding gatesOnly if server config enables them

Step 1: Basic Information

FieldValidationNotes
full_nameRequired, non-emptyFrom Fluent Community config
emailRequired, regex: /^[^\s@]+@[^\s@]+\.[^\s@]+$/Server also checks uniqueness
usernameRequired, 4+ chars, /^[a-z0-9_]+$/Lowercase + underscores only
passwordRequired, shown with PasswordStrengthMeter
conf_passwordMust match password (if required by FC)Optional based on FC config

Step 2: Custom Fields + Terms

Renders any custom fields where show_on_signup = true in the WordPress admin. All fields are rendered via the DynamicFormField component. A terms inline checkbox is always present (required).

Step Transitions

After step 1 (or step 2 if custom fields exist), handleSubmitRegistration() sends a POST /register request. The server response determines what happens next:

  • email_verification_required: true → advance to step 3
  • Module-specific flags (e.g., verification_required: true) → advance to module steps
  • success: true with tokens → auto-login, then step 5 or main app

All endpoints in services/api/registration.ts. Registration endpoints are public (no auth required). Profile completion endpoints require JWT auth.

RegisterResponse

The POST /register response has three possible shapes:

// ✓ Success — user created, tokens returned
{
  success: true,
  access_token: "eyJ...",
  refresh_token: "eyJ...",
  user: { id: 42, username: "jane", display_name: "Jane", email: "jane@..." }
}

// → Email verification needed (re-submit with code + token)
{
  success: false,
  email_verification_required: true,
  verification_token: "signed-jwt-token",
  message: "Please check your email for a verification code."
}

// → Module verification needed (add-on plugins can return custom verification requirements)
// The app checks each registered module's shouldActivate() against this response.
{
  success: false,
  // Add-on-specific fields, e.g.:
  // my_verification_required: true,
  // session_key: "abc123",
  message: "Additional verification required."
}

// ✗ Validation errors
{
  success: false,
  message: "Registration failed.",
  errors: {
    email: "This email is already registered.",
    username: "This username is already taken."
  }
}
Re-submission pattern: After email verification or module-injected verification steps, the app re-calls submitRegistration() with the original form data plus the verification data (e.g., __two_fa_code / __two_fa_signed_token for email). The server validates the verification and creates the user.

4. Module-Injected Verification Steps

Add-on modules can inject verification steps into the registration wizard using the registrationSteps manifest field. These steps appear after the core email verification step and before user creation completes.

How It Works

  1. User submits the registration form
  2. Server returns a response with add-on-specific fields (via tbc_ca_registration_config hooks)
  3. The app checks each registered module step's shouldActivate(submitResponse)
  4. If a step activates, the wizard advances to show that module's verification UI
  5. After the module step completes, the app re-submits registration with the verification data
  6. Server validates and creates the user

RegistrationStepRegistration

interface RegistrationStepRegistration {
  id: string;             // Unique step ID
  order: number;          // Sort position (core email-verify = 10)
  phase: 'pre-creation';  // When the step runs
  title: string;          // Step title in wizard
  component: ComponentType<RegistrationStepProps>;
  shouldActivate: (submitResponse: Record<string, unknown>) => boolean;
}

See Module System → Registration Types for full details on building custom verification steps.

Server-driven activation. The server response determines which module steps activate. The app never hardcodes which steps to show — it queries each registered module's shouldActivate() function against the server response.

5. Email Verification

Email verification uses Fluent Community's built-in 2FA system. When enabled on the site, registration requires a 6-digit email code before proceeding.

Flow

  1. User submits registration form (step 1 + 2)
  2. Server returns email_verification_required: true + verification_token (signed JWT)
  3. App advances to step 3 (EmailVerifyStep)
  4. User checks email, enters 6-digit code
  5. App re-submits registration with __two_fa_code + __two_fa_signed_token
  6. Server validates code → proceeds to module steps or user creation

EmailVerifyStep Component

components/register/EmailVerifyStep.tsx

  • 6-digit code input with number-pad keyboard
  • 60-second resend countdown timer
  • Token passed as prop from parent (stored in verificationToken state)
  • On expired token → shows error, back button returns to previous step

6. Profile Completion

After registration (or on next login if skipped), users may be required to complete their profile. This feature is powered by an add-on module that uses the module system's responseHeaders mapping to detect incomplete profiles.

How It Works

A profile completion add-on module registers a responseHeaders mapping in its manifest. The API client intercepts the header on every authenticated response and updates AuthContext. If the profile is incomplete, the auth guard redirects to /profile-complete.

Login-Time Gate

app/profile-complete.tsx — standalone screen shown to returning users whose profile is incomplete.

  • Fetches existing profile data via checkProfileComplete()
  • Pre-populates form with existing bio, avatar, social links
  • Shows which fields are missing (status.missing array)

Completion API

// Check profile status (authenticated)
const status = await checkProfileComplete();
// → { profile_complete: false, missing: ['avatar', 'bio'], existing: { bio: '', avatar: '', ... } }

// After user finishes the steps:
await completeRegistration();
// → POST /auth/register/complete marks profile as complete
markProfileComplete(); // Updates AuthContext
router.replace('/(tabs)');

Response Header Pattern

Add-on modules can register response headers via the module system. The API client intercepts these headers on every authenticated response and stores the values for the module to act on. See Module System → ResponseHeaderMapping for details.

7. Password Recovery

Password recovery endpoints are on the tbc-ca/v1 prefix (tbc-community-app plugin). The server determines which verification method to use (email or add-on module verification).

Flow

  1. User enters username or email → forgotPassword(login)
  2. Server returns a verification method (e.g., email_sent: true for email link, or add-on-specific verification)
  3. User completes verification → server returns reset_token + login
  4. User enters new password → resetPassword(token, login, newPassword)

Password Recovery API

FunctionMethodEndpointPurpose
forgotPassword(login)POSTtbc-ca/v1/password/forgotInitiate password reset (sends verification)
resetPassword(token, login, pwd)POSTtbc-ca/v1/password/resetSet new password with reset token
Extensible recovery: The server decides which verification method to use. Add-on modules can hook into the password recovery flow to provide additional verification methods beyond the default email reset.

8. Auth Guard & Route Protection

app/_layout.tsx controls which routes are accessible based on auth state.

Auth Group Routes

const inAuthGroup =
  currentSegment === 'login' ||
  currentSegment === 'register' ||
  currentSegment === 'forgot-password' ||
  currentSegment === 'webview' ||
  currentSegment === 'profile-complete';

Navigation Logic

  1. Unauthenticated + NOT in auth group → redirect to /login
  2. Authenticated + profile incomplete + NOT on gate → redirect to /profile-complete
  3. Authenticated + profile complete + on login page → redirect to /(tabs)
Register stays accessible after auth. The /register route is in the auth group, so it remains accessible even after registerAndLogin() completes. This is intentional — the avatar upload step (step 5) happens after the user is already authenticated.

Auto-Login After Registration

// In register.tsx, after successful registration:
const response = await submitRegistration(formData);

if (response.success && response.access_token) {
  // Store tokens + set isAuthenticated = true
  await registerAndLogin(response.access_token, response.refresh_token!, response.user!);

  // Then advance to next module step if enabled
  if (hasPostLoginStep(fieldsConfig)) {
    setStep(5);
  } else {
    router.replace('/(tabs)');
  }
}

9. Custom Profile Fields

Custom fields are defined in Fluent Community's native custom profile fields (FC 2.3.0+) and returned dynamically by GET /auth/register/fields. The tbc-community-app plugin maps FC field types to app input types.

RegistrationField Type

interface RegistrationField {
  label: string;
  placeholder?: string;
  instructions?: string;
  type: 'text' | 'email' | 'password' | 'phone' | 'date' | 'number'
      | 'select' | 'radio' | 'gender' | 'textarea' | 'url'
      | 'inline_checkbox' | 'checkbox' | 'multiselect';
  input_type?: string;
  required: boolean;
  options?: string[];      // For select/radio/checkbox/multiselect
  inline_label?: string;   // For inline_checkbox (e.g., terms text)
  step: 1 | 2;            // Which wizard step to show this field on
}

Field Rendering

All custom fields are rendered via DynamicFormField (components/common/DynamicFormField.tsx). This component maps the type to the appropriate input: text input, date picker, select modal, radio group, checkbox, textarea, etc.

Field Storage (WordPress)

  • Definitions managed via Fluent Community admin → Custom Profile Fields (stored in fcom_meta)
  • User values stored in fcom_xprofile.custom_fields JSON column (FC native storage)

13. Rules

  1. All registration behavior is server-driven. The app reads GET /register/fields on mount and adapts the wizard steps. Don't hardcode which steps to show — let the server config decide.
  2. Re-submit the full form after verification. Email verification and module-injected verification steps are not standalone — the app re-calls submitRegistration() with the original form data plus verification tokens. The server creates the user on this second call.
  3. Keep the register route in the auth group. The avatar upload step (step 5) happens after registerAndLogin() — the user is already authenticated but still on the register screen.
  4. Default checkProfileComplete to true on error. If the profile status check fails, assume the profile is complete. Never block users from the app because of a transient API error.
  5. Registration endpoints are public. The tbcPublicRequest helper uses raw fetch(), not the authenticated API client. Don't add auth headers to registration calls.
  6. Field errors are keyed by field name. Server validation errors come back as errors: { fieldKey: "message" }. Display them inline next to the corresponding field, not as a generic banner.
  7. Custom fields use DynamicFormField. All field types (text, select, radio, date, checkbox, textarea, etc.) are rendered by the same component. Don't create separate field components for registration.
