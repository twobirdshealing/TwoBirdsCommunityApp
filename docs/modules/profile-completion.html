Overview

The Profile Completion module acts as a login gate that ensures users have filled out essential profile fields before they can access the app. Unlike the OTP module which runs during registration, this module activates after account creation and intercepts every authenticated session where the profile is still incomplete.

Prerequisites

  • The tbc-profile-completion companion plugin installed and activated on your WordPress site
  • The Profile Completion module enabled in modules/_registry.ts

Setup

1. Install the companion plugin

Upload companion plugins/tbc-profile-completion to your WordPress site and activate it.

2. Configure required fields

In WordPress admin, go to the Profile Completion settings page and choose which fields are required:

  • Bio — require users to write a short biography
  • Avatar — require users to upload a profile photo
  • Social links — optionally require social media links (configurable per admin settings)

3. Enable the module

In modules/_registry.ts, ensure the Profile Completion module is registered:

import profileCompletion from './profile-completion/module';

export const modules = [
  profileCompletion,
  // ...other modules
];

How It Works

The module uses a response header detection pattern rather than a registration step. This means it works for both new sign-ups and existing users who have not yet completed their profiles.

User logs in or opens the app ↓ App makes any authenticated API request ↓ Server attaches header: X-TBC-Profile-Incomplete: 1 ↓ Module's responseHeaders config extracts header: key: "profileIncomplete" transform: value → boolean ↓ ProfileIncompleteProvider detects flag ↓ User is redirected to LoginGateScreen (/profile-complete route) ↓ ProfileCompletionSteps shown: • Bio input • Social links (if configured) • Avatar upload ↓ User submits → POST tbc-pcom/v1/complete ↓ Server clears X-TBC-Profile-Incomplete header ↓ Provider detects header is gone ↓ User accesses the app normally

Detection Flow

The detection is continuous, not one-time. The ProfileIncompleteProvider wraps the entire app (registered at order 90) and monitors every API response for the X-TBC-Profile-Incomplete header. This means:

  • If an admin later marks additional fields as required, existing users will be gated again on their next session
  • The gate cannot be bypassed by navigating directly to app screens — the provider intercepts at the top level
  • Once the header clears (profile is complete), the redirect stops immediately without requiring a restart
No registration steps. Unlike the OTP module, Profile Completion does not register any registrationSteps. It operates entirely as a post-login gate via the provider + response header pattern.

API Endpoints

All endpoints use the base path ${SITE_URL}/wp-json/tbc-pcom/v1.

GET tbc-pcom/v1/status

Check whether the current user's profile is complete. Returns a list of missing fields.

ParameterTypeDescription
No parameters — uses the authenticated user's session

Response:

{
  "complete": false,
  "missing": ["bio", "avatar"]
}

POST tbc-pcom/v1/complete

Submit the completed profile data. On success, the server stops sending the X-TBC-Profile-Incomplete header.

ParameterTypeDescription
biostringUser's biography text
avatarfileProfile photo (multipart upload)
social_linksobjectSocial media links (keys depend on admin configuration)
Field requirements vary. Which fields are required depends on the admin configuration. The app should check the GET status endpoint to determine which fields to show.

Configuration

WordPress Admin Settings

After installing the tbc-profile-completion plugin, navigate to its settings page in the WordPress admin:

SettingDescription
Require BioWhen enabled, users must enter a biography before accessing the app
Require AvatarWhen enabled, users must upload a profile photo
Required Social LinksConfigure which social media fields are required (if any)

Module Manifest

The module manifest in modules/profile-completion/module.ts registers three things:

  • providers — ProfileIncompleteProvider at order 90, wrapping the app to detect the incomplete profile header
  • responseHeaders — maps X-TBC-Profile-Incomplete to the key profileIncomplete, transformed to a boolean
  • routePrefixes — registers /profile-complete as a known route prefix so navigation works correctly
Provider order 90 means the ProfileIncompleteProvider wraps most other providers but runs after lower-order providers (like auth) have initialized. This ensures the user is authenticated before the profile check runs.
