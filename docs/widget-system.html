Widget System

Home screen widgets with drag-and-drop reordering, show/hide toggles, and module registration

Key files: components/home/widgetRegistry.ts (registry), hooks/useWidgetPreferences.ts (preferences), app/(tabs)/index.tsx (home screen)
Widgets are customizable content blocks on the home screen. Core and module widgets register in a unified registry. Users can reorder via drag-and-drop and toggle visibility. Preferences persist in AsyncStorage.

1. Overview

Home Screen (app/(tabs)/index.tsx)
  ├── Welcome Banner (pinned, not in registry)
  ├── Widget List (FlashList or DraggableFlatList)
  │     ├── HomeWidget wrapper (header, icon, "See all")
  │     │     └── Widget Component (receives refreshKey)
  │     └── ... more widgets
  └── Edit Mode Bar (Reset / Done)

Widget Registry (core + modules)
  └── User Preferences (AsyncStorage)
        └── Order + enabled/disabled state

Two rendering modes:

  • Normal mode: FlashList with pull-to-refresh. Long-press a widget header to enter edit mode.
  • Edit mode: DraggableFlatList with drag handles and toggle switches. Separate list component because DraggableFlatList conflicts with RefreshControl.

2. Widget Registration

The WidgetRegistration interface in modules/_types.ts:

interface WidgetRegistration {
  id: string;                    // Stable unique ID (AsyncStorage key)
  title: string;                 // Display title in header
  icon?: keyof typeof Ionicons.glyphMap;  // Header icon
  seeAllRoute?: string;          // Navigation route for "See all"
  featureFlag?: keyof typeof FEATURES;    // Conditional inclusion
  defaultEnabled: boolean;       // Default state for new users
  canDisable: boolean;           // Whether user can toggle
  externalWrapper: boolean;      // true = wrapped in HomeWidget
  component: React.ComponentType<{ refreshKey: number }>;
}

Key Fields

FieldDescription
idMust be stable across app updates. Used as AsyncStorage preference key. Never change once shipped.
featureFlagIf set, widget only included when FEATURES[flag] === true. Excluded entirely when flag is off.
externalWrapperWhen true, HomeWidget provides header, icon, and "See all" link. When false, widget handles its own layout.
seeAllRouteTapping "See all" navigates to this route. If undefined, no link shown.

3. Registry & Merging

components/home/widgetRegistry.ts combines core and module widgets:

const CORE_WIDGETS: WidgetRegistration[] = [
  {
    id: 'my-courses',
    title: 'My Courses',
    icon: 'school-outline',
    seeAllRoute: '/courses',
    featureFlag: 'COURSES',
    defaultEnabled: true,
    canDisable: true,
    externalWrapper: true,
    component: CoursesWidget,
  },
];

export const WIDGET_REGISTRY: WidgetRegistration[] = [
  ...CORE_WIDGETS,
  ...getModuleWidgets(),  // From modules/_registry.ts
];

Module Widget Registration

Modules register widgets in their manifest:

// modules/blog/module.ts
export const blogModule: ModuleManifest = {
  id: 'blog',
  widgets: [{
    id: 'latest-blog',
    title: 'Latest Blog',
    icon: 'newspaper-outline',
    seeAllRoute: '/blog',
    defaultEnabled: true,
    canDisable: true,
    externalWrapper: true,
    component: BlogWidget,
  }],
};
Dev-mode duplicate detection. In __DEV__ mode, modules/_registry.ts checks for duplicate widget IDs across all modules and warns. Production skips this check.

4. User Preferences

hooks/useWidgetPreferences.ts manages widget order and visibility in AsyncStorage:

Storage Format

// AsyncStorage key: tbc_widget_preferences
{
  "version": 1,
  "order": [
    { "id": "my-courses", "enabled": true },
    { "id": "latest-blog", "enabled": true },
    { "id": "latest-youtube", "enabled": false }
  ]
}

Operations

FunctionDescription
reorder(newOrder)Save new order after drag-and-drop
toggleWidget(id)Flip enabled/disabled state
resetToDefaults()Restore all widgets to default order and enabled state

Forward Compatibility

When loading saved preferences, the hook merges them with the current widget registry:

  • New widgets (added since last save) are appended to the end with their defaultEnabled state
  • Removed widgets (no longer in registry) are silently dropped
  • Disabled feature flags exclude widgets entirely
  • Existing order is preserved across app updates

5. Rendering Pipeline

  1. Load preferences from AsyncStorage via useWidgetPreferences()
  2. Merge with current WIDGET_REGISTRY (forward compatibility)
  3. Build WidgetItem[] list (preference + registration config)
  4. Filter: only enabled widgets in normal mode, all widgets in edit mode
  5. Render each widget through HomeWidget wrapper (or raw if externalWrapper: false)

HomeWidget Wrapper

components/home/HomeWidget.tsx provides consistent UI for each widget:

  • Header row with icon + title + "See all" link
  • In edit mode: drag handle (left) + toggle switch (right)
  • Disabled widgets show at 0.4 opacity in edit mode

7. Data Refresh

All widgets receive a refreshKey prop from the home screen. When it increments, widgets re-fetch data.

Widget Data Pattern

function MyWidget({ refreshKey }: { refreshKey: number }) {
  const { data } = useCachedData({
    cacheKey: 'tbc_widget_my_data',
    fetcher: () => api.getMyData(),
    refreshKey,
    refreshOnFocus: false,          // Widgets don't refresh on app focus
    staleTime: WIDGET_STALE_TIME,   // 120 seconds
  });
  // render...
}

Refresh Triggers

TriggerMechanism
Pull-to-refreshsetRefreshKey(prev + 1) — all widgets update
App returns from backgrounduseAppFocus increments refreshKey
Cache eventSpecific widgets listen for events like CACHE_EVENTS.FEEDS
120-second stale time. WIDGET_STALE_TIME (120s) prevents redundant network calls. Rapid pull-to-refresh within 120s uses cached data. Widgets stay mounted on the home screen, so this cooldown avoids unnecessary re-fetches.

9. Welcome Banner

components/home/WelcomeBannerWidget.tsx is a special widget pinned at the top of the home screen:

  • Not in the widget registry — always shown first, not reorderable or toggleable
  • Server-driven: Fetches from /feeds/welcome-banner REST endpoint
  • Dismissible: Users can dismiss for 7 days (or until content changes)
  • Fingerprint-based: Content hash detects changes; changed banner re-shows after dismissal
  • Returns null if disabled by admin or already dismissed within expiry

11. Rules

  1. Widget IDs must be stable. The id field is used as an AsyncStorage key for user preferences. Changing it after release resets user preferences for that widget. Never rename a shipped widget ID.
  2. All widgets must accept refreshKey prop. The home screen controls refresh for all widgets through this prop. Use useCachedData with refreshKey and staleTime: WIDGET_STALE_TIME (120s).
  3. Set refreshOnFocus: false in widget data hooks. The home screen manages focus-based refresh via useAppFocus. If widgets also refresh on focus, you get double fetches.
  4. Use featureFlag for conditional widgets. When a feature flag is off, the widget is excluded from the registry entirely — not just hidden. It won't appear in preferences or edit mode.
  5. Module widgets go in the module manifest, not the core registry. Register via widgets: [...] in your ModuleManifest. The registry merges them automatically via getModuleWidgets().
  6. New widgets auto-append with defaults. When a user updates the app and a new widget exists, it appends to their saved order with defaultEnabled state. No user action needed.
  7. Don't use DraggableFlatList in normal mode. It conflicts with RefreshControl (steals the vertical pan gesture). The home screen uses FlashList for normal mode and only switches to DraggableFlatList in edit mode.
  8. The welcome banner is NOT in the registry. It's hardcoded as the first item in the home screen list. Don't try to register it as a widget — it has special dismissal and fingerprint logic that the registry doesn't support.
  9. Widgets should return null when empty. If a widget has no data to show (e.g., no enrolled courses), return null. The HomeWidget wrapper handles the absence gracefully.
  10. Use WIDGET_STALE_TIME (120s) for all widget fetches. This prevents rapid pull-to-refresh from hammering the API. Don't use shorter intervals for widgets — they stay mounted on the home screen.
