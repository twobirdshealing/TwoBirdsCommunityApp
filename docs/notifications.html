Notifications

2. Notification Types

Defined in types/notification.ts.

AppNotification

interface AppNotification {
  // Content
  content: string;                   // HTML from server
  action: NotificationAction;        // e.g. 'feed/mentioned', 'feed/commented'

  // Navigation
  route?: NotificationRoute;         // { name, params } for app navigation

  // Metadata
  subscriber: NotificationSubscriber; // { id, object_id, user_id, is_read, timestamps }
  xprofile?: XProfile;               // Actor who triggered the notification

  // Computed (added by transformNotification)
  id: number;                        // From subscriber.id
  is_read: boolean;                  // Parsed from subscriber.is_read ("0"/"1")
  created_at: string;                // From subscriber.created_at
  type: NotificationAction;          // Alias for action

  // Multi-reaction data (injected by tbc-multi-reactions plugin)
  tbc_reaction_type?: string;
  tbc_reaction_emoji?: string;
  tbc_reaction_icon_url?: string;

  // Legacy/convenience
  message?: string;                  // HTML-stripped version of content
  title?: string;                    // First line of message
}

NotificationAction Types

ActionTrigger
feed/mentionedUser mentioned in a post
feed/commentedComment on user's post
feed/repliedReply to user's comment
feed/reactedReaction on user's post or comment
profile/followedUser was followed
space/invitedInvited to a space
space/joinedUser joined a space
course/enrolledEnrolled in a course
lesson/completedCompleted a lesson

transformNotification()

Raw API data is transformed via transformNotification() which parses is_read strings, strips HTML for message previews, and maps reaction metadata. All notifications pass through this transform before reaching the UI.

3. Push Notification Setup

Prerequisites

  • Firebase: google-services.json (Android) and GoogleService-Info.plist (iOS) in project root
  • EAS Project ID: Set in app.json → expo.extra.eas.projectId
  • Physical device required: Push tokens can't be obtained on simulators

Foreground Handler

services/push.ts configures how notifications appear when the app is in the foreground:

Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldPlaySound: true,
    shouldSetBadge: false,    // App handles badge manually
    shouldShowBanner: true,
    shouldShowList: true,
  }),
});

Android Notification Channel

Created at runtime during token registration:

Notifications.setNotificationChannelAsync('default', {
  name: 'Default',
  importance: AndroidImportance.MAX,
  vibrationPattern: [0, 250, 250, 250],
  lightColor: '#6366F1',
});

4. Token Registration

services/push.ts handles the full lifecycle.

Registration Flow

  1. Check Device.isDevice (simulators can't get push tokens)
  2. Check/request notification permissions
  3. Get Expo push token via Notifications.getExpoPushTokenAsync({ projectId })
  4. Create Android notification channel (if Android)
  5. Send token to backend: POST tbc-ca/v1/push/device with { token, platform }

When Registration Happens

  • On login: AuthContext calls registerDeviceToken(authToken)
  • On every app startup: Re-registers to ensure fresh token even if device state changed
  • On logout: Calls unregisterDeviceToken() → DELETE tbc-ca/v1/push/device

Push API Endpoints

services/api/push.ts — TBC-CA plugin endpoints (not Fluent Community).

FunctionMethodEndpointPurpose
registerDevice(auth, token, platform)POSTtbc-ca/v1/push/deviceRegister Expo push token
unregisterDevice(auth, token)DELETEtbc-ca/v1/push/deviceRemove device token on logout
getPushSettings(auth)GETtbc-ca/v1/push/settingsGet push notification preferences

Push API uses raw fetch(), not the authenticated client. The push.ts service receives the auth token as a parameter to avoid circular dependency with auth.ts. Don't import from services/auth.ts in push-related files.

How Counts Are Updated

Response headersEvery authenticated API responseX-TBC-Unread-Notifications, X-TBC-Unread-Messages, X-TBC-Cart-Count

OS Badge Sync

// Set app icon badge count
await syncBadgeCount(count);   // Deduplicates (skips if same as last)

// Clear badge on logout
await clearBadgeCount();       // Sets to 0

syncBadgeCount() in services/push.ts calls Notifications.setBadgeCountAsync(count). It deduplicates — if the count hasn't changed since the last call, it's a no-op.

Email Preferences

services/api/emailPrefs.ts

interface EmailUserGlobals {
  com_my_post_mail: 'yes' | 'no';       // Comments on my posts
  reply_my_com_mail: 'yes' | 'no';      // Replies to my comments
  mention_mail: 'yes' | 'no';           // Mentions
  digest_mail: 'yes' | 'no';            // Weekly digest

// Per-space email notification level
type SpacePref = '' | 'admin_only_posts' | 'all_member_posts';
Email prefs are full replacement. POST /notification-preferences replaces ALL preferences, not just the changed ones. Always send the complete user_globals + space_prefs object. Missing keys will be reset to defaults.

10. Server-Side Push Sending

The TBC-CA companion plugin handles all push notification sending via the companion plugins/tbc-community-app/includes/push/ directory.

Push Infrastructure Files

FilePurpose
class-api.phpREST endpoints (device registration, settings)
class-devices.phpDevice token storage (wp_tbc_ca_device_tokens table)
class-preferences.phpUser preference storage (wp_usermeta)
class-hooks.phpEvent hooks — listens to Fluent Community actions
class-firebase.phpExpo Push API integration (sends notifications)
class-registry.phpNotification type registration (defines all types + defaults)
class-log.phpPush delivery logging
class-manual.phpAdmin UI for manual push notifications

Sending Flow

  1. Fluent Community hook fires (e.g., fluent_community/feed/created)
  2. Hook handler in class-hooks.php determines recipients and builds notification data
  3. as_enqueue_async_action() schedules async batch job via Action Scheduler
  4. Async job calls TBC_CA_Push_Firebase::send() for each recipient
  5. Fetches user's device tokens from database
  6. Sends to Expo Push API: POST https://exp.host/--/api/v2/push/send
  7. Invalid tokens (401 from Expo) are automatically removed

Expo Push Payload

{
  "to": "ExponentPushToken[...]",
  "title": "Jane commented on your post",
  "body": "Great article! I really enjoyed...",
  "sound": "default",
  "badge": 5,                    // Unread count for app icon
  "data": {
    "feed_id": 123,
    "space_slug": "general",
    "route": "feed_detail",      // For deep linking
    ...
  },
  "channelId": "default",        // Android only
  "priority": "high"             // Android only
}

Event Hooks

The plugin listens to these Fluent Community events:

  • Feed: created, space feed created, reaction, mention
  • Comments: author notification, mention notification, thread notification
  • Reactions: comment reaction
  • Spaces: joined, join requested, role updated
  • Invitations: invitation created
  • Messaging: new message
  • Pro features: followers, leaderboard, course quizzes (if enabled)

Device Token Management

  • Stored in wp_tbc_ca_device_tokens table (user_id, token, platform, timestamps)
  • Multi-device: one user can have multiple tokens (phone + tablet)
  • Deduplication: existing token updates user_id (handles account switching)
  • Cleanup: tokens older than 90 days automatically removed

12. Rules

  1. Check FEATURES.PUSH_NOTIFICATIONS before any push operations. When disabled, skip token registration, hide push toggles in settings, and don't show push-related UI.
  2. Always pass auth token as parameter to push functions. The services/push.ts and services/api/push.ts files use raw fetch() to avoid circular dependency with auth.ts. Never import from services/auth.ts in push code.
  3. Email prefs are full replacement. When updating email preferences, always send the complete user_globals + space_prefs object. Omitting keys resets them to defaults.
  4. Use UnreadCountsContext for all badge counts. Don't maintain separate unread state — all screens and components read from this single context. It's updated by response headers, startup batch, and mark-as-read APIs.
  5. Sync OS badge after count changes. After updating unreadNotifications in context, call syncBadgeCount(count) to update the app icon badge. It deduplicates automatically.
  6. Transform notifications before use. All raw API notification data must pass through transformNotification(). The API service functions do this automatically — don't transform again in UI code.
  7. Deep link via route field, not legacy fields. Use notification.route (server-provided) for navigation. Legacy fields (feed_id, space_slug) are fallbacks only.
  8. Register device token on every app startup. Don't skip registration if a token was previously obtained — device tokens can change. Re-registering ensures the backend always has the current token.
  9. Push requires physical device. Check Device.isDevice before requesting permissions or tokens. Simulators will always return null.
  10. Use the unified settings hook for the settings screen. useNotificationSettings merges push and email prefs into UnifiedSection[]. Don't build the settings UI directly from the raw APIs.
