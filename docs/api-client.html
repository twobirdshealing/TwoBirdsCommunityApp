1. How It Works

Every authenticated API call flows through a single request() function in services/api/client.ts. This function handles the full request lifecycle:

request(endpoint, config) ↓ NetInfo connectivity check (fail fast if offline) ↓ Get JWT from SecureStore → add Authorization header ↓ fetch() with AbortController timeout (20s default) ↓ Parse JSON (fallback to text on parse failure) ↓ Extract X-TBC-* response headers → callback to React ↓ Response OK? → return { success: true, data } ↓ 401/403 JWT error? → silentRefresh() → retry once ↓ Still failing? → clearAuth() → return { success: false, error }

API services (feeds, spaces, profiles, etc.) are thin wrappers around the client’s convenience methods (get, post, put, patch, del). They handle request building and response transformation but don’t deal with auth, retries, or error normalization.

3. HTTP Client

Convenience Methods

For most API calls, use the convenience methods:

import { get, post, put, patch, del } from '@/services/api/client';

// GET with query params
const feeds = await get<FeedsResponse>('/feeds', { per_page: 20, page: 1 });

// POST with JSON body
const result = await post<CreateResult>('/feeds', { message: 'Hello', type: 'text' });

// PUT with body
const updated = await put<Feed>('/feeds/42', { message: 'Updated' });

// PATCH with body
const patched = await patch<Profile>('/profile/john', { bio: 'New bio' });

// DELETE
const deleted = await del<void>('/feeds/42');

Direct request() for Advanced Use

When you need baseUrl, rawBody, timeout, or includeHeaders, use request() directly:

import { request } from '@/services/api/client';

// FormData upload (rawBody skips JSON.stringify, omits Content-Type)
const formData = new FormData();
formData.append('media', { uri, name, type });
const result = await request<UploadResult>('/media/upload', {
  method: 'POST',
  body: formData,
  rawBody: true,
});

// Custom timeout (e.g., slow upload)
const result = await request<ImportResult>('/import', {
  method: 'POST',
  body: payload,
  timeout: 60_000, // 60 seconds
});

// Different base URL
const config = await request<AppConfig>('/app-config', {
  baseUrl: TBC_CA_URL,
});

4. ApiResponse Type

Every API call returns a discriminated union. Check success first, then access data or error:

// Type definition (services/api/client.ts)
type ApiResponse<T> =
  | { success: true; data: T }
  | { success: false; error: ApiError };

// ApiError shape (types/api.ts)
interface ApiError {
  code: string;       // e.g. 'rest_forbidden', 'tbc_auth_expired_token'
  message: string;    // Human-readable error message
  data: {
    status: number;   // HTTP status code (0 for network/timeout errors)
  };
}

Usage Pattern

const response = await get<FeedsResponse>('/feeds');

// Early-return pattern (recommended — TypeScript narrows correctly)
if (!response.success) {
  throw new Error(response.error.message);
}

// Now TypeScript knows response.data exists
const feeds = response.data.feeds;
Always use early-return for error checking. TypeScript can narrow ApiResponse through if (!response.success) { return/throw } but cannot narrow through else branches. This pattern is used consistently throughout the codebase.

Special Status Codes

CodeMeaning
0Network error or timeout (no HTTP response received)
'network_error'NetInfo reported offline, or fetch threw a network error
'timeout'Request exceeded timeout (AbortController fired)
'parse_error'Server returned non-JSON (e.g., HTML from 502 proxy error)

Auth → React Bridge

Since auth.ts is a plain service (not React), it uses a callback pattern to notify AuthContext when auth is cleared:

// auth.ts — registers callback
export function setOnAuthCleared(callback: (() => void) | null): void { ... }

// AuthContext.tsx — sets the callback on mount
useEffect(() => {
  setOnAuthCleared(() => {
    setUser(null);
    setIsAuthenticated(false);
  });
  return () => setOnAuthCleared(null);
}, []);

6. Silent Token Refresh

When the server returns a 401 or a 403 with a JWT error code, the client automatically attempts to refresh the token and retry the original request.

Flow

Server returns 401 or 403 with JWT code ↓ Is this a retry (_isRetry)? → Yes: clearAuth(), return error ↓ No: call silentRefresh() ↓ Read refresh token from SecureStore ↓ POST /tbc-ca/v1/auth/refresh (direct fetch, NOT through client) ↓ Success? → Store new access token, retry original request with _isRetry=true ↓ Failed? → clearAuth() (triggers logout via onAuthCleared callback)

Key Design Decisions

  • Direct fetch() for refresh — The refresh call bypasses the HTTP client entirely. Using the client would create an infinite loop (refresh fails → tries to refresh again → ...).
  • _isRetry flag — Prevents infinite retry loops. The retry gets one chance; if it also fails with a JWT error, auth is cleared.
  • Concurrent refresh prevention — If multiple requests fail at the same time, only one refresh happens. Others wait on the same refreshInProgress promise.
// Concurrent refresh prevention (auth.ts)
let refreshInProgress: Promise<boolean> | null = null;

export async function silentRefresh(): Promise<boolean> {
  if (refreshInProgress) {
    return refreshInProgress; // Wait for the in-flight refresh
  }
  refreshInProgress = (async () => {
    // ... do refresh ...
  })();
  return refreshInProgress;
}

7. Error Handling

Error Normalization

WordPress and Fluent Community return errors in several different shapes. The client normalizes all of them into a consistent ApiError:

Server ReturnsNormalized To
{ message: "string" }{ message: "string" }
{ message: ["err1", "err2"] }{ message: "err1. err2" }
{ message: { field: ["err"] } }{ message: "err" } (flattened)
{ code: "x", message: "y", data: { status: 403 } }{ code: "x", message: "y", data: { status: 403 } } (passthrough)
"plain string" or null{ code: "error", message: "string" | "An unknown error occurred" }

Network & Timeout Errors

These never reach the server, so they get synthetic error codes:

// No connectivity (NetInfo check)
{ code: 'network_error', message: 'No internet connection', data: { status: 0 } }

// Request timed out (AbortController)
{ code: 'timeout', message: 'Request timed out — ...', data: { status: 0 } }

// Fetch threw (DNS failure, etc.)
{ code: 'network_error', message: error.message, data: { status: 0 } }

// Non-JSON response (e.g., 502 proxy HTML page)
{ code: 'parse_error', message: 'Server returned invalid JSON (HTTP 502)', data: { status: 502 } }

Readable Error Messages (Auth)

services/auth.ts has its own getReadableErrorMessage() that maps error codes and HTTP statuses to user-friendly messages for the login screen. These are separate from the client's error normalization because login happens before the client is available.

8. Response Headers

The TBC Community App plugin sends live counts and status flags in custom response headers on every authenticated response. The HTTP client extracts these and forwards them to React via a callback.

Custom Headers

HeaderTypePurpose
X-TBC-Unread-NotificationsnumberUnread notification count → badge on bell icon
X-TBC-Unread-MessagesnumberUnread message thread count → badge on chat icon
X-TBC-Cart-CountnumberCart item count → badge on cart icon
X-TBC-Maintenance'1' or '0'Site in maintenance mode → show maintenance screen
X-TBC-Profile-Incomplete'1' or '0'User profile is incomplete → prompt to complete
X-TBC-Min-App-VersionstringMinimum app version required → force update screen

How It Works

The callback is registered by _layout.tsx on mount using the same service → React bridge pattern as auth:

// client.ts — callback registration
export function setOnResponseHeaders(
  callback: ((data: ResponseHeaderData) => void) | null
): void { ... }

// _layout.tsx — registers the callback
useEffect(() => {
  setOnResponseHeaders((data) => {
    if (data.unreadNotifications !== undefined) setNotifCount(data.unreadNotifications);
    if (data.unreadMessages !== undefined) setMessageCount(data.unreadMessages);
    if (data.maintenance !== undefined) setMaintenanceMode(data.maintenance);
    // etc.
  });
  return () => setOnResponseHeaders(null);
}, []);
Badges stay live without polling. Every API response piggybacks the latest counts, so badges update on every screen transition or data fetch — no dedicated polling or WebSocket channel needed for counts.

9. Batch API

services/api/batch.ts combines multiple REST requests into a single HTTP call. The TBC Community App WordPress plugin dispatches each request internally and returns all responses at once.

Usage

import { batchRequest, findBatchResponse } from '@/services/api/batch';

const responses = await batchRequest([
  { path: '/tbc-ca/v1/app-config' },
  { path: '/fluent-community/v2/notifications/unread' },
  { path: '/fluent-community/v2/courses?type=enrolled&per_page=5' },
]);

// Find a specific response by path prefix
const config = findBatchResponse<AppConfigResponse>(responses, '/tbc-ca/v1/app-config');
const courses = findBatchResponse<CoursesResponse>(responses, '/fluent-community/v2/courses');

Response Shape

interface BatchResponseItem {
  path: string;                     // The requested path
  status: number;                   // HTTP status (200, 404, etc.)
  body: unknown;                    // Response body (parsed JSON)
  headers: Record<string, string>; // Response headers
}

findBatchResponse() returns the body if status is 2xx, or null otherwise. It matches by path prefix, so '/fluent-community/v2/courses' matches both /courses and /courses?type=enrolled.

The batch is used once per session — on startup, by hooks/useStartupData.ts. See the Caching System doc for details on how it coordinates with useCachedData.

10. Writing API Services

API services are organized by domain in services/api/. Each file exports a named object with methods.

Template

// services/api/myFeature.ts

import { ENDPOINTS } from '@/constants/config';
import { get, post, del } from './client';
import type { ApiResponse } from './client';

interface MyItem {
  id: number;
  title: string;
}

interface MyListResponse {
  items: { data: MyItem[] };
}

export const myFeatureApi = {
  /** Get items with optional pagination */
  getItems(options: { page?: number; per_page?: number } = {}): Promise<ApiResponse<MyListResponse>> {
    return get<MyListResponse>(ENDPOINTS.MY_ITEMS, {
      page: options.page || 1,
      per_page: options.per_page || 20,
    });
  },

  /** Create a new item */
  createItem(data: { title: string }): Promise<ApiResponse<MyItem>> {
    return post<MyItem>(ENDPOINTS.MY_ITEMS, data);
  },

  /** Delete an item */
  deleteItem(id: number): Promise<ApiResponse<void>> {
    return del<void>(`${ENDPOINTS.MY_ITEMS}/${id}`);
  },
};

Some services bypass the HTTP client. Registration, OTP, app config, and push use their own fetch() wrappers because they handle unauthenticated requests or receive tokens as parameters (before auth state is established). These are the only exceptions — all other services use the standard client.

11. Rules

  1. Use the HTTP client for all authenticated requests. Import get, post, put, patch, del from @/services/api/client. Don’t use fetch() directly unless the request is unauthenticated.
  2. Use early-return for error handling. Check if (!response.success) first, then work with response.data. TypeScript can’t narrow through else branches on discriminated unions.
  3. Use ENDPOINTS constants for paths. Don’t hardcode REST paths in service files. Add new paths to constants/config.ts.
  4. Use baseUrl to target non-default APIs. Pass baseUrl: TBC_CA_URL (or other URL constant) in the request config. Don’t construct full URLs manually.
  5. Let the client handle auth and retries. Don’t read tokens or implement retry logic in service files. The client does this automatically.
  6. Use rawBody: true for FormData. Don’t manually set Content-Type for file uploads — rawBody tells the client to skip JSON.stringify and let the browser set the multipart boundary.
  7. Don’t guess API response shapes. Check the actual server response (via curl or dev tools) before writing types. WordPress/Fluent returns different shapes for different endpoints.
  8. Export services as named objects. Use export const myApi = { ... } pattern. This makes imports explicit and tree-shakeable.
  9. Don’t add new custom fetch wrappers. If you need unauthenticated access, check if one of the existing wrappers (tbcPublicRequest in registration.ts, etc.) works. Don’t duplicate the pattern.
  10. Handle status: 0 as "no server response." It means network error, timeout, or offline. Don’t retry automatically — the user likely needs to check their connection.
