Widget System

Home screen widgets with drag-and-drop reordering and module registration

Key files: components/home/widgetRegistry.ts (registry), hooks/useWidgetPreferences.ts (preferences), app/(tabs)/index.tsx (home screen)
Widgets are customizable content blocks on the home screen. Core and module widgets register in a unified registry. Users can long-press to drag and reorder. Preferences persist in MMKV.

1. Overview

Home Screen (app/(tabs)/index.tsx)
  ├── Animated.ScrollView (pull-to-refresh)
  │     └── Sortable.Grid (columns=1, long-press to reorder)
  │           ├── Welcome Banner (fixed-order, not draggable)
  │           ├── Widget Component (wraps itself in HomeWidget)
  │           │     └── HomeWidget (header, icon, "See all")
  │           │           └── Widget content
  │           └── ... more widgets
  │
  └── Widget Registry (core + modules)
        └── User Preferences (MMKV) → order

Each widget owns its HomeWidget wrapper — when there's no data to show, the widget returns null and the entire section (header included) disappears cleanly.

Long-press any widget to drag and reorder. Uses react-native-sortables (Sortable.Grid) with customHandle — the welcome banner is a fixed-order item pinned at the top, all other widgets are draggable. Order persists to MMKV.

2. Widget Registration

The WidgetRegistration interface in modules/_types.ts:

interface WidgetComponentProps {
  refreshKey: number;            // Incremented on pull-to-refresh / app resume
  title: string;                 // Widget title — pass to HomeWidget
  icon?: keyof typeof Ionicons.glyphMap;  // Header icon — pass to HomeWidget
  onSeeAll?: () => void;         // "See all" handler — pass to HomeWidget
}

interface WidgetRegistration {
  id: string;                    // Stable unique ID (preference key)
  title: string;                 // Display title in header
  icon?: keyof typeof Ionicons.glyphMap;  // Header icon
  seeAllRoute?: string;          // Navigation route for "See all"
  featureFlag?: BooleanFeatureKey;          // Conditional inclusion (server-controlled)
  defaultEnabled: boolean;       // Default state for new users
  component: React.ComponentType<WidgetComponentProps>;
  hideKey?: string;              // Server visibility key (hide_menu[])
}

Key Fields

FieldDescription
idMust be stable across app updates. Used as MMKV preference key. Never change once shipped.
featureFlagIf set, widget only included when the server-controlled flag is enabled (checked via useFeatures()). Excluded entirely when flag is off.
seeAllRouteTapping "See all" navigates to this route. If undefined, no link shown.
hideKeyServer visibility key. When this key appears in the hide_menu[] array from /app-config, the widget is excluded entirely (same as disabled feature flag).

3. Registry & Merging

components/home/widgetRegistry.ts combines core and module widgets:

const CORE_WIDGETS: WidgetRegistration[] = [
  {
    id: 'my-courses',
    title: 'My Courses',
    icon: 'school-outline',
    seeAllRoute: '/courses',
    featureFlag: 'courses',
    defaultEnabled: true,

    component: CoursesWidget,
    hideKey: 'courses',
  },
];

export const WIDGET_REGISTRY: WidgetRegistration[] = [
  ...CORE_WIDGETS,
  ...getModuleWidgets(),  // From modules/_registry.ts
];

Helper Functions

FunctionDescription
getAvailableWidgets(features, hideMenu)Filters WIDGET_REGISTRY by two checks: (1) featureFlag must be enabled in the provided FeaturesConfig (widgets without a flag always pass), and (2) hideKey must not appear in the hideMenu array (server-controlled visibility via /app-config).
getCoreWidgetComponentMap()Returns a Record<string, Component> mapping core widget IDs to their components.

Module Widget Registration

Modules register widgets in their manifest:

// modules/yourmodule/module.ts
export const yourModule: ModuleManifest = {
  id: 'yourmodule',
  widgets: [{
    id: 'yourmodule-widget',
    title: 'Your Module',
    icon: 'grid-outline',
    seeAllRoute: '/yourmodule',
    defaultEnabled: true,
    component: YourModuleWidget,
  }],
};

modules/_registry.ts collects all module widgets via getModuleWidgets() and provides getWidgetComponentMap() (module widget ID → component). The home screen merges both maps into a single WIDGET_COMPONENTS lookup:

const WIDGET_COMPONENTS = {
  ...getCoreWidgetComponentMap(),   // from widgetRegistry.ts
  ...getWidgetComponentMap(),       // from _registry.ts (modules)
};
Dev-mode duplicate detection. In __DEV__ mode, modules/_registry.ts checks for duplicate widget IDs across all modules and warns. Production skips this check.

4. User Preferences

hooks/useWidgetPreferences.ts manages widget order and visibility in MMKV:

Storage Format

// MMKV key: tbc_widget_preferences
{
  "version": 1,
  "order": [
    { "id": "my-courses", "enabled": true },
    { "id": "module-widget-a", "enabled": true },
    { "id": "module-widget-b", "enabled": false }
  ]
}

Hook Return Values

Property / FunctionDescription
preferencesCurrent WidgetPreferences object (or null while loading)
isLoadingTrue while loading from MMKV on mount
reorder(newOrder)Save new order after drag-and-drop

Forward Compatibility

When loading saved preferences, the hook merges them with the current widget registry (via getAvailableWidgets(features)):

  • New widgets (added since last save) are appended to the end with their defaultEnabled state
  • Removed widgets (no longer in registry) are silently dropped
  • Disabled feature flags or hidden via hide_menu (both server-controlled via /app-config) exclude widgets entirely — they never appear in preferences
  • Existing order is preserved across app updates

The hook re-runs when features changes (via useFeatures()), so toggling a feature flag in wp-admin takes effect on next app config fetch.

5. Rendering Pipeline

  1. Load preferences from MMKV via useWidgetPreferences()
  2. Get available widgets via getAvailableWidgets(features, hideMenu) (feature-flag + visibility filtering)
  3. Build WidgetItem[] list by matching each preference entry to its registry config. Items with no matching config (removed widgets) are dropped.
  4. Welcome banner prepended as a fixed-order item (not draggable). Disabled widgets filtered out.
  5. Each item renders through renderWidget():
    • Looks up the component from the merged WIDGET_COMPONENTS map
    • Passes title, icon, onSeeAll, and refreshKey as props
    • Wraps in Sortable.Handle — widget wraps itself in HomeWidget internally
    • Long-press activates drag via react-native-sortables

HomeWidget Wrapper

components/home/HomeWidget.tsx provides consistent UI for each widget:

  • Header row with icon + title + "See all" link (configurable label via seeAllLabel prop, defaults to "See all")
  • Stateless wrapper — no internal state, just renders header + children
  • Each widget wraps itself in HomeWidget internally — when a widget returns null (no data), the header hides too

HomeWidget Props

interface HomeWidgetProps {
  title: string;
  icon?: keyof typeof Ionicons.glyphMap;
  seeAllLabel?: string;          // defaults to 'See all'
  onSeeAll?: () => void;
  children: React.ReactNode;
}

6. Drag-and-Drop Reorder

The home screen uses react-native-sortables (Sortable.Grid with columns={1}) for drag-and-drop reordering. No edit mode — long-press any widget to start dragging immediately.

7. Data Refresh

All widgets receive a refreshKey prop from the home screen. When it increments, widgets re-fetch data.

Widget Data Pattern

import { HomeWidget } from '@/components/home/HomeWidget';
import type { WidgetComponentProps } from '@/modules/_types';

function MyWidget({ refreshKey, title, icon, onSeeAll }: WidgetComponentProps) {
  const { data } = useAppQuery({
    cacheKey: 'tbc_widget_my_data',
    fetcher: () => api.getMyData(),
    refreshKey,
    refreshOnFocus: false,          // Widgets don't refresh on app focus
    staleTime: WIDGET_STALE_TIME,   // 120 seconds
  });

  if (!data) return null;           // No data → entire widget (header too) hidden

  return (
    <HomeWidget title={title} icon={icon} onSeeAll={onSeeAll}>
      {/* your widget content */}
    </HomeWidget>
  );
}

Refresh Triggers

TriggerMechanism
Pull-to-refreshsetRefreshKey(prev + 1) — all widgets update
App returns from backgrounduseAppFocus callback increments refreshKey
Cache eventSpecific widgets listen for events like CACHE_EVENTS.FEEDS
120-second stale time. WIDGET_STALE_TIME (120s) prevents redundant network calls. Rapid pull-to-refresh within 120s uses cached data. Widgets stay mounted on the home screen, so this cooldown avoids unnecessary re-fetches.

9. Welcome Banner

components/home/WelcomeBannerWidget.tsx is a special widget pinned at the top of the home screen:

  • Not in the widget registry — always shown first, not draggable
  • Rendered as a fixed-order item inside the Sortable.Grid via Sortable.Handle
  • Server-driven: Fetches from /feeds/welcome-banner REST endpoint via useAppQuery
  • Dismissible: Users can dismiss for 7 days (or until content changes)
  • Fingerprint-based: Content hash (title, description, media, CTA buttons) detects changes; changed banner re-shows after dismissal
  • Returns null if disabled by admin, already dismissed within expiry, or fetch returns no data

11. Rules

  1. Widget IDs must be stable. The id field is used as a MMKV key for user preferences. Changing it after release resets user preferences for that widget. Never rename a shipped widget ID.
  2. All widgets must accept WidgetComponentProps. The home screen passes refreshKey, title, icon, and onSeeAll. Use useAppQuery with refreshKey and staleTime: WIDGET_STALE_TIME (120s). Wrap rendered content in <HomeWidget title={title} icon={icon} onSeeAll={onSeeAll}>.
  3. Set refreshOnFocus: false in widget data hooks. The home screen manages focus-based refresh via useAppFocus. If widgets also refresh on focus, you get double fetches.
  4. Use featureFlag and/or hideKey for conditional widgets. Feature flags are server-controlled (wp-admin → TBC Community App → Features tab) and accessed via useFeatures(). hideKey checks against the hide_menu[] array from /app-config. When either condition excludes a widget, it is removed from the registry entirely — not just hidden. It won't appear in preferences.
  5. Module widgets go in the module manifest, not the core registry. Register via widgets: [...] in your ModuleManifest. The registry merges them automatically via getModuleWidgets().
  6. New widgets auto-append with defaults. When a user updates the app and a new widget exists, it appends to their saved order with defaultEnabled state. No user action needed.
  7. The welcome banner is NOT in the registry. It's rendered as a fixed-order item inside the Sortable.Grid. Don't try to register it as a widget — it has special dismissal and fingerprint logic that the registry doesn't support.
  8. Widgets must return null when empty. If a widget has no data to show, return null before the HomeWidget wrapper. This hides the entire section — header included — so there are no orphaned headers on the home screen.
  9. Use WIDGET_STALE_TIME (120s) for all widget fetches. This prevents rapid pull-to-refresh from hammering the API. Don't use shorter intervals for widgets — they stay mounted on the home screen.
