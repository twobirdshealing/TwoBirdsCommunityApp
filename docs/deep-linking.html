1. How It Works

Three entry points funnel web URLs into Expo Router navigation:

Entry Point Trigger Handler
Universal / App Links User taps a https://site.com/spaces/general link externally Linking.getInitialURL() + addEventListener('url')
Push Notification Tap User taps a notification with routing data Notifications.addNotificationResponseReceivedListener()
In-App Link Tap User taps a link inside rendered HTML (posts, comments) HtmlContent.tsx onPress handler

All three use mapUrlToRoute(url, portalSlug) from utils/deepLinkMapper.ts to convert a web URL into an Expo Router route object. If the URL doesn't match any known route, it opens externally in the browser.

2. Configuration

Custom Scheme

Defined in app.json:

"scheme": "twobirdscommunity"

Allows links like twobirdscommunity://spaces/general. Used mainly for web-to-app redirects. The constant APP_SCHEME in constants/config.ts mirrors this value.

iOS Universal Links

Generated dynamically in app.config.ts:

ios: {
  associatedDomains: [`applinks:${hostname}`],
}

The hostname is extracted from SITE_URL. Requires an apple-app-site-association file on the server (served by WordPress).

Android App Links

Also generated in app.config.ts as intent filters:

android: {
  intentFilters: [{
    action: 'VIEW',
    autoVerify: true,
    data: [
      // Core deep links
      { scheme: 'https', host: hostname, pathPrefix: '/spaces/' },
      { scheme: 'https', host: hostname, pathPrefix: '/u/' },
      { scheme: 'https', host: hostname, pathPrefix: '/courses/' },
      { scheme: 'https', host: hostname, pathPrefix: '/notifications' },
      { scheme: 'https', host: hostname, pathPrefix: '/leaderboard' },
      { scheme: 'https', host: hostname, pathPrefix: '/chat/' },
      // Module deep links (from moduleDeepLinkPaths array)
      ...moduleDeepLinkPaths.map(p => ({ scheme: 'https', host: hostname, pathPrefix: p })),
    ],
    category: ['BROWSABLE', 'DEFAULT'],
  }],
}
Buyer setup: Buyers configure moduleDeepLinkPaths in app.config.ts to add deep link support for their active modules. The core paths are always included.

Typed Routes

Expo Router's typed routes experiment is enabled in app.json:

"experiments": {
  "typedRoutes": true
}

This gives TypeScript autocomplete for route paths when using router.push().

4. Deep Link Mapper

The mapper in utils/deepLinkMapper.ts converts web URLs to app routes in three steps:

  1. Extract path — strips the site origin, handles custom scheme (twobirdscommunity://), and relative paths
  2. Strip portal prefix — removes the Fluent Community portal slug (e.g., /community) to get the community-relative path
  3. Match route — pattern-matches against known app routes

URL-to-Route Mapping Table

Web URL PatternApp Route
//(tabs) (home)
/spaces/(tabs)/spaces
/spaces/{slug}/space/[slug]
/spaces/{slug}/{title}~{id}/feed/[id]
/u/{username}/profile/[username]
/courses/courses/index
/courses/{slug}/courses/[slug]
/courses/{slug}/lessons/{lessonSlug}/courses/[slug]/lesson/[lessonSlug]
/notifications/notifications
/leaderboard/(tabs) (fallback — web-only feature)
Post IDs use tilde format. Fluent Community URLs encode the post ID after a tilde: /spaces/general/my-post-title~123. The mapper extracts 123 using extractPostId() which finds the last ~ and verifies the remainder is numeric.

Return Type

type AppRoute = {
  pathname: string;
  params?: Record<string, string>;
} | null;

Returns null when no route matches, signaling the caller to open the URL externally.

5. URL Listener (Cold & Warm Start)

In app/_layout.tsx, two listeners handle universal links:

// Cold start — URL that launched the app
Linking.getInitialURL().then((url) => {
  if (url) handleDeepLink(url);
});

// Warm start — URL while app is open
const sub = Linking.addEventListener('url', ({ url }) => handleDeepLink(url));
return () => sub.remove();

Auth-Aware Queueing

If the user isn't authenticated when a deep link arrives, it's queued in a pendingDeepLink ref and replayed after login:

const handleDeepLink = useCallback((url: string) => {
  const route = mapUrlToRoute(url, portalSlug);
  if (!route) return;

  if (isAuthenticated) {
    router.push(route);
  } else {
    pendingDeepLink.current = route;  // Queue for after login
  }
}, [portalSlug, isAuthenticated, router]);

// Replay after auth
useEffect(() => {
  if (isAuthenticated && pendingDeepLink.current) {
    const route = pendingDeepLink.current;
    pendingDeepLink.current = null;
    setTimeout(() => router.push(route), 300);
  }
}, [isAuthenticated, router]);
300ms delay: The timeout gives Expo Router time to mount the tab navigator before pushing a deep link route. Without it, navigation can silently fail on cold start.

6. Push Notification Routing

Push notification taps are handled in app/_layout.tsx with a priority chain:

  1. Server-specified route (highest priority) — if data.route is a valid string matching VALID_ROUTE_PREFIXES
  2. Module notification handler — calls handleModuleNotification(data, router); returns true if handled
  3. Fallback parameters:
    • data.feed_id → /feed/[id]
    • data.space_slug → /space/[slug]
    • data.course_slug → /courses/[slug]
    • data.profile_username → /profile/[username]
  4. Default → /notifications screen

Route Validation

Server-specified routes are validated against a prefix allowlist before navigation:

const CORE_ROUTE_PREFIXES = [
  '/(tabs)', '/feed/', '/profile/', '/space/', '/messages',
  '/notifications', '/courses', '/directory', '/bookmarks',
  '/notification-settings', '/webview', '/create-post', '/comments/',
];

const VALID_ROUTE_PREFIXES = [
  ...CORE_ROUTE_PREFIXES,
  ...getModuleRoutePrefixes(),  // e.g. ['/blog', '/bookclub', '/youtube']
];
Always sanitize notification params. All values from push notification data are sanitized via sanitizeParam() which strips everything except a-zA-Z0-9_-. Never pass raw notification data to the router.

Parameter Sanitization

function sanitizeParam(value: unknown): string {
  return String(value ?? '').replace(/[^a-zA-Z0-9_\-]/g, '');
}

7. In-App Link Interception

components/common/HtmlContent.tsx renders HTML from feed posts, comments, and blogs. Links are intercepted and routed in-app when they match:

const renderersProps = useMemo(() => ({
  a: {
    onPress: (_event, href) => {
      const route = mapUrlToRoute(href, portalSlug);
      if (route) {
        onLinkNavigate?.();   // Close bottom sheet if open
        router.push(route);
      } else {
        Linking.openURL(href);  // External link
      }
    },
  },
}), [onLinkNavigate, router, portalSlug]);

The onLinkNavigate callback is used by bottom sheets to dismiss themselves before navigation (e.g., tapping a profile link in a comment sheet).

8. Module Routes

Modules register routes through two mechanisms:

Route Stubs

Each module route needs a thin file stub in app/ that re-exports the module's screen component. Expo Router discovers routes from the file system, so the stub must exist:

// app/bookclub/index.tsx — one-line re-export
export { default } from '@/modules/bookclub/screens/BookListScreen';

Route Prefixes for Validation

Modules register their route prefixes in the manifest for push notification validation:

// modules/bookclub/module.ts
export const bookclubModule: ModuleManifest = {
  id: 'bookclub',
  name: 'Book Club',
  routePrefixes: ['/bookclub'],  // Added to VALID_ROUTE_PREFIXES
  // ...
};

These prefixes are collected by getModuleRoutePrefixes() and merged into the route validation allowlist at load time.

Tab Press Interception

Some modules intercept their tab press to open a WebView instead of navigating to a screen:

// Donate module — tab press opens external payment page in WebView
tab: {
  name: 'donate',
  interceptPress: (router) => {
    router.push({
      pathname: '/webview',
      params: { url: DONATE_URL, title: 'Give' },
    });
  },
}

The app/(tabs)/donate.tsx file exists as a placeholder (never renders) but is required for Expo Router to register the route.

9. Auth Guard & Navigation

The root layout in app/_layout.tsx enforces auth-based redirects:

ConditionAction
Not authenticated & not on auth routeRedirect to /login
Authenticated & profile incompleteRedirect to /profile-complete
Authenticated & on login screenRedirect to /(tabs)

Auth routes are: login, register, forgot-password, webview, profile-complete. These are accessible without authentication.

Maintenance gate: During maintenance mode, auth redirects are suppressed unless the user explicitly initiated login from the maintenance screen (via maintenanceLoginMode flag).

10. WebView Routes

The app/webview.tsx screen provides an authenticated WebView for external content:

router.push({
  pathname: '/webview',
  params: {
    url: 'https://example.com/checkout',
    title: 'Checkout',
    rightIcon: 'cart-outline',     // Optional Ionicons name
    rightAction: 'cart',           // Optional action type
    noAuth: 'true',                // Optional: skip session creation
  },
});
ParameterRequiredDescription
urlYesURL to load
titleYesHeader title text
rightIconNoIonicons name for header right button
rightActionNoAction type (e.g., 'cart' navigates to cart)
noAuthNoSkip authenticated session creation for public pages

The WebView creates an authenticated session via appApi.createWebSession() and injects theme-matching CSS for dark mode. Links inside the WebView load within the WebView itself (no in-app routing interception).

11. Portal Slug Handling

Fluent Community can run at a sub-path (e.g., /community) or at the site root. The mapper handles both:

Portal SlugExample URLCommunity Path
'' (empty — root)/spaces/general/spaces/general
'community'/community/spaces/general/spaces/general
'community'/portal/spaces/general/spaces/general (fallback)

The portalSlug value comes from useAppConfig() which reads it from the server's startup batch response. A fallback to /portal handles legacy URLs from before a slug rename.

Non-community paths return null. If the URL doesn't start with the portal slug (or the fallback), stripPortalPrefix() returns null and the URL is treated as external. This prevents the app from trying to route non-community pages (e.g., /about-us).

13. Rules

  1. All URL-to-route conversion goes through mapUrlToRoute(). Never parse URLs manually in screens or components. The centralized mapper ensures consistent portal slug handling and route matching.
  2. Module routes need both a file stub and a prefix registration. The stub in app/ makes the route exist for Expo Router. The routePrefixes in the module manifest makes it valid for push notification routing.
  3. Never trust push notification data. Always validate routes against VALID_ROUTE_PREFIXES and sanitize parameters with sanitizeParam(). Raw notification data could contain arbitrary strings.
  4. Deep links before auth are queued, not dropped. The pendingDeepLink ref stores the route and replays it after login completes. Don't add a second queueing mechanism.
  5. Post IDs use tilde extraction, not URL segment parsing. Fluent Community encodes IDs as title~123. Use extractPostId() which handles this format. Don't split on / or - to find the ID.
  6. Only same-origin URLs are routed in-app. extractPath() compares the URL's origin against SITE_URL's origin. Links to other domains always open externally.
  7. Add new core deep links in two places. Add the path pattern to matchRoute() in deepLinkMapper.ts AND add the prefix to CORE_ROUTE_PREFIXES in _layout.tsx.
  8. Add new Android deep link paths in app.config.ts. Core paths go in the data array directly. Module paths go in moduleDeepLinkPaths (buyer-configurable). iOS universal links work at the domain level — no per-path config needed.
  9. Use router.push() for deep link navigation, not replace(). Push preserves back navigation so users can return to where they were. Only the auth guard uses replace() (login redirects shouldn't be in the back stack).
  10. The WebView doesn't intercept internal links. Links tapped inside the WebView load within the WebView. If you need in-app routing from WebView content, you'd need to add onNavigationStateChange handling — but currently this isn't needed.
