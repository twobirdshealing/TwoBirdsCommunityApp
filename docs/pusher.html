1. Overview

The app uses Pusher WebSockets for real-time chat. When another user sends a message or reacts to one, the server broadcasts the event through Pusher and the app updates instantly — no polling required.

Pusher is provided by Fluent Community's Fluent Messaging plugin. The app doesn't run its own Pusher server — it connects to the same Pusher/Soketi instance the web portal uses.

Two events, one channel. The entire real-time system handles just two event types (message and reaction) on one private channel per user. Everything else (deletions, new threads, read receipts) uses polling fallbacks. Simple by design.

2. Architecture

Three layers, each with a single responsibility:

LayerFileResponsibility
Service (singleton) services/pusher.ts Pusher client, connection, channel subscription, raw event dispatch to handler Sets
Context (provider) contexts/PusherContext.tsx React lifecycle (connect on auth, disconnect on logout, reconnect on foreground), convenience hooks
Consumers (components/hooks) Various Subscribe via useNewMessageListener / useReactionListener, update local UI state

Data flow

Server broadcasts event via Pusher ↓ Pusher WebSocket delivers to app ↓ services/pusher.ts receives on channel ↓ Iterates handler Set (messageHandlers / reactionHandlers) ↓ Each registered handler fires (multiple consumers) ↓ └ useChatMessages → appends message to chat UI └ Messages screen → updates thread preview + unread count └ TopHeader → bumps unread badge

Handler registry pattern

The service uses two Set<Function> collections as event registries. Components register handlers on mount and unregister on unmount. This decouples the WebSocket layer from React — any number of components can listen without the service knowing about React at all.

// services/pusher.ts — handler pattern
const messageHandlers: Set<MessageHandler> = new Set();

// Components add handlers
export function onNewMessage(handler: MessageHandler): () => void {
  messageHandlers.add(handler);
  return () => messageHandlers.delete(handler);  // cleanup
}

// When event arrives, all handlers fire
userChannel.bind('message', (data: PusherMessage) => {
  messageHandlers.forEach(handler => handler(data));
});

3. Configuration

Pusher credentials live in constants/config.ts:

// constants/config.ts
export const PUSHER_CONFIG = {
  APP_KEY: 'YOUR_PUSHER_APP_KEY',     // SETUP: From Pusher dashboard or Fluent settings
  CLUSTER: 'YOUR_PUSHER_CLUSTER',     // SETUP: e.g. 'us3', 'eu', 'ap1'
  AUTH_ENDPOINT: `${API_URL}/chat/broadcast/auth`,  // Fluent Messaging's auth endpoint
};
ValueWhere to find it
APP_KEY WordPress admin → Fluent Community → Settings → Messaging → Real-time config. Or wp-config.php constant FLUENT_SOCKET_APP_KEY.
CLUSTER Same location. Common values: mt1 (default Soketi), us2, us3, eu, ap1.
AUTH_ENDPOINT Auto-derived from API_URL. Don't change unless using a custom auth proxy.
Fluent Messaging required. Real-time messaging requires the Fluent Messaging add-on (part of Fluent Community Pro). Without it, there's no Pusher config, no channels, and no chat. The app's messaging features gracefully degrade — the Messages tab just won't appear if the FEATURES.MESSAGING flag is off.

4. Connection Flow

1. User logs in → AuthContext sets isAuthenticated = true 2. PusherContext effect fires → calls initializePusher(user.id) 3. initializePusher: a. Fetches JWT from SecureStore b. Creates Pusher client with custom authorizer c. Subscribes to private-chat_user_{userId} d. Pusher calls authorizer → POST /chat/broadcast/auth e. Server validates JWT + channel access → returns auth signature f. Channel subscription succeeds g. Binds 'message' + 'reaction' event listeners 4. Ticker polling starts (keeps last_activity fresh) 5. App is now receiving real-time events

Auth token handling

The authorizer fetches a fresh JWT on every channel auth request, not the token captured at init time. This handles the case where the token was silently refreshed between init and the actual channel subscription:

// services/pusher.ts — custom authorizer
authorizer: (channel) => ({
  authorize: async (socketId, callback) => {
    // Fresh token each time — may have been refreshed since init
    const freshToken = await getAuthToken();
    if (!freshToken) {
      callback(new Error('No auth token'), null);
      return;
    }

    const response = await fetch(PUSHER_CONFIG.AUTH_ENDPOINT, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${freshToken}`,
      },
      body: JSON.stringify({
        socket_id: socketId,
        channel_name: channel.name,
      }),
    });

    const data = await response.json();
    callback(null, data);  // { auth: 'KEY:SIGNATURE' }
  },
}),

5. Channels & Auth

Channel naming

Each user subscribes to one private channel:

ChannelPatternWho subscribes
DM channel private-chat_user_{userId} The user with that ID (only they can authorize)

The server also supports space/group channels (private-chat_space_{spaceId}) for community chat, but the app currently uses only user channels for DMs.

Server-side authorization

The auth endpoint at /wp-json/fluent-community/v2/chat/broadcast/auth validates:

  1. User is authenticated (valid JWT in Authorization header)
  2. User has an active Fluent Community profile (status === 'active')
  3. User can access the channel (for private-chat_user_*: the user ID in the channel name matches the authenticated user)

Returns { auth: "APP_KEY:HMAC_SHA256_SIGNATURE" } — standard Pusher protocol.

6. Events

The server broadcasts two event types. All other data changes (thread creation, message deletion, read receipts) are handled by polling.

message

Fired when someone sends a message in a thread the user is part of.

// PusherMessage type
{
  thread_id: number | string;
  message: {
    id: number;
    thread_id: number | string;
    user_id: number | string;
    text: string;                    // HTML: <div class="chat_text"><p>...</p></div>
    created_at: string;
    xprofile: XProfile;              // Sender info (name, avatar, username)
    meta?: {
      reactions?: Record<string, number[]>;
      reply_to?: number;
      reply_text?: string;
    };
    attachments?: ChatAttachment[];
  };
}

reaction

Fired when someone adds or removes a reaction on a message.

// PusherReaction type
{
  message_id: number;
  thread_id: number | string;
  reactions: Record<string, number[]>;  // { "❤️": [123, 456], "👍": [789] }
}

Reactions use a Discord-style format: each emoji key maps to an array of user IDs who reacted with it. Empty arrays are cleaned up server-side when the last user removes their reaction.

7. Subscribing to Events

Components use convenience hooks from contexts/PusherContext.tsx:

import { useNewMessageListener, useReactionListener } from '@/contexts/PusherContext';

// Listen for new messages
useNewMessageListener((data) => {
  if (data.thread_id === currentThreadId) {
    appendMessage(data.message);
    scrollToBottom();
  }
});

// Listen for reaction changes
useReactionListener((data) => {
  if (data.thread_id === currentThreadId) {
    updateReactions(data.message_id, data.reactions);
  }
});

How the hooks work

Both hooks follow the same pattern:

// contexts/PusherContext.tsx
export function useNewMessageListener(handler: MessageHandler) {
  const { subscribeToMessages } = usePusher();
  const stableHandler = useEffectEvent(handler);  // stable ref, no dep array issues

  useEffect(() => {
    const unsubscribe = subscribeToMessages(stableHandler);
    return unsubscribe;  // auto-cleanup on unmount
  }, [subscribeToMessages]);
}

useEffectEvent (React 19) wraps the handler so it always calls the latest version without triggering effect re-runs. This means you can reference local state in your handler without adding it to deps.

Don't call onNewMessage / onReaction directly. Always use the hooks (useNewMessageListener, useReactionListener). They handle stable refs and cleanup. Direct calls risk handler leaks.

8. Connection Lifecycle

EventWhat happensHandlers cleared?
Auth confirmed initializePusher(userId) — creates client, subscribes to channel No (fresh start)
App backgrounded Pusher's built-in connection management pauses the WebSocket No
App foregrounded reconnectPusher() — checks connection state, reconnects if needed No (preserves handlers)
User switch Disconnects old channel, initializes for new user No
Logout disconnectPusher() + clearHandlers() Yes (full cleanup)
Provider unmount disconnectPusher() only (handlers are React-lifecycle-managed) No

Reconnect cooldown

The reconnect function has a 10-second cooldown to prevent thrashing when the app quickly backgrounds and foregrounds:

// services/pusher.ts
const RECONNECT_COOLDOWN = 10_000;

export async function reconnectPusher(): Promise<boolean> {
  // Skip if connection is fresh (connected less than 10s ago)
  if (connectedAt && Date.now() - connectedAt < RECONNECT_COOLDOWN) {
    return true;
  }

  // Skip full teardown if WebSocket is still alive
  if (pusherClient?.connection.state === 'connected') {
    connectedAt = Date.now();
    return true;
  }

  // Full reconnect: disconnect (keep handlers) → reinitialize
  // ...
}
Handlers survive reconnect. The reconnectPusher function tears down and rebuilds the Pusher client and channel, but the handler Sets are module-level state that persist across reconnections. Components don't need to re-subscribe.

9. Activity Keepalive (Ticker)

The Fluent Messaging server skips Pusher broadcasts to users inactive for more than 5 minutes. This is a server-side performance optimization — it avoids waking sleeping clients.

The app works around this with a ticker poll that keeps xprofile.last_activity fresh:

PropertyValue
EndpointGET /feeds/ticker
Interval3.5–4.5 minutes (randomized to avoid thundering herd)
Fires onInitial auth, each interval, and app foreground
Pauses onApp background
Error handlingSilently ignored (background keepalive, not critical)
Don't remove the ticker. Without it, the server stops sending Pusher events after 5 minutes of apparent inactivity. Messages would still arrive when the user manually refreshes, but real-time delivery would silently stop.

How it works

// hooks/useTickerPolling.ts (simplified)
const MIN_INTERVAL = 210_000;  // 3.5 minutes
const MAX_INTERVAL = 270_000;  // 4.5 minutes

function randomInterval(): number {
  return MIN_INTERVAL + Math.random() * (MAX_INTERVAL - MIN_INTERVAL);
}

// Tick immediately on start, then schedule next tick
const tick = async () => {
  await getTicker(sinceRef.current);
  timerRef.current = setTimeout(tick, randomInterval());
};

// Pause on background, resume + tick immediately on foreground
AppState.addEventListener('change', (state) => {
  if (state === 'active' && !activeRef.current) {
    activeRef.current = true;
    tick();
  } else {
    activeRef.current = false;
    clearTimeout(timerRef.current);
  }
});

10. Server-Side Broadcasting

Understanding the server side helps debug why events might not arrive.

Broadcast flow

User sends message via app ↓ POST /wp-json/fluent-community/v2/chat/messages/{threadId} ↓ Fluent Messaging creates message record in DB ↓ do_action('fluent_messaging/after_add_message', $message) ↓ For each participant in the thread: └ Check last_activity < 5 minutes ago └ If active: PusherHelper::sendMessage( 'private-chat_user_{userId}', 'message', $payload ) ↓ Pusher routes event to subscribed clients

Server-side activity gate

The server only broadcasts to users whose xprofile.last_activity is within 5 minutes of the current time. This is why the ticker polling is essential.

// Fluent Messaging — ChatController.php (simplified)
if (current_time('timestamp') - strtotime($xprofile->last_activity) < 300) {
  PusherHelper::sendMessage(
    'private-chat_user_' . $userId,
    'message',
    ['thread_id' => $thread->id, 'message' => $message]
  );
}

11. Event Consumers

Every component that listens to Pusher events and what it does with them:

ConsumerFileEventBehavior
Chat thread hooks/useChatMessages.ts message Appends message to list if thread_id matches, scrolls to bottom
Chat thread hooks/useChatMessages.ts reaction Updates reactions on specific message by ID
Messages list app/messages/index.tsx message Updates thread preview text, bumps unread count. Refetches if thread is unknown.
Top header components/navigation/TopHeader.tsx message Increments unread message badge (filters out own messages)

Pattern: filter by thread_id

Most consumers check thread_id before processing. This is important because all DM events arrive on a single channel — a handler in Thread A shouldn't process messages for Thread B:

// hooks/useChatMessages.ts
useNewMessageListener((data) => {
  if (data.thread_id !== threadId) return;  // not our thread
  setMessages(prev => [...prev, data.message]);
  scrollToBottom();
});

12. Buyer Setup

For white-label buyers setting up real-time messaging:

  1. Install Fluent Community Pro + Fluent Messaging add-on on WordPress
  2. Go to Fluent Community → Settings → Messaging → enable real-time
  3. Note the App Key and Cluster from the real-time config section
  4. Update constants/config.ts: 
    export const PUSHER_CONFIG = {
  APP_KEY: 'your_app_key_here',
  CLUSTER: 'your_cluster_here',
  AUTH_ENDPOINT: `${API_URL}/chat/broadcast/auth`,
};
  5. Ensure FEATURES.MESSAGING is true in the same file
  6. Build and test — messages should arrive in real-time
Where do the credentials come from? Fluent Community can use Pusher Cloud (you create an app at pusher.com) or Fluent's built-in Soketi service (credentials appear in Fluent admin). Either way, the app key and cluster go in PUSHER_CONFIG.

13. Rules

  1. Always use the hooks (useNewMessageListener, useReactionListener) to subscribe to events. Never call onNewMessage / onReaction directly from components.
  2. Filter by thread_id in your handler. All DM events arrive on a single channel — without filtering, you'll process events for unrelated threads.
  3. Don't remove the ticker polling. Without it, the server stops broadcasting to the app after 5 minutes.
  4. Don't clear handlers on reconnect. Handlers are React-lifecycle-managed. Only clearHandlers() on logout.
  5. Auth endpoint is fixed. It's always {API_URL}/chat/broadcast/auth — provided by Fluent Messaging. Don't create a custom endpoint.
  6. Fresh token per auth. The authorizer fetches a new JWT each time, not the one captured at init. This handles silent token refresh.
  7. Dev logging: Pusher.logToConsole = true is enabled in __DEV__ mode for raw WebSocket traffic. Zero-cost in production.
  8. No new events. If you need to add a new real-time event type, add a new handler Set in pusher.ts, bind it to the channel event, create a convenience hook in PusherContext.tsx, and update this doc.
  9. Reconnect cooldown is 10 seconds. Don't decrease it — rapid reconnects waste bandwidth and can trigger Pusher rate limits.
  10. Connection state logging uses the Pusher tag via createLogger('Pusher'). Filter console output with [Pusher] to debug connection issues.
