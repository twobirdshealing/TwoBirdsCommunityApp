2. The Step Wizard

app/register.tsx renders a multi-step form. Steps 3+ are conditional based on server config and module registrations.

StepComponentPurposeConditional?
1Inline (renderStep1)Name, email, username, passwordAlways shown
2Inline (renderStep2)Custom profile fields + terms checkboxSkipped if no custom fields (terms merged into step 1)
3EmailVerifyStep6-digit email verification codeOnly if email_verification_required: true
4+Module-injected stepsPre-creation verification steps from add-on modulesOnly if module's shouldActivate() returns true

Step 1: Basic Information

FieldValidationNotes
full_nameRequired, non-emptyFrom Fluent Community config
emailRequired, regex: /^[^\s@]+@[^\s@]+\.[^\s@]+$/Server also checks uniqueness
usernameRequired, 4+ chars, /^[a-z0-9_]+$/Lowercase + underscores only
passwordRequired, shown with PasswordStrengthMeter
conf_passwordMust match password (if required by FC)Optional based on FC config

Step 2: Custom Fields + Terms

Renders any custom fields where show_on_signup = true in the WordPress admin. All fields are rendered via the DynamicFormField component. A terms inline checkbox is always present (required). When there are no custom fields, step 2 is skipped entirely and the terms checkbox is merged into step 1.

Step Transitions

After step 1 (or step 2 if custom fields exist), handleSubmitRegistration() sends a POST /register request. The server response determines what happens next:

  • email_verification_required: true + verification_token → advance to step 3 (email verify)
  • Module-specific flags (checked via each step's shouldActivate()) → advance to step 4+ (module steps)
  • success: true with tokens → auto-login via registerAndLogin(), then navigate to /(tabs)

Step Indicator

A dot-based step indicator dynamically adjusts its count based on which steps are active. It accounts for skipped steps (no custom fields, no email verify) so the visual count always matches the actual number of steps the user will see.

All endpoints in services/api/registration.ts. Registration endpoints are public (no auth required).

RegisterResponse

The POST /register response has three possible shapes:

// Success — user created, tokens returned
{
  success: true,
  access_token: "eyJ...",
  refresh_token: "eyJ...",
  user: { id: 42, username: "jane", display_name: "Jane", email: "jane@..." }
}

// Email verification needed (re-submit with code + token)
{
  success: false,
  email_verification_required: true,
  verification_token: "signed-jwt-token",
  message: "Please check your email for a verification code."
}

// Module verification needed (add-on plugins can return custom verification requirements)
// The app checks each registered module's shouldActivate() against this response.
{
  success: false,
  // Add-on-specific fields, e.g.:
  // my_verification_required: true,
  // session_key: "abc123",
  message: "Additional verification required."
}

// Validation errors
{
  success: false,
  message: "Registration failed.",
  errors: {
    email: "This email is already registered.",
    username: "This username is already taken."
  }
}

Password Reset Response Types

interface ForgotPasswordResponse {
  success: boolean;
  email_sent?: boolean;
  message?: string;
}

interface PasswordResetResponse {
  success: boolean;
  message?: string;
}
Re-submission pattern: After email verification or module-injected verification steps, the app re-calls submitRegistration() with the original form data plus the verification data (e.g., __two_fa_code / __two_fa_signed_token for email). Verification extras from all prior steps are accumulated via verificationExtrasRef so each resubmit includes all prior verification params. The server validates the verification and creates the user.

4. Module-Injected Verification Steps

Add-on modules can inject verification steps into the registration wizard using the registrationSteps manifest field. These steps appear after the core email verification step and before user creation completes.

How It Works

  1. User submits the registration form
  2. Server returns a response with add-on-specific fields (via tbc_ca_registration_config hooks)
  3. The app checks each registered module step's shouldActivate(context) where context includes both submitResponse and registrationConfig
  4. If a step activates, the wizard advances to show that module's verification UI (starting at step 4)
  5. After the module step completes, it calls onResubmit(extras) which re-submits registration with the verification data merged
  6. Server validates and creates the user

RegistrationStepRegistration

interface RegistrationStepRegistration {
  id: string;             // Unique step ID (e.g. 'email-verify')
  order: number;          // Sort position (core email-verify = 10)
  phase: 'pre-creation';  // When the step runs
  title: string;          // Step title shown in the register screen header
  component: ComponentType<RegistrationStepProps>;
  shouldActivate: (context: {
    submitResponse: Record<string, any>;
    registrationConfig: RegistrationConfig | null;
  }) => boolean;
}

RegistrationStepProps

Props passed to every module registration step component rendered inline in the register screen:

interface RegistrationStepProps {
  formData: Record<string, any>;
  submitResponse: Record<string, any>;
  submitting: boolean;
  /** Resubmit the registration form with verification extras merged in */
  onResubmit: (extras: Record<string, any>) => Promise<void>;
  /** Signal that this step is complete (navigates to main app) */
  onComplete: () => void;
  /** Go back to the previous step */
  onBack: () => void;
}

See Module System → Registration Types for full details on building custom verification steps.

Server-driven activation. The server response determines which module steps activate. The app never hardcodes which steps to show — it queries each registered module's shouldActivate() function against a context containing both the server response and the app's registration config.

Core Email-Verify Step in Registry

The core email verification step is registered in modules/_registry.ts as a CORE_REGISTRATION_STEPS entry (not a module). It has order: 10 and its shouldActivate checks for submitResponse.email_verification_required and submitResponse.verification_token. However, the register screen renders it directly via the EmailVerifyStep component (not through the registry component), filtering it out of the dynamic module step list by id !== 'email-verify'.

5. Email Verification

Email verification uses Fluent Community's built-in 2FA system. When enabled on the site, registration requires a 6-digit email code before proceeding.

Flow

  1. User submits registration form (step 1 + 2)
  2. Server returns email_verification_required: true + verification_token (signed JWT)
  3. App advances to step 3 (EmailVerifyStep)
  4. User checks email, enters 6-digit code
  5. App re-submits registration with __two_fa_code + __two_fa_signed_token merged into form data
  6. Server validates code → proceeds to module steps or user creation

EmailVerifyStep Component

components/register/EmailVerifyStep.tsx

  • 6-digit code input with number-pad keyboard, digits-only filtering
  • 60-second resend countdown timer
  • Props: email, verificationToken, submitting, onVerify(code, token), onResend(), onBack()
  • onVerify returns a Promise resolving to an error string or null
  • onResend resubmits the form without the email code to trigger a new verification email

Token Expiry Handling

If the email verification token expires mid-flow (e.g., the user is on a module step past step 3), the register screen detects the email_verification_required response, clears accumulated verification extras, resets to step 3, and shows a "session expired" error prompting the user to re-verify.

6. Profile Completion (Add-on)

Profile completion is not a core feature — it is provided by the optional profile-completion add-on module. The module uses the module system's responseHeaders mapping to detect incomplete profiles and the auth guard to redirect users.

How It Works

The profile-completion module registers a responseHeaders mapping in its manifest. The API client intercepts the header on every authenticated response and stores the value. If the profile is incomplete, the module's logic redirects to its screen.

Route Stub

app/profile-complete.tsx is a thin route stub that re-exports the module's screen:

export { default } from '@/modules/profile-completion/screens/LoginGateScreen';

Response Header Pattern

Add-on modules can register response headers via the module system. The API client intercepts these headers on every authenticated response and stores the values for the module to act on. See Module System → ResponseHeaderMapping for details.

Add-on, not core. The profile-completion endpoints (GET /tbc-pcom/v1/status, POST /tbc-pcom/v1/complete) and the login-time gate screen are provided by the profile-completion module and its companion WordPress plugin. They live under their own REST namespace (tbc-pcom/v1) and are not part of the base registration API.

7. Password Recovery

Password recovery endpoints are on the tbc-ca/v1 prefix (tbc-community-app plugin).

Flow

  1. User enters username or email → forgotPassword(login)
  2. Server sends a reset email (returns email_sent: true on success)
  3. User clicks link in email, gets a reset token
  4. User enters new password → resetPassword(token, login, newPassword)

8. Auth Guard & Route Protection

app/_layout.tsx controls which routes are accessible based on auth state.

Auth Group Routes

const inAuthGroup =
  currentSegment === 'login' ||
  currentSegment === 'register' ||
  currentSegment === 'forgot-password' ||
  currentSegment === 'webview';

Navigation Logic

  1. Unauthenticated + NOT in auth group → redirect to /login
  2. Authenticated + on login page → redirect to /(tabs)
Register stays accessible after auth. The /register route is in the auth group, so it remains accessible even after registerAndLogin() completes. This is intentional — module steps (step 4+) may run after the user is already authenticated but still on the register screen.
Profile completion is not in the core auth guard. The profile-complete route is not in the core inAuthGroup list. Profile completion redirection is handled by the profile-completion add-on module's own logic, not by the core layout guard.

Auto-Login After Registration

// In register.tsx, after successful registration:
const result = await submitRegistration(payload);

if (result.success && result.access_token && result.user) {
  // Store tokens + set isAuthenticated = true
  await registerAndLogin(
    result.access_token,
    result.refresh_token || '',
    {
      id: result.user.id,
      username: result.user.username,
      displayName: result.user.display_name,
      email: result.user.email,
    },
  );
  router.replace('/(tabs)');
}

registerAndLogin()

contexts/AuthContext.tsx — stores tokens via storeAuthDirect() from services/auth.ts, sets user state, sets isAuthenticated = true, and registers the device for push notifications.

9. Custom Profile Fields

Custom fields are defined in Fluent Community's native custom profile fields (FC 2.3.0+) and returned dynamically by GET /auth/register/fields. The tbc-community-app plugin maps FC field types to app input types.

RegistrationField Type

interface RegistrationField {
  label: string;
  placeholder?: string;
  instructions?: string;
  type: 'text' | 'email' | 'password' | 'phone' | 'date' | 'number'
      | 'select' | 'radio' | 'gender' | 'textarea' | 'url'
      | 'inline_checkbox' | 'checkbox' | 'multiselect';
  input_type?: string;
  required: boolean;
  options?: string[];      // For select/radio/checkbox/multiselect
  inline_label?: string;   // For inline_checkbox (e.g., terms text)
  step: number;            // Which wizard step to show this field on
}

Field Rendering

All custom fields are rendered via DynamicFormField (components/common/DynamicFormField.tsx). This component maps the type to the appropriate input: text input, date picker, select modal, radio group, checkbox, textarea, etc. The PasswordStrengthMeter component is injected as extraContent for the password field.

Field Storage (WordPress)

  • Definitions managed via Fluent Community admin → Custom Profile Fields (stored in fcom_meta)
  • User values stored in fcom_xprofile.custom_fields JSON column (FC native storage)

10. Social Link Providers

Social link providers (Instagram, YouTube, Facebook, etc.) are configured on the server and define which social profile links users can add to their profiles. These are not social login providers — they are fields for users' social media profile URLs.

Architecture

  • services/api/socialProviders.ts — types, icon mapping, module-level cache with listener pattern
  • hooks/useSocialProviders.ts — React hook that subscribes to provider updates
  • Data arrives via the /app-config endpoint (consumed by ThemeContext), not a separate API call

SocialProvider Type

interface SocialProvider {
  key: string;       // Provider key (e.g. 'instagram', 'youtube', 'fb')
  title: string;     // Display name
  placeholder: string;
  domain: string;    // Base URL for the provider
}

Usage

import { useSocialProviders, getProviderIcon } from '@/hooks/useSocialProviders';

const providers = useSocialProviders();
const icon = getProviderIcon('instagram'); // → 'logo-instagram'

14. Rules

  1. All registration behavior is server-driven. The app reads GET /register/fields on mount and adapts the wizard steps. Don't hardcode which steps to show — let the server config decide.
  2. Re-submit the full form after verification. Email verification and module-injected verification steps are not standalone — the app re-calls submitRegistration() with the original form data plus accumulated verification extras. The server creates the user on the final successful call.
  3. Keep the register route in the auth group. Module steps (4+) may run after registerAndLogin() — the user is already authenticated but still on the register screen.
  4. Registration endpoints are public. The tbcPublicRequest helper uses raw fetch(), not the authenticated API client. Don't add auth headers to registration calls.
  5. Field errors are keyed by field name. Server validation errors come back as errors: { fieldKey: "message" }. Display them inline next to the corresponding field, not as a generic banner. The register screen navigates back to the step containing the errored field.
  6. Custom fields use DynamicFormField. All field types (text, select, radio, date, checkbox, textarea, etc.) are rendered by the same component. Don't create separate field components for registration.
  7. shouldActivate receives a context object. The function signature is shouldActivate(context: { submitResponse, registrationConfig }), not just the submit response. Modules can check both the server response and the app's registration config.
  8. Profile completion is an add-on. Don't add profile completion logic (endpoints, auth guard checks, or context state) to core code. It belongs in the profile-completion module.
