Module System

Self-contained features that plug into the app without touching core code

Source files: modules/_types.ts (type definitions) · modules/_registry.ts (active modules)
Modules register tabs, widgets, menu items, header icons, context providers, tab bar addons, registration steps, response headers, and UI slots via a manifest. Core code reads from the registry — modules never import core layouts or navigation.
Per-module documentation: See docs/modules/ for individual module docs covering setup, configuration, endpoints, and file structure.

1. Overview

The module system lets you add features to the app — bottom tabs, home screen widgets, menu items, header icons, context providers, UI slots, and persistent UI above the tab bar — without editing any core files. Each module is a self-contained folder under modules/ with a manifest that declares what it registers.

Zero core changes. Adding or removing a module requires exactly two edits: one import line and one array entry in modules/_registry.ts. If the module needs deep linking, also add its path prefix to moduleDeepLinkPaths in app.config.ts. Core code reads from the registry at build time — no runtime discovery, no dynamic imports.

The system is static by design. All modules are resolved at import time, producing a flat list of registrations that core code maps over. This keeps the bundle predictable, startup fast, and debugging simple.

2. Architecture

How it fits together

modules/
  _types.ts          ← TypeScript interfaces (ModuleManifest, TabRegistration, etc.)
  _registry.ts       ← Active modules array + derived functions
  yourmodule/
    module.ts        ← Manifest (exports yourModule)
    screens/         ← Screen components
    widgets/         ← Home screen widgets
    services/        ← API calls
    types/           ← TypeScript types
    components/      ← UI components
    contexts/        ← Context providers (if needed)
    hooks/           ← Custom hooks (if needed)

app/
  _layout.tsx        ← Injects module providers, lifecycle, notifications
  (tabs)/
    _layout.tsx      ← Renders module tabs + tab bar addons
    index.tsx        ← Renders module widgets on home screen
    yourmodule.tsx   ← Route stub (one-line re-export, only if module has a tab)
  yourmodule/
    index.tsx        ← Route stub for /yourmodule
    [id].tsx         ← Route stub for /yourmodule/:id

components/
  navigation/
    TopHeader.tsx    ← Renders module header icons
    UserMenu.tsx     ← Renders module menu items
  home/
    widgetRegistry.ts ← Merges core + module widgets

Data flow

Module manifests are imported statically into _registry.ts. The registry exposes getter functions (getModuleTabs(), getModuleWidgets(), etc.) that core files call at render time. Modules never import core layouts — the dependency arrow is always core → registry → module manifest.

3. Module Manifest

Every module exports a ModuleManifest object from its module.ts file. This is the only file the registry imports.

// modules/_types.ts

export interface ModuleManifest {
  // Required identification
  id: string;              // Unique module ID (never change once shipped)
  name: string;            // Human-readable name
  version: string;         // Semver version

  // Module metadata (optional)
  description?: string;       // What the module does
  author?: string;            // Author name
  authorUrl?: string;         // Author website URL
  license?: string;           // License type (e.g. 'Proprietary', 'MIT')
  minAppVersion?: string;     // Minimum app version required

  // Registration slots (all optional)
  tab?: TabRegistration;
  widgets?: WidgetRegistration[];
  providers?: ProviderRegistration[];
  menuItems?: MenuItemRegistration[];
  headerIcons?: HeaderIconRegistration[];
  registrationSteps?: RegistrationStepRegistration[];
  responseHeaders?: ResponseHeaderMapping[];
  slots?: SlotRegistration[];
  tabBarAddon?: React.ComponentType;
  routePrefixes?: string[];

  // Metadata (informational)
  hideMenuKey?: string;       // Server visibility key (inherited by tab)
  companionPlugin?: string;   // WordPress plugin slug
  apiBase?: string;           // REST API base URL

  // Lifecycle hooks
  onInit?: () => void | Promise<void>;
  onCleanup?: () => void;

  // Push notification handler
  notificationHandler?: (data: Record<string, unknown>, router: Router) => boolean;
}

4. Registration Types

TabRegistration

Adds a bottom tab to the tab bar.

interface TabRegistration {
  /** Must match the filename in app/(tabs)/ (e.g. 'shop' → app/(tabs)/shop.tsx) */
  name: string;
  /** Display title shown under the tab icon */
  title: string;
  /** Ionicon name for focused state */
  icon: keyof typeof Ionicons.glyphMap;
  /** Ionicon name for unfocused state */
  iconOutline: keyof typeof Ionicons.glyphMap;
  /** Sort position (core tabs: Home=10, Activity=20, Spaces=30) */
  order: number;
  /** The tab screen component (re-exported from the route stub) */
  component: React.ComponentType;
  /** Server visibility key — tab hidden when this key is in hide_menu[] */
  hideMenuKey?: string;
  /** Intercept tab press (e.g. to open WebView instead of navigating) */
  interceptPress?: (router: any) => void;
  /** Custom tab icon component (overrides icon/iconOutline if provided) */
  tabBarIcon?: (props: { focused: boolean; color: string }) => React.ReactNode;
  /** Override tab color — use a theme token name (e.g. 'error' for red). Defaults to tabBar.active/inactive */
  tabColor?: ColorTokenKey;
}

Example:

tab: {
  name: 'events',
  title: 'Events',
  icon: 'calendar',
  iconOutline: 'calendar-outline',
  order: 40,
  component: EventsTab,
  hideMenuKey: 'events',
}
Route stub required. If your module has a tab, you must create a one-line re-export file in app/(tabs)/. Expo Router requires a real file for each route.

WidgetRegistration

Adds a card to the home screen.

interface WidgetRegistration {
  /** Unique stable ID — never change once shipped (used as MMKV preference key) */
  id: string;
  /** Display title shown in HomeWidget header */
  title: string;
  /** Ionicon name for header */
  icon?: keyof typeof Ionicons.glyphMap;
  /** "See all" route — if undefined, no "See all" link */
  seeAllRoute?: string;
  /** Feature flag key — if set and false, widget is excluded entirely */
  featureFlag?: BooleanFeatureKey;
  /** Default enabled state for new users */
  defaultEnabled: boolean;
  /** Whether the user can disable this widget */
  canDisable: boolean;
  /** Whether the home screen wraps this in HomeWidget (true) or widget handles its own layout (false) */
  externalWrapper: boolean;
  /** The widget component — receives refreshKey prop */
  component: React.ComponentType<{ refreshKey: number }>;
}

Example:

widgets: [{
  id: 'upcoming-event',
  title: 'Upcoming Event',
  icon: 'calendar-outline',
  seeAllRoute: '/events',
  featureFlag: 'events',     // Only shown if events feature is enabled
  defaultEnabled: true,
  canDisable: true,
  externalWrapper: true,     // HomeWidget wrapper handles header/chrome
  component: EventsWidget,
}]
refreshKey prop. The home screen passes a refreshKey number that increments on pull-to-refresh. Use it to trigger data refetch. Return null from your widget if there's no data — never render an empty card.
externalWrapper explained. When true, the home screen wraps your widget in the standard HomeWidget chrome (header bar with title, icon, and "See All" link). When false, your widget component is responsible for its own layout — useful for full-bleed or custom-styled widgets.

ProviderRegistration

Injects a React context provider into the app tree.

interface ProviderRegistration {
  /** Unique ID for the provider */
  id: string;
  /** Sort order — lower numbers wrap outer (rendered first in the tree) */
  order: number;
  /** The context provider component */
  component: React.ComponentType<{ children: React.ReactNode }>;
}

Example:

providers: [{
  id: 'polls-provider',
  order: 50,
  component: PollsProvider,   // Wraps app with polls context (state, modals)
}]
Provider ordering. Lower order values wrap outer in the React tree. If Provider A (order 10) needs to be available inside Provider B (order 20), give A the lower number. Module providers are injected after core providers (Auth, AppConfig, Pusher, BottomSheetModal) so they have access to all core contexts.

MenuItemRegistration

Adds an item to the avatar dropdown menu.

interface MenuItemRegistration {
  /** Unique ID for this menu item */
  id: string;
  /** Display label */
  label: string;
  /** Ionicon name */
  icon: keyof typeof Ionicons.glyphMap;
  /** Route to navigate to (e.g. '/events' or { pathname: '/webview', params: {...} }) */
  route: string | { pathname: string; params: Record<string, string> };
  /** Sort position — core items use 10-90, place yours between them */
  order: number;
  /** Server visibility key — item hidden when this key is in hide_menu[] */
  hideMenuKey?: string;
  /** Override icon color — use a theme token name (e.g. 'error' for red). Defaults to textSecondary */
  iconColor?: string;
}

Example — in-app route:

menuItems: [{
  id: 'events',
  label: 'Events',
  icon: 'calendar-outline',
  route: '/events',
  order: 50,
  hideMenuKey: 'events',
}]

Example — WebView route:

menuItems: [{
  id: 'resources',
  label: 'Resources',
  icon: 'document-text-outline',
  route: {
    pathname: '/webview',
    params: { url: `${SITE_URL}/resources/`, title: 'Resources' },
  },
  order: 55,
}]

HeaderIconRegistration

Adds an icon button to the top header bar.

interface HeaderIconRegistration {
  /** Unique ID for this header icon */
  id: string;
  /** Ionicon name */
  icon: keyof typeof Ionicons.glyphMap;
  /** Route to navigate to */
  route: string | { pathname: string; params: Record<string, string> };
  /** Sort position — core icons: Messages=10, Notifications=20, Cart=30 */
  order: number;
  /** Accessibility label */
  accessibilityLabel?: string;
  /** Server visibility key — icon hidden when this key is in hide_menu[] */
  hideMenuKey?: string;
  /** React hook returning a badge count. Called inside a wrapper component so hook rules are safe. */
  useBadgeCount?: () => number;
}

Example:

headerIcons: [{
  id: 'events-header',
  icon: 'calendar-outline',
  route: '/events',
  order: 25,          // Between Notifications (20) and Cart (30)
  accessibilityLabel: 'Events',
  hideMenuKey: 'events',
  useBadgeCount: () => {
    // Custom hook that returns pending event count
    const { pendingCount } = useEventNotifications();
    return pendingCount;
  },
}]
useBadgeCount is a React hook. The registry wraps each header icon in a component, so useBadgeCount is called in a valid React context. Return 0 to hide the badge. This lets modules display dynamic badge counts without the header knowing about module-specific state.

RegistrationStepRegistration

Injects a step into the multi-step registration wizard. Currently only pre-creation phase is supported — the step runs before the user account is created on the server.

interface RegistrationStepRegistration {
  /** Unique step ID (e.g. 'phone-verify') */
  id: string;
  /** Sort order: core email-verify = 10. Module steps can use any value (e.g. 5, 20, 30). */
  order: number;
  /** Step runs before user account creation (all module steps are pre-creation) */
  phase: 'pre-creation';
  /** Step title shown in the register screen header */
  title: string;
  /** The component to render for this step (rendered inline in the register screen) */
  component: React.ComponentType<RegistrationStepProps>;
  /**
   * Determines if this step is active for the current registration session.
   * Called after form submission with the server response and app registration config.
   */
  shouldActivate: (context: {
    submitResponse: Record<string, any>;
    registrationConfig: RegistrationConfig | null;
  }) => boolean;
}

RegistrationStepProps

Every registration step component receives these props, provided by the register screen:

interface RegistrationStepProps {
  /** Current form data (name, email, etc.) */
  formData: Record<string, any>;
  /** Server's response from the initial registration submission */
  submitResponse: Record<string, any>;
  /** Whether a resubmission is currently in progress */
  submitting: boolean;
  /** Resubmit the registration form with verification extras merged in */
  onResubmit: (extras: Record<string, any>) => Promise<void>;
  /** Signal that this step is complete — advances to the next step */
  onComplete: () => void;
  /** Go back to the previous step */
  onBack: () => void;
}

Example:

registrationSteps: [{
  id: 'phone-verify',
  order: 5,             // Before core email-verify (order 10)
  phase: 'pre-creation',
  title: 'Verify Phone',
  component: PhoneVerifyStep,
  shouldActivate: ({ submitResponse, registrationConfig }) => {
    // Only show if server says phone verification is required
    return !!submitResponse?.phone_verification_required;
  },
}]
shouldActivate context. The function receives an object with submitResponse (the server's response to the registration form submission) and registrationConfig (the app's registration configuration from /app-config). Return true if your step should be shown for this registration session.
Core email-verify step. The registry includes a built-in email verification step at order 10. It activates when the server response includes email_verification_required and verification_token. Place your module steps before (order < 10) or after (order > 10) as needed.

ResponseHeaderMapping

Extracts custom HTTP response headers from every API response and stores them for the module to act on. Useful for server-driven gates (e.g., post-login profile completion checks).

interface ResponseHeaderMapping {
  /** HTTP response header name (e.g. 'X-My-Custom-Header') */
  header: string;
  /** Key name in the header data object (e.g. 'profileIncomplete') */
  key: string;
  /** Transform the raw header string value. Default: pass through as-is. */
  transform?: (value: string) => any;
}

Example:

responseHeaders: [{
  header: 'X-App-Maintenance-Mode',
  key: 'maintenanceMode',
  transform: (value) => value === '1',   // Convert "1"/"0" to boolean
}]
How it works. The API client (services/api/client.ts) checks every response for registered headers. When found, the value (optionally transformed) is stored and accessible to the module. Headers are registered at import time via registerModuleResponseHeaders() in the registry, so extraction is automatic for any authenticated API call.

SlotRegistration

Injects a UI component into a named area inside a core component. Used when a module needs to replace or enhance built-in UI (e.g., replacing the default like button with a multi-reaction picker).

interface SlotRegistration<P = any> {
  /** Unique slot registration ID */
  id: string;
  /** Named slot this fills (e.g. 'feedReactions', 'commentReactions') */
  slot: string;
  /** Priority — lower number wins if multiple modules target the same slot */
  priority: number;
  /** The component to render in this slot — receives typed props from the host */
  component: React.ComponentType<P>;
}

Example:

slots: [
  {
    id: 'my-feed-reactions',
    slot: 'feedReactions',
    priority: 10,
    component: CustomFeedReactionButton,
  },
  {
    id: 'my-feed-breakdown',
    slot: 'feedReactionBreakdown',
    priority: 10,
    component: CustomFeedBreakdown,
  },
]
One winner per slot. If multiple modules register for the same slot, the lowest priority number wins. In dev mode, a warning is logged when slot conflicts occur.

Available slots

Slot NameHost ComponentProps From HostPurpose
feedReactionsFeedCard footerobjectId, hasReacted, onReact, etc.Reaction button (tap/long-press)
feedReactionBreakdownFeedCard footer rightobjectId, reactions, etc.Emoji stack + count summary
commentReactionsCommentItemobjectId, hasReacted, onReact, etc.Comment reaction button
commentReactionBreakdownCommentItemobjectId, reactions, etc.Comment emoji stack + count

When no module fills a slot, the host component renders a default fallback (e.g., a simple heart like button). Slot resolution happens at module scope (outside the component body) via getSlotComponent(), so it's computed once at import time — not on every render.

5. The Registry

The registry (modules/_registry.ts) is where modules are activated. It's the only file you edit when adding or removing a module.

Structure

// modules/_registry.ts

// =============================================
// YOUR MODULES — Import and register below.
// This section is yours to edit. Core updates won't touch it.
// =============================================

import { eventsModule } from './events/module';
import { pollsModule } from './polls/module';

export const MODULES: ModuleManifest[] = [
  eventsModule,
  pollsModule,
  // Add new modules here — comment out to disable
];

// =============================================
// END YOUR MODULES — Everything below is core. Do not edit.
// =============================================

Derived functions

The registry exports getter functions that core code calls. These sort by order, inherit visibility keys, and flatten nested arrays:

FunctionReturnsConsumed by
getModuleTabs()TabRegistration[] sorted by orderapp/(tabs)/_layout.tsx
getModuleWidgets()WidgetRegistration[] flat listcomponents/home/widgetRegistry.ts
getModuleProviders()ProviderRegistration[] sorted by orderapp/_layout.tsx
getModuleMenuItems()MenuItemRegistration[] sorted by ordercomponents/navigation/UserMenu.tsx
getModuleHeaderIcons()HeaderIconRegistration[] sorted by ordercomponents/navigation/TopHeader.tsx
getTabBarAddons()ComponentType[] in module orderapp/(tabs)/_layout.tsx
getModuleRoutePrefixes()string[] flat listapp/_layout.tsx
getModuleResponseHeaders()ResponseHeaderMapping[]services/api/client.ts
getRegistrationSteps()RegistrationStepRegistration[] sorted by order (core + module)app/register.tsx
getSlotComponent<P>(name)ComponentType<P> | nullAny core component with a named slot
getWidgetComponentMap()Record<string, ComponentType<{ refreshKey: number }>>app/(tabs)/index.tsx
initModules()Promise<void>app/_layout.tsx (after auth)
cleanupModules()voidCache registry (on logout)
handleModuleNotification(data, router)booleanapp/_layout.tsx (notification tap)

Key behaviors

  • getModuleTabs() inherits module-level hideMenuKey onto the tab registration (unless the tab sets its own).
  • getRegistrationSteps() merges core steps (email-verify at order 10) with module steps, then sorts by order.
  • getModuleResponseHeaders() also calls registerModuleResponseHeaders() at import time to wire headers into the API client.
  • cleanupModules() is auto-registered with the cache registry so it runs automatically on logout.

Dev-mode validation

In __DEV__ mode, the registry checks for:

  • Duplicate module IDs
  • Duplicate widget IDs
  • Multiple modules filling the same slot (warns; lowest priority wins)
  • Multiple tabBarAddon registrations (warns they'll stack vertically)

6. Integration Points

Here's exactly where each registration type gets rendered in the app:

Bottom tabs

app/(tabs)/_layout.tsx calls getModuleTabs() and renders a Tabs.Screen for each. Module tabs appear after core tabs (Home, Activity, Spaces), sorted by order.

// How core code renders module tabs (simplified from _layout.tsx)
{moduleTabs.map((tab) => {
  const isHidden = tab.hideMenuKey && hideMenu.includes(tab.hideMenuKey);
  return (
    <Tabs.Screen
      key={tab.name}
      name={tab.name}
      options={{
        title: tab.title,
        href: isHidden ? null : undefined,
        tabBarIcon: tab.tabBarIcon
          ? tab.tabBarIcon
          : ({ focused, color }) => (
              <TabIcon name={tab.icon} nameOutline={tab.iconOutline}
                focused={focused} color={color} />
            ),
      }}
      listeners={tab.interceptPress ? {
        tabPress: (e) => {
          e.preventDefault();
          tab.interceptPress!(router);
        },
      } : undefined}
    />
  );
})}
tabColor support. Module tabs can override their active/inactive color using the tabColor field. This accepts a theme token name (e.g. 'error' for red, 'success' for green). The custom tab bar resolves the token against the current theme colors. If not set, the tab uses the standard tabBar.active / tabBar.inactive colors.

Home screen widgets

components/home/widgetRegistry.ts merges core and module widgets into a single WIDGET_REGISTRY array. Core widgets (like Courses) come first, then module widgets from getModuleWidgets(). The getAvailableWidgets(features) helper filters out widgets whose featureFlag is disabled. Users can reorder and toggle widgets in edit mode (long-press). Preferences persist in MMKV.

Avatar dropdown menu

components/navigation/UserMenu.tsx renders module menu items after core items (Profile, Spaces, Bookmarks, etc.), sorted by order.

Top header icons

components/navigation/TopHeader.tsx renders module icons after core icons (Messages, Notifications) and before the avatar button, sorted by order.

Context providers

app/_layout.tsx wraps the app in module providers using reduceRight:

// ModuleProviders component in app/_layout.tsx
function ModuleProviders({ children }) {
  const providers = getModuleProviders();
  return providers.reduceRight(
    (acc, { component: Provider }) => <Provider>{acc}</Provider>,
    children
  );
}

// Position in provider tree:
<ThemeProvider>
  <AuthProvider>
    <AppConfigProvider>
      <PusherProvider>
        <BottomSheetModalProvider>
          <ModuleProviders>       ← Module providers injected here
            <KeyboardProvider>
              <RootLayoutNav />
            </KeyboardProvider>
          </ModuleProviders>
        </BottomSheetModalProvider>
      </PusherProvider>
    </AppConfigProvider>
  </AuthProvider>
</ThemeProvider>
Bottom sheets in module providers. BottomSheetModalProvider wraps ModuleProviders, so module context providers can render bottom sheets (e.g., reaction picker, breakdown modal). If your module provider renders a BottomSheet, it will work without additional setup.

Tab bar addons

app/(tabs)/_layout.tsx renders addon components in a measured container above the tab buttons. The height is tracked via BottomOffsetContext so content below knows where the tab bar starts. Multiple addons stack vertically.

Slots (injected UI)

Core components call getSlotComponent(slotName) at module scope (outside the component body) to resolve which module fills a named UI area. If no module fills it, the host renders its own default. Slot resolution is static — computed once at import time.

// How FeedCard resolves reaction slots (actual pattern from codebase)
import { getSlotComponent } from '@/modules/_registry';

const FeedReactionSlot = getSlotComponent('feedReactions');
const FeedBreakdownSlot = getSlotComponent('feedReactionBreakdown');

function FeedCard({ feed }) {
  // ... in footer:
  {FeedReactionSlot
    ? <FeedReactionSlot objectId={feed.id} hasReacted={...} onReact={...} />
    : <DefaultLikeButton />  // Simple heart icon fallback
  }
}

Response headers

Response header mappings are registered with the API client at import time via registerModuleResponseHeaders(). The client checks every authenticated response for registered headers and stores extracted values. Modules can then read these values to drive UI behavior (e.g., showing a profile completion gate when a header indicates the profile is incomplete).

Push notification routing

app/_layout.tsx calls handleModuleNotification(data, router) when a notification is tapped. Modules are checked in array order — the first to return true claims the notification and stops further routing.

7. Visibility & Feature Flags

Three layers control what the user sees:

LayerMechanismControlled by
Build-time Module is in/out of MODULES array in _registry.ts Developer
Server-side hide_menu[] from /app-config API — matched against hideMenuKey WordPress admin (per-role visibility)
Feature flags featureFlag on widgets — checked against server-controlled flags via useFeatures() WordPress admin (wp-admin → TBC Community App → Features tab)

hideMenuKey inheritance

If a module sets hideMenuKey at the manifest level, it's inherited by the tab registration (unless the tab sets its own). Menu items and header icons set their own hideMenuKey individually.

// Module-level hideMenuKey inherited by tab
export const eventsModule: ModuleManifest = {
  id: 'events',
  hideMenuKey: 'events',    // ← Tab inherits this
  tab: { ... },             // Tab doesn't need its own hideMenuKey
  menuItems: [{
    hideMenuKey: 'events',  // ← Menu items set their own
    ...
  }],
};

Widget user preferences

Users can reorder and toggle widgets on the home screen (long-press to enter edit mode). These preferences are stored in MMKV and persist across sessions. The defaultEnabled and canDisable fields on WidgetRegistration control the initial state and whether toggling is allowed.

8. Lifecycle Hooks

onInit

Called once per app session after authentication is confirmed. Fire-and-forget — errors are caught, logged in dev mode, and never crash the app. Async functions are awaited sequentially (each module's onInit completes before the next runs).

// How the registry calls onInit (from _registry.ts)
export async function initModules(): Promise<void> {
  for (const mod of MODULES) {
    try {
      await mod.onInit?.();
    } catch (e) {
      if (__DEV__) console.warn(`[Modules] ${mod.id} onInit failed:`, e);
    }
  }
}

// Called in app/_layout.tsx after auth is confirmed
useEffect(() => {
  if (isAuthenticated) {
    initModules();
  }
}, [isAuthenticated]);

Use cases: Pre-fetch module data, initialize third-party SDKs, set up listeners.

// Example: pre-fetch events on app start
onInit: async () => {
  const log = createLogger('Events');
  log('Initializing events module');
  await prefetchUpcomingEvents();
},

onCleanup

Called automatically on logout via the cache registry system. The registry self-registers with registerCache({ clearMemory: cleanupModules }), so module cleanup runs alongside core cache clearing. Errors are caught and logged in dev mode.

// How the registry calls onCleanup (from _registry.ts)
export function cleanupModules(): void {
  for (const mod of MODULES) {
    try {
      mod.onCleanup?.();
    } catch (e) {
      if (__DEV__) console.warn(`[Modules] ${mod.id} onCleanup failed:`, e);
    }
  }
}

// Self-registered with cache system
registerCache({ clearMemory: cleanupModules });

Use cases: Clear module-specific MMKV keys, reset in-memory state, disconnect websockets.

// Example: clear cached events on logout
onCleanup: () => {
  storage.remove('tbc_events_cache');
  eventsStore.reset();
},

9. Push Notification Handler

Modules can handle push notification taps by providing a notificationHandler function. When a notification is tapped, the registry iterates through all modules in array order. The first module to return true claims the notification and prevents further routing.

// How the registry dispatches notifications (from _registry.ts)
export function handleModuleNotification(
  data: Record<string, unknown>,
  router: Router
): boolean {
  for (const mod of MODULES) {
    if (mod.notificationHandler?.(data, router)) return true;
  }
  return false;  // No module handled it — falls through to core routing
}

Example:

// In your module manifest
notificationHandler: (data, router) => {
  // Check if this notification is for our module
  if (data.type === 'new_event' && data.event_id) {
    router.push(`/events/${data.event_id}`);
    return true;   // Claimed — stop checking other modules
  }
  return false;     // Not ours — let core or other modules handle it
},
Route prefixes. Set routePrefixes in your manifest (e.g. ['/events']) to ensure push notification deep links for your module pass the route validation check. Without this, the app may reject the route as unrecognized.

12. API Service Patterns

Authenticated endpoints — use request()

For any endpoint that requires a logged-in user, use the centralized request() from @/services/api/client. It handles JWT injection, 401/403 retry, and silent token refresh.

// modules/yourmodule/services/yourApi.ts

import { SITE_URL } from '@/constants/config';
import { request } from '@/services/api/client';

const API_BASE = `${SITE_URL}/wp-json/your-plugin/v1`;

export async function getItems() {
  return request<ItemsListResponse>('/items', { baseUrl: API_BASE });
}

export async function createItem(data: CreateItemPayload) {
  return request<CreateItemResponse>('/items', {
    baseUrl: API_BASE,
    method: 'POST',
    body: data,
  });
}

Public endpoints — fetch() is fine

For truly public endpoints that never need authentication (e.g., proxied external data), you can use fetch() directly. This avoids unnecessary JWT overhead and 401 retry logic.

// modules/yourmodule/services/publicApi.ts

import { SITE_URL } from '@/constants/config';

const PUBLIC_URL = `${SITE_URL}/wp-json/your-plugin/v1`;

export async function getPublicData(limit: number = 5) {
  const response = await fetch(`${PUBLIC_URL}/latest?limit=${limit}`, {
    headers: { 'Accept': 'application/json' },
  });
  if (!response.ok) return null;
  const data = await response.json();
  return data.success ? data : null;
}

WP REST API (standard WordPress)

For standard WordPress endpoints (posts, comments, users), use request() with WP_REST_URL as the base. A thin wpRequest() wrapper keeps it clean:

// modules/yourmodule/services/wpApi.ts

import { WP_REST_URL } from '@/constants/config';
import { request } from '@/services/api/client';

async function wpRequest<T>(endpoint: string, options = {}) {
  return request<T>(endpoint, {
    ...options,
    baseUrl: WP_REST_URL,
    includeHeaders: true,  // Needed for X-WP-Total pagination headers
  });
}

Logging in services

Use createLogger() from @/utils/logger. Follow the convention: log writes, skip reads. See Logging docs for full details.

import { createLogger } from '@/utils/logger';
const log = createLogger('EventsAPI');

export async function createEvent(data) {
  log('createEvent:', data.title);
  return request('/events', { baseUrl: API_BASE, method: 'POST', body: data });
}

14. Rules

Modules never import core layouts or navigation files. The dependency arrow is always core → registry → module. If you need to render something in a specific location, register it via the manifest.
  1. One manifest, one export. Your module.ts exports a single ModuleManifest object. The registry imports only this file.
  2. No barrel files. Import directly from source files. Never create index.ts re-exports.
  3. Import core via @/ aliases. Modules can import from @/components, @/contexts, @/services, @/hooks, @/constants, @/utils. Never import from other modules.
  4. No inter-module dependencies. Module A must never import from Module B. If two of your modules need shared utilities, create a modules/_shared/ folder for common code. The underscore prefix signals “infrastructure, not a module” (like _types.ts and _registry.ts). Core utilities (@/utils, @/hooks, etc.) are always available to all modules.
  5. IDs are permanent. Widget id and module id are used as MMKV keys. Never change them after shipping — you'll lose user preferences.
  6. Route stubs are one-liners. Files in app/ for module routes must be a single re-export: export { default } from '@/modules/...'.
  7. Always use useTheme(). Never hardcode colors. Import useTheme from @/contexts/ThemeContext and use semantic tokens. See Theme System docs.
  8. Use createLogger() for logging. Tag pattern: {Module}API for services, {Module}{Name} for components. See Logging docs.
  9. Use useAppQuery() for data fetching. The centralized hook handles caching, loading states, and refresh. Prefix cache keys with tbc_{module}_.
  10. Widgets return null when empty. Never render an empty card or placeholder. The home screen skips null widgets cleanly.
  11. Include metadata fields. Set description, author, authorUrl, and license in every manifest. The setup dashboard displays these on module cards — they help buyers understand what each module does and where it came from.
  12. Register companion plugin and apiBase. Even though they're informational, they make the manifest self-documenting for other developers.
  13. Don't create new dependencies without checking. The core already has centralized utilities for HTTP, caching, theming, logging, haptics, and date formatting. Check before adding.
