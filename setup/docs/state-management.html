State Management

React Context providers, nesting order, data flow, and service ↔ React bridges

Source directory: contexts/
The app uses React Context for all global state. No external state library (Redux, Zustand, etc.). Screen-level data uses useAppQuery. Contexts handle auth, config, theme, real-time, badges, and UI coordination.

1. Provider Nesting Order

Providers are nested in app/_layout.tsx. Order matters — each provider can only access contexts above it.

GestureHandlerRootView └─ ErrorBoundary └─ PersistQueryClientProvider ← TanStack Query client + MMKV-persisted cache └─ ThemeProvider ← Colors, dark mode, maintenance, update, branding └─ AuthProvider ← Login/logout, user state └─ AppConfigProvider ← Visibility flags, portal slug, socket config, features, registration └─ UnreadCountsProvider ← Badge counts (notifications, messages) └─ PusherProvider ← Real-time WebSocket (needs Auth + AppConfig) └─ BottomSheetModalProvider ← Gorhom bottom sheet host └─ ModuleProviders ← Dynamic providers from module registry └─ KeyboardProvider └─ RootLayoutNav (screens)

Additional providers are nested inside tab screens, not in the root layout:

ProviderLocationWhy Not Root
TabBarProviderapp/(tabs)/_layout.tsxOnly tab screens need scroll-hide behavior
BottomOffsetProviderapp/(tabs)/_layout.tsxOnly tab screens need addon height offset
FeedModalsProvidercomponents/feed/FeedList.tsxWraps each FeedList instance so modals are scoped to the list

2. AuthContext

contexts/AuthContext.tsx

State

FieldTypePurpose
isAuthenticatedbooleanWhether user has a valid JWT
isLoadingbooleanTrue during initial auth check on mount
userAuthUser | nullCurrent user profile (from SecureStore)

Key Methods

const { login, logout, refreshAuth, updateUser, registerAndLogin } = useAuth();
MethodSignaturePurpose
login(username: string, password: string) => Promise<{ success: boolean; error?: string }>Authenticate → store tokens → set user state
logout() => Promise<void>Revoke session → clear all user caches → clear SecureStore
registerAndLogin(accessToken: string, refreshToken: string, userData: AuthUser) => Promise<void>Store tokens directly after registration (no login call needed)
updateUser(updates: Partial<AuthUser>) => Promise<void>Merge partial updates into user state + SecureStore
refreshAuth() => Promise<void>Re-read user from SecureStore (after external changes)

Startup Flow

  1. Check hasStoredAuth() — is there a JWT in SecureStore?
  2. If yes: load stored user, set isAuthenticated = true, re-register push token (if push feature flag enabled)
  3. If no: set isLoading = false (shows login screen)
  4. Register onAuthCleared callback so API client can force logout on refresh failure
Auth is checked client-side only. On mount, the app checks if a JWT exists in SecureStore — it doesn’t validate it with the server. If the token is expired, the first API call triggers silent refresh.

3. ThemeContext

contexts/ThemeContext.tsx

Manages the app’s visual theme and also holds server-synced config data (maintenance mode, update prompts, branding).

State

FieldTypePurpose
theme'light' | 'dark'Current theme mode
isDarkbooleanConvenience: theme === 'dark'
colorsColorThemeResolved color tokens (base + Fluent overrides)
updateUpdateConfig | nullApp update info from server (version, URL, force flag)
maintenanceMaintenanceConfig | nullMaintenance mode (message, bypass flag)
brandingBrandingConfig | nullServer-synced logos and brand colors

Key Methods

MethodPurpose
setTheme(mode)Switch between 'light' and 'dark'; persists to MMKV
refreshAppConfig()Fetch fresh config from /tbc-ca/v1/app-config and apply (theme, maintenance, update, branding, social providers)
setFromBatch(data)Accept pre-fetched AppConfigResponse from _layout.tsx startup batch

Usage

const { colors, isDark, theme, setTheme, update, maintenance, branding } = useTheme();

// Theme colors (most common usage)
<View style={{ backgroundColor: colors.background }}>
  <Text style={{ color: colors.text }}>Hello</Text>
</View>

// Check maintenance mode
if (maintenance?.enabled) { /* show maintenance screen */ }

// Check for updates
if (update?.force) { /* show force update screen */ }

Color Resolution

Colors are resolved by merging the base palette with Fluent Community’s server-sent overrides:

// Base colors (constants/colors.ts)
const base = isDark ? darkColors : lightColors;

// Server overrides (from /app-config → theme_data)
const overrides = fluentOverrides?.[isDark ? 'dark' : 'light'] || {};

// Merged result (with nested tabBar merge)
const colors = { ...base, ...overrides };

See Theme System doc for full token reference and usage rules.

Data Flow

ThemeContext receives config data from two sources:

  1. Startup batch — _layout.tsx calls setFromBatch(data) with pre-fetched config
  2. Background refresh — refreshAppConfig() fetches /tbc-ca/v1/app-config directly

Both paths extract theme colors, maintenance status, update info, branding, and social providers from the same AppConfigResponse.

Cache Persistence

Full app config response is cached to MMKV (synchronous) (tbc_app_config_cache) so the app has correct theme colors on cold start before the batch response arrives. Theme preference (theme_preference) is also persisted.

4. AppConfigContext

contexts/AppConfigContext.tsx

Holds feature visibility flags, the portal slug, socket config, registration capabilities, time format, and feature flags. This context is a passive state holder — it doesn’t fetch data itself. All data is fed in from _layout.tsx via setFromBatch().

State

FieldTypePurpose
visibilityVisibilityConfig | nullFeature visibility from server (hide_menu[] array)
portalSlugstringFluent Community portal slug (used in deep link URL mapping)
socketConfigSocketConfig | nullPusher/Soketi connection config (api_key, cluster, wsHost)
registrationRegistrationConfig | nullRegistration capabilities from server (enabled methods, terms URL, etc.)
is24HourbooleanWhether the site uses 24-hour time format (derived from WordPress time_format setting)
featuresFeaturesConfig | nullFeature flags from server (wp-admin controlled) — null until config loads

Key Methods

MethodPurpose
setFromBatch(data)Accept pre-fetched AppConfigResponse and apply visibility, socket, registration, features, time format, and portal slug

Convenience Hooks

import { useAppConfig, useFeatures } from '@/contexts/AppConfigContext';

// Full context
const { visibility, portalSlug, socketConfig, registration, is24Hour, features } = useAppConfig();

// Feature flags with safe defaults (returns DEFAULT_FEATURES when config hasn't loaded yet)
const features = useFeatures();
if (features.courses) { /* show courses tab */ }

// Check if a menu item should be hidden
const showCourses = !visibility?.hide_menu?.includes('courses');

Cache Persistence

Several fields are cached to MMKV (synchronous) so the app has correct state on cold start before the batch response arrives:

  • tbc_app_visibility_cache — survives logout (site-level, not user-specific)
  • tbc_socket_config_cache — survives logout
  • tbc_registration_config_cache — survives logout
  • tbc_feature_flags_cache — survives logout (via featureFlags utility)

Socket Config Stability

Socket config uses shallow comparison on key fields (api_key, cluster, wsHost) before updating state. This prevents unnecessary Pusher reconnections when the config hasn’t actually changed.

Features Stability

Feature flags use JSON string comparison before updating state, preventing unnecessary re-renders when the same flags arrive from a refresh.

5. UnreadCountsContext

contexts/UnreadCountsContext.tsx

Single source of truth for badge counts displayed on notification bell and chat icon.

State

FieldTypeWritten By
unreadNotificationsnumberStartup batch, response headers, push listener
unreadMessagesnumberStartup batch, response headers, Pusher real-time

Full Interface

const {
  unreadNotifications,
  unreadMessages,
  setUnreadNotifications,   // (count | updater) for direct writes
  setUnreadMessages,        // (count | updater) for direct writes
  resetCounts,              // Zeros all counts (called on logout)
} = useUnreadCounts();

How Counts Stay Updated

  1. Startup batch — populates initial counts from a single API call
  2. Response headers — every authenticated API response piggybacks current counts via X-TBC-* headers (see API Client → Response Headers)
  3. Real-time — Pusher events bump message count; push notifications bump notification count

Usage

const { unreadNotifications, unreadMessages, resetCounts } = useUnreadCounts();

// Display badge
<BadgeIcon count={unreadNotifications} />

// On logout
resetCounts();

6. PusherContext

contexts/PusherContext.tsx

Manages the Pusher WebSocket connection for real-time messaging. See Pusher doc for full details.

State

FieldTypePurpose
isConnectedbooleanWhether the Pusher socket is currently connected

Methods

MethodPurpose
subscribeToMessages(handler)Register a callback for new message events; returns unsubscribe function
subscribeToReactions(handler)Register a callback for reaction events; returns unsubscribe function

Dependencies

Requires both AuthContext (for user ID + auth state) and AppConfigContext (for socket config). Connects only when all three conditions are met: authenticated, user ID available, and server provided socket config. Disconnects on logout.

Key Behavior

  • Reconnects on app foreground (via useAppFocus)
  • Polls /feeds/ticker every ~4 minutes via useTickerPolling to keep last_activity fresh (server skips broadcasts to inactive users)
  • Only activates ticker polling and foreground reconnect when socket config is present

Convenience Hooks

Two convenience hooks are exported from the same file for cleaner consumer code:

import { useNewMessageListener, useReactionListener } from '@/contexts/PusherContext';

// Subscribe to new messages (auto-cleanup on unmount, stable handler via useEffectEvent)
useNewMessageListener((message) => {
  // handle incoming message
});

// Subscribe to reactions
useReactionListener((reaction) => {
  // handle reaction event
});

7. TabBarContext

contexts/TabBarContext.tsx

Controls scroll-aware tab bar hiding. Only available inside app/(tabs)/_layout.tsx.

Interface

Field/MethodTypePurpose
translateYSharedValue<number>Reanimated shared value: 0 = visible, positive = hidden
handleScroll(event) => voidAttach to FlashList/ScrollView onScroll
showTabBar()() => voidForce tab bar visible (e.g. on tab switch)
setLocked(locked)(boolean) => voidLock tab bar visible, disable scroll-hiding

How It Works

Tab bar visibility is driven by a Reanimated SharedValue. Scroll direction is detected via plain JS onScroll (FlashList compatible), and the animation runs on the UI thread.

const { handleScroll, showTabBar, setLocked } = useTabBar();

// Attach to any ScrollView or FlashList
<FlashList onScroll={handleScroll} />

// Force visible (e.g., on tab switch)
showTabBar();

// Lock visible during edit mode
setLocked(true);

Scroll Logic

  • Scroll down past threshold (10px) → hide tab bar
  • Scroll up past threshold → show tab bar
  • Near top of list → always show
  • Locked → always show, ignore scroll events

8. BottomOffsetContext

contexts/BottomOffsetContext.tsx

Tracks extra height above the tab bar from module addons (e.g., mini audio player). Tab content screens use this to set correct bottom padding.

Exported Hooks

HookReturnsPurpose
useBottomOffset()numberRaw addon height (0 when no addon)
useSetAddonHeight()(h: number) => voidSetter — only used by the tab bar layout
useTabContentPadding()numbertab bar height + safe area + addon height + spacing

Usage

import { useTabContentPadding } from '@/contexts/BottomOffsetContext';

// Returns: tab bar height + safe area + addon height + spacing
const bottomPadding = useTabContentPadding();

<FlatList contentContainerStyle={{ paddingBottom: bottomPadding }} />

How Addon Height Is Set

The tab bar layout measures the addon area via onLayout and calls setAddonHeight(). When no addon is rendered, height is 0. The setter uses stable-reference guard (prev === h ? prev : h) to avoid unnecessary re-renders.

9. FeedModalsContext

contexts/FeedModalsContext.tsx

Hoists all feed card modals (menu, media viewer, report) into a single provider. Without this, every FeedCard in a FlashList would render its own modals.

Where It Lives

FeedModalsProvider wraps FeedList directly (not in the root layout). Each feed list has its own provider instance. It’s also used in app/feed/[id].tsx for the single-feed detail view.

10. Module Providers

Modules can register their own context providers via the module manifest. These are dynamically nested inside ModuleProviders in app/_layout.tsx.

// modules/yourmodule/module.ts
export const yourModule: ModuleManifest = {
  providers: [{ component: YourModuleProvider }],
  // ...
};

Providers are nested from outside-in based on registration order using reduceRight. Module providers have access to all core contexts (Auth, Theme, AppConfig, etc.) since they’re nested inside them.

Module providers are automatically included. Register them in the module manifest and they’ll be nested in the root layout. No need to edit _layout.tsx.

12. Service ↔ React Bridges

Several non-React services need to communicate with React contexts. Since services can’t call hooks, they use a callback registration pattern:

ServiceRegistration FunctionConsumerPurpose
services/auth.tssetOnAuthCleared(callback)AuthContextForce logout when silent refresh fails
services/api/client.tsaddResponseHeaderListener(listener)_layout.tsxExtract badge counts + maintenance from response headers; returns unsubscribe function

Pattern A — Single Callback (1:1)

Used by setOnAuthCleared where only one consumer exists:

// In the service (non-React):
let callback: ((data: SomeData) => void) | null = null;

export function setOnSomething(cb: typeof callback): void {
  callback = cb;
}

// Inside the service, when an event occurs:
if (callback) callback(data);

// In the React component (registers on mount, cleans up on unmount):
useEffect(() => {
  setOnSomething((data) => {
    setContextState(data);
  });
  return () => setOnSomething(null);
}, []);

Pattern B — Multi-Listener (1:N)

Used by addResponseHeaderListener where multiple consumers (core + modules) need the same events:

// In the service (non-React):
const listeners = new Set<(data: SomeData) => void>();

export function addSomethingListener(listener: (data: SomeData) => void): () => void {
  listeners.add(listener);
  return () => listeners.delete(listener);  // returns unsubscribe
}

// Inside the service, when an event occurs:
listeners.forEach(fn => fn(data));

// In the React component:
useEffect(() => {
  const unsubscribe = addSomethingListener((data) => {
    setContextState(data);
  });
  return unsubscribe;
}, []);
Why not events or observables? Both patterns are the simplest approach for their use case. Pattern A is ideal for 1:1 communication between a singleton service and a single React consumer. Pattern B supports multiple listeners (e.g., modules subscribing to response headers) with automatic cleanup via the returned unsubscribe function.

13. Rules

  1. Use useAppQuery for screen data, contexts for global state. Don’t create a context to hold data that only one screen uses. Contexts are for state shared across multiple screens or the entire app.
  2. Don’t add new contexts without checking existing ones. The current set covers auth, config, theme, badges, real-time, and UI coordination. Most new features don’t need a new context.
  3. Module providers go in the module manifest. Don’t edit _layout.tsx to add module providers. Register them via providers: [{ component: MyProvider }] in the module’s module.ts.
  4. Use the callback bridge pattern for service → React communication. Don’t import React contexts into plain services. Register a callback from the React side.
  5. Always throw in useMyContext() hooks. All context hooks should throw an error if used outside their provider. This prevents silent failures.
  6. Memoize context values. Wrap the value prop in useMemo to prevent unnecessary re-renders in consumers. All existing contexts follow this pattern.
  7. Keep contexts in contexts/. Module-specific contexts live in modules/{name}/contexts/. Don’t scatter context files across the project.
  8. Never import lightColors/darkColors directly. Always use const { colors } = useTheme(). Direct imports bypass Fluent overrides. See Theme System rules.
  9. Use useUnreadCounts() for badge counts. Don’t fetch notification counts independently. The context is the single source of truth, updated by startup batch + response headers + real-time events.
  10. Use useTabContentPadding() for bottom padding in tab screens. Don’t manually calculate tab bar + safe area + addon height. The hook handles it.
  11. Use useFeatures() for feature flag checks. Returns safe defaults until config loads, so screens never need null guards on feature flags.
