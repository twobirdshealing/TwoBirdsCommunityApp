2. Notification Types

Defined in types/notification.ts.

AppNotification

interface AppNotification {
  // Content
  content: string;                   // HTML from server
  action: NotificationAction;        // e.g. 'feed/mentioned', 'feed/commented'

  // Navigation
  route?: NotificationRoute;         // { name, params } for app navigation

  // Metadata
  subscriber: NotificationSubscriber; // { id, object_id, user_id, is_read, timestamps }
  xprofile?: XProfile;               // Actor who triggered the notification

  // Computed (added by transformNotification)
  id: number;                        // From raw.id (fallback: subscriber.object_id)
  is_read: boolean;                  // Parsed from subscriber.is_read ("0"/"1"/true)
  created_at: string;                // From subscriber.created_at
  type: NotificationAction;          // Alias for action

  // Multi-reaction data (injected by tbc-multi-reactions plugin)
  tbc_reaction_type?: string;
  tbc_reaction_emoji?: string | null;
  tbc_reaction_icon_url?: string | null;

  // Legacy/convenience
  message?: string;                  // HTML-stripped version of content
  title?: string;                    // First line of message
  action_url?: string | null;        // Not used (route is used instead)
  actor_id?: number | null;
  object_id?: number | null;
  object_type?: string | null;
  read_at?: string | null;
}

NotificationAction Types

ActionTrigger
feed/mentionedUser mentioned in a post
feed/commentedComment on user's post
feed/repliedReply to user's comment
feed/reactedReaction on user's post or comment
profile/followedUser was followed
space/invitedInvited to a space
space/joinedUser joined a space
course/enrolledEnrolled in a course
lesson/completedCompleted a lesson

transformNotification()

Raw API data is transformed via transformNotification() which parses is_read strings, strips HTML for message previews, and maps reaction metadata. All notifications pass through this transform before reaching the UI.

Prerequisites

  • Firebase: google-services.json (Android) and GoogleService-Info.plist (iOS) in project root
  • EAS Project ID: Set in app.json → expo.extra.eas.projectId

4. Token Registration

services/push.ts handles the full lifecycle.

Registration Flow

  1. Check Device.isDevice (simulators can't get push tokens)
  2. Check/request notification permissions
  3. Get Expo push token via Notifications.getExpoPushTokenAsync({ projectId })
  4. Create Android notification channel (if Android)
  5. Send token to backend: POST tbc-ca/v1/push/device with { token, platform }

When Registration Happens

  • On login: AuthContext calls registerDeviceToken(authToken)
  • On every app startup: Re-registers to ensure fresh token even if device state changed
  • On logout: Calls unregisterDeviceToken() → DELETE tbc-ca/v1/push/device

Push API Endpoints

services/api/push.ts — TBC-CA plugin endpoints (not Fluent Community).

6. Badge Counts

UnreadCountsContext

contexts/UnreadCountsContext.tsx — single source of truth for all badge counts.

const {
  unreadNotifications,         // number
  unreadMessages,              // number
  setUnreadNotifications,      // setter (accepts number or updater function)
  setUnreadMessages,           // setter (accepts number or updater function)
  resetCounts,                 // clear all on logout
} = useUnreadCounts();

How Counts Are Updated

SourceWhenMechanism

OS App Icon Badge

The BadgeSync component in app/_layout.tsx reactively syncs the OS badge whenever either count changes:

// BadgeSync — isolated component so count changes don't re-render the layout
function BadgeSync() {
  const { unreadNotifications, unreadMessages } = useUnreadCounts();
  useEffect(() => {
    syncBadgeCount(unreadNotifications + unreadMessages);
  }, [unreadNotifications, unreadMessages]);
  return null;
}

syncBadgeCount() in services/push.ts calls Notifications.setBadgeCountAsync(count). It deduplicates — if the count hasn't changed since the last call, it's a no-op.

Platform Differences

PlatformHow Badge WorksWhat We Control
iOSsetBadgeCountAsync() directly sets the red number on the app icon. Push payload badge field also sets it when app is killed.Full control — badge = notifications + messages, always accurate.
Android (Samsung, Pixel)Badge count = number of pending notifications in the notification shade. setBadgeCountAsync() is ignored by Samsung — it uses its own notification-based counting.Limited — badge appears when push notifications arrive and clears when the user dismisses them. In-app badges (TopHeader) are always accurate regardless.
Android (other launchers)Varies by manufacturer. Some support setBadgeCountAsync(), many don't.Best-effort. No reliable cross-manufacturer badge API exists on Android.
Why Samsung badges disappear when notifications are dismissed: Samsung's One UI ties the app icon badge directly to pending notifications in the shade — not to setBadgeCountAsync(). Apps like Gmail maintain persistent badges because they use native Android foreground services, which isn't available in Expo's managed workflow. This is a known platform limitation, not a bug.

8. Deep Linking from Notifications

Push notification taps are handled in app/_layout.tsx via Notifications.addNotificationResponseReceivedListener. In-app notification taps are handled in app/notifications.tsx via the navigateToRoute() function.

Push Notification Tap — Route Resolution Priority

When a push notification is tapped (app/_layout.tsx), the data payload is checked in this order:

  1. data.route (string) — server-provided route path (e.g. /feed/123), validated against known route prefixes. Highest priority.
  2. Module handler — handleModuleNotification(data, router) lets modules claim the notification.
  3. Legacy fields — feed_id, space_slug, course_slug, profile_username.
  4. Default — navigate to /notifications screen.

In-App Notification Tap — Route Object Navigation

When a notification is tapped from the notification list (app/notifications.tsx), the notification.route object is used:

Route NameApp Path
space_feed/feed/{feed_slug}
feed, feed_detail/feed/{id|slug|feed_slug}
space, space_detail/space/{slug|space}
profile, user_profile/profile/{username|user}
course, course_detail/courses/{slug|course}

If no route matches, falls back to navigating to the actor's profile (via notification.xprofile.username).

10. Server-Side Push Sending

The TBC-CA companion plugin handles all push notification sending via the companion plugins/tbc-community-app/includes/push/ directory.

Push Infrastructure Files

Sending Flow

  1. Fluent Community hook fires (e.g., fluent_community/feed/created)
  2. Hook handler in class-hooks.php determines recipients and builds notification data
  3. as_enqueue_async_action() schedules async batch job via Action Scheduler
  4. Async job calls TBC_CA_Push_Firebase::send() for each recipient
  5. Fetches user's device tokens from database
  6. Sends to Expo Push API: POST https://exp.host/--/api/v2/push/send
  7. Invalid tokens (401 from Expo) are automatically removed

  "to": "ExponentPushToken[...]",
  "title": "Jane commented on your post",
  "body": "Great article! I really enjoyed...",
  "sound": "default",
  "badge": 5,                    // Unread count for app icon
  "data": {
    "feed_id": 123,
    "space_slug": "general",
    "route": "/feed/123",        // For deep linking (string path)
    ...

}

Event Hooks

The plugin listens to these Fluent Community events:

  • Feed: created, space feed created, reaction, mention
  • Comments: author notification, mention notification, thread notification
  • Reactions: comment reaction
  • Spaces: joined, join requested, role updated
  • Invitations: invitation created
  • Messaging: new message
  • Pro features: followers, leaderboard, course quizzes (if enabled)

Device Token Management

  • Stored in wp_tbc_ca_device_tokens table (user_id, token, platform, timestamps)
  • Multi-device: one user can have multiple tokens (phone + tablet)
  • Deduplication: existing token updates user_id (handles account switching)
  • Cleanup: tokens older than 90 days automatically removed

12. Rules

  1. Check features.push_notifications (via useFeatures()) before any push operations. This flag is server-controlled (wp-admin → TBC Community App → Features tab). When disabled, skip token registration, hide push toggles in settings, and don't show push-related UI.
  2. Always pass auth token as parameter to push functions. The services/push.ts and services/api/push.ts files use raw fetch() to avoid circular dependency with auth.ts. Never import from services/auth.ts in push code.
  3. Email prefs are full replacement. When updating email preferences, always send the complete user_globals + space_prefs object. Omitting keys resets them to defaults.
  4. Use UnreadCountsContext for all badge counts. Don't maintain separate unread state — all screens and components read from this single context. It's updated by response headers, startup batch, Pusher events, push notification listener, and mark-as-read APIs.
  5. Never call syncBadgeCount() directly. The BadgeSync component in _layout.tsx reactively syncs the OS badge whenever unreadNotifications or unreadMessages changes. Just update the context — the badge follows automatically.
  6. OS badge = notifications + messages. The badge formula is unreadNotifications + unreadMessages. On iOS this sets the app icon badge directly. On Android (Samsung), the badge is tied to pending notifications in the shade — setBadgeCountAsync() is ignored.
  7. Transform notifications before use. All raw API notification data must pass through transformNotification(). The API service functions do this automatically — don't transform again in UI code.
  8. Deep link via route field, not legacy fields. Push notifications use a string data.route path. In-app notification objects use a route object with { name, params }. Legacy fields (feed_id, space_slug) are fallbacks only.
  9. Register device token on every app startup. Don't skip registration if a token was previously obtained — device tokens can change. Re-registering ensures the backend always has the current token.
  10. Push requires physical device. Check Device.isDevice before requesting permissions or tokens. Simulators will always return null.
  11. Use the unified settings hook for the settings screen. useNotificationSettings merges push and email prefs into UnifiedSection[]. Don't build the settings UI directly from the raw APIs.
