Feed System

Posts, reactions, comments, sticky posts, surveys, bookmarks, and media in the community feed.

Key files: services/api/feeds.ts (feed API), services/api/comments.ts (comment API), types/feed.ts & types/comment.ts (types), components/feed/ (all feed UI), hooks/useFeedReactions.ts & hooks/useFeedActions.ts (shared logic), contexts/FeedModalsContext.tsx (hoisted modals).

The feed is the core content surface. Posts come from the Fluent Community API. Multi-reactions are handled by the tbc-multi-reactions companion plugin.

Each feed screen follows the same pattern:

  1. Fetch feeds via useCachedData (stale-while-revalidate from AsyncStorage)
  2. Pass feeds + setFeeds to useFeedReactions and useFeedActions
  3. Render with FeedList → FeedCard components
  4. All modals (menu, picker, breakdown, media, report) are hoisted into FeedModalsProvider — one instance per screen, not per card

7. Multi-Reaction System

Powered by the tbc-multi-reactions companion plugin. Fluent Community core only understands "like" — the plugin adds type tracking, custom emoji, and breakdown data.

Reaction Config

hooks/useReactionConfig.ts fetches GET TBC_MR_URL/config once per session. Shared via module-level cache (not Context).

interface ReactionConfig {
  id: string;              // 'like', 'love', 'laugh', 'wow', 'sad', 'angry', 'custom_123'
  name: string;            // Display name
  emoji: string;           // '👍', '❤️', etc.
  icon_url: string | null; // Custom icon URL (null = use emoji)
  color: string;           // Hex color
  order: number;           // Sort order in picker
}

interface DisplayConfig {
  count: number;           // How many emoji to show in summary (default 5)
  overlap: number;         // Negative margin px for stacked icons
  stroke: number;          // Border width on summary icons
}

Reaction Flow

  1. Tap reaction button — toggles the default reaction (like). If already reacted with the same type, removes it.
  2. Long-press reaction button — opens ReactionPicker popover with all configured types.
  3. Pick a type — if same type as current, removes. If different type, calls swapReactionType(). If no existing reaction, calls reactToFeed().
  4. Optimistic update — useFeedReactions immediately updates the feed state (count, breakdown, user reaction).
  5. API call — fires in background via optimisticUpdate(). On failure, reverts to snapshot.
  6. Reconciliation — after successful API call, reconcileReactionBreakdown() fetches server-accurate counts and silently updates state.
How the header trick works: Feed reactions always send react_type: 'like' to Fluent Community (it only understands "like"). The actual type is sent in the X-TBC-Reaction-Type header, which the tbc-multi-reactions plugin intercepts and stores.

useFeedReactions Hook

hooks/useFeedReactions.ts — shared across all feed screens.

const handleReact = useFeedReactions(feeds, setFeeds);

// Usage: handleReact(feedId, 'love');
// Determines add/remove/swap automatically from current state

Optimistic Breakdown Updates

utils/reactionHelpers.ts provides two functions:

FunctionPurpose
updateBreakdownOptimistically(breakdown, type, action, oldType, getReaction)Returns a new reaction_breakdown[] reflecting add/remove/swap. Increments/decrements counts, adds new entries from config, removes entries at count 0.
reconcileReactionBreakdown(objectType, objectId, setState)Non-blocking fetch of server-accurate breakdown. Silently merges into state. Failure is ignored (optimistic UI is already correct).

8. Comments

Loading Comments

getComments(postId) returns a flat array of all comments plus an optional sticky_comment. Comments with non-null parent_id are replies.

Nested Replies

Comments use parent_id to indicate hierarchy. The UI renders replies with left-border indentation under their parent. Only one level of nesting is supported in the UI (replies to replies appear at the same indent level).

Comment Editor

CommentSheet uses a 10tap rich text editor with toolbar for formatting, media attachment, and GIF picker. When editing an existing comment, the HTML content is converted to markdown via htmlToMarkdown() utility for the editor input.

Comment Actions

  • Create: createComment(postId, { comment, content_type?, parent_id?, media_images? })
  • Edit: updateComment(postId, commentId, { comment, content_type? })
  • Delete: deleteComment(postId, commentId) with confirmation dialog
  • React: reactToComment(postId, commentId, hasReacted, type) — state toggle format
  • Pin: pinComment(postId, commentId, pin) — top-level only, auto-unpins previous (FC 2.2.01+)

Comment Images

Images can appear in multiple locations within the comment meta object:

  • meta.media_images[] — dedicated array
  • meta.media_items[] — generic media array
  • meta.media_preview.image — single preview image

The UI checks all three locations and displays any found images below the comment text.

9. Sticky / Pinned Posts

Pinned posts appear at the top of the feed, above regular posts.

API

// Pin a post
feedsApi.toggleSticky(feedId, true);
// → PATCH /feeds/{id} with { is_sticky: 1, query_timestamp: Date.now() }

// Unpin a post
feedsApi.toggleSticky(feedId, false);
// → PATCH /feeds/{id} with { is_sticky: 0, query_timestamp: Date.now() }

Response Handling

getFeeds() returns sticky posts in a separate sticky field (not mixed into feeds.data). Feed screens merge them at the front of the array:

// Activity screen merges sticky + regular feeds
const allFeeds = [...(sticky || []), ...feeds.data];

UI

  • Pinned posts show a blue badge with pin icon and "Pinned" text
  • Activity screen: users can pin their own posts
  • Space screen: moderators/admins can pin any post (if canPin callback is provided)

Pinned Comments

Comments also support pinning via pinComment(). The pinned comment appears at the top of the comment list with a pin icon. Only one comment per post can be pinned (pinning a new one auto-unpins the previous).

10. Surveys & Polls

Surveys are posts with content_type: 'survey' and config in meta.survey_config.

Creating a Survey

feedsApi.createFeed({
  message: 'What should we study next?',
  content_type: 'survey',
  space: 'book-club',
  survey: {
    type: 'single_choice',       // or 'multi_choice'
    options: [
      { label: 'Romans', slug: 'opt_1' },
      { label: 'Psalms', slug: 'opt_2' },
      { label: 'Genesis', slug: 'opt_3' },
    ],
    end_date: '2026-04-01',      // or '' for no expiration
  },
});

Voting

// Cast a vote (send option slugs, not indices)
feedsApi.castSurveyVote(feedId, ['opt_2']);

// Multi-choice: send multiple slugs
feedsApi.castSurveyVote(feedId, ['opt_1', 'opt_3']);

SurveyCard Behavior

  • Options always show results inline (percentage bars + vote counts)
  • Single choice: tap same option = unvote, tap different = switch
  • Multi-choice: tap toggles each option independently
  • Optimistic update with server reconciliation (returns updated survey_config)
  • Expired surveys (end_date in past) are read-only

11. Bookmarks

Bookmarks use the same /feeds/{id}/react endpoint as reactions, with react_type: 'bookmark'.

// Bookmark a post
feedsApi.toggleBookmark(feedId, false);
// → POST /feeds/{id}/react with { react_type: 'bookmark' }

// Unbookmark
feedsApi.toggleBookmark(feedId, true);
// → POST /feeds/{id}/react with { react_type: 'bookmark', remove: true }

The bookmark icon appears in FeedCard. Toggling emits CACHE_EVENTS.BOOKMARKS so the bookmarks screen refreshes.

Bookmarks response shape varies. The getBookmarks() response structure differs between Fluent Community versions — the app normalizes it before use. Always access feeds via the normalized array.

12. Media in Posts

Media data lives in feed.meta. The type of media determines the rendering strategy.

Images

SourceFieldRendering
Single imagemeta.media_preview.imageFull-width image, tappable → MediaViewer
Multiple imagesmeta.media_items[]Grid layout: 1, 2, or 2×2. 4+ images show "+N" overlay on last cell.
Featured imagefeatured_imageFallback if no media_preview/items

YouTube Videos

Detected via meta.media_preview.provider === 'youtube' and content_type === 'video'. Rendered as thumbnail + play button initially. Tapping mounts an in-place YouTubeEmbed component.

Direct Video

Detected via meta.media_preview.content_type === 'video' (non-YouTube). Rendered with a black background and play button overlay. Tapping mounts the native VideoPlayer (ExoPlayer on Android, AVPlayer on iOS).

GIFs

Attached via meta.media_preview with provider: 'giphy'. Rendered as an auto-playing image.

14. Hoisted Modals Pattern

contexts/FeedModalsContext.tsx solves the "N modals per card" problem.

Problem: Each FeedCard in a FlashList would render its own menu, reaction picker, breakdown modal, media viewer, and report modal — that's 5 modal components × N visible cards.

Solution: FeedModalsProvider renders exactly one instance of each modal at the screen level. Cards call useFeedModals() to open them with parameters.

// Inside FeedCard:
const { openMenu, openReactionPicker, openReactionBreakdown, openMediaViewer, openReport } = useFeedModals();

// Open menu from 3-dot button
openMenu({
  feed,
  anchor: { top: buttonY, right: buttonX },
  isOwner: feed.user_id === currentUser.id,
  canEditOrDelete: true,
  canPin: false,
  isSticky: !!feed.is_sticky,
  onEdit: () => handleEdit(feed),
  onDelete: () => handleDelete(feed),
});

The five hoisted modals:

  1. DropdownMenu — post options (copy link, pin, edit, delete, report)
  2. ReactionPicker — emoji reaction selector
  3. ReactionBreakdownModal — who reacted with what
  4. MediaViewer — full-screen image gallery with swipe
  5. ReportModal — report post/comment with reason picker

15. Ticker Polling

hooks/useTickerPolling.ts — keeps the user marked as "active" so the Fluent Community server doesn't skip Pusher broadcasts.

  • Polls GET /feeds/ticker every ~4 minutes (randomized 3.5–4.5 min to avoid thundering herd)
  • Server skips Pusher message broadcasts to users inactive >5 minutes
  • Fires immediately on mount and on app foreground
  • Pauses when app is backgrounded
  • Only runs when user is authenticated
// In the main layout (authenticated area):
useTickerPolling(isAuthenticated);

17. Rules

  1. Always send space (slug), never space_id. The Fluent Community API expects the space slug string, not the numeric ID. Using space_id silently fails.
  2. Comment field is comment, not message. Both create and update comment endpoints use the comment field name. This matches the web app payload.
  3. Use useFeedReactions and useFeedActions for all feed screens. Don't implement reaction or action handling inline — these hooks handle optimistic updates, error recovery, and cache invalidation.
  4. Wrap feed screens in FeedModalsProvider. Never render modals inside FeedCard — that creates N modal instances in a FlashList. Use the hoisted modals pattern.
  5. Send X-TBC-Reaction-Type header with all reaction API calls. Without it, the tbc-multi-reactions plugin can't track the actual reaction type.
  6. Feed reactions send react_type: 'like'. Fluent Community core only understands "like". The header carries the real type. Comment reactions use { state: 0|1 } instead.
  7. Reconcile after mutation. After any successful reaction API call, call reconcileReactionBreakdown() to fetch server-accurate counts. This is non-blocking and silently updates state.
  8. Handle sticky post shape variance. The sticky field in FeedsResponse can be null, empty array, or populated array. Always handle all cases when merging.
  9. Survey votes use option slugs, not indices. Send vote_indexes: ['opt_1', 'opt_2'] — these are slug strings from SurveyOption.slug, not numeric indices.
  10. Emit cache events after mutations. After creating/editing/deleting a post, emit CACHE_EVENTS.FEEDS. After toggling a bookmark, emit CACHE_EVENTS.BOOKMARKS.
  11. Check FEATURES.MULTI_REACTIONS before showing reaction picker and breakdown. If the multi-reactions plugin isn't installed, only simple like/unlike is available.
  12. Use message_rendered for display, message for editing. The server returns pre-rendered HTML in message_rendered — render it with HtmlContent. When editing, use htmlToMarkdown(message_rendered) to convert back for the editor.
