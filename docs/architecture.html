2. Boot Sequence

What happens when a user opens the app, from cold start to interactive:

1. Expo loads → app/_layout.tsx mounts 2. Provider stack wraps the app (Theme, Auth, Config, Pusher, Modules...) 3. AuthContext.checkAuthStatus() → Reads SecureStore for stored JWT + user info → If valid: sets isAuthenticated = true → If missing/expired: routes to /login 4. Auth guard in RootLayoutNav evaluates route access 5. useStartupData fires (single batch HTTP request) → 7 API paths in one request (replaces ~12 individual calls) → Distributes results to contexts + AsyncStorage caches 6. ThemeContext receives app-config with Fluent Community colors 7. Pusher connects to private channel for real-time messaging 8. Module onInit() callbacks fire (non-blocking) 9. Home screen renders with cached widget data (instant) → Widgets refresh in background via useCachedData
Fast perceived startup. Cached data renders immediately. The batch request populates fresh data in the background. Users see content within milliseconds of auth check completing.

4. Authentication

JWT Token Flow

Login: POST /tbc-ca/v1/auth/login { username, password } → { access_token (1d), refresh_token (6mo), user } → Stored in expo-secure-store → Push token registered (non-blocking) Silent Refresh (on 401/403): API client detects expired token → POST /tbc-ca/v1/auth/refresh { refresh_token } → New access_token stored → Original request retried with _isRetry flag → If refresh fails: clear auth, route to /login Logout: Unregister push token → server-side session revoke → Clear SecureStore (jwt, refresh, user) → Clear AsyncStorage caches → Module cleanup callbacks → Pusher disconnect

SecureStore Keys

KeyContents
tbc_auth_jwtAccess token (Bearer)
tbc_auth_refreshRefresh token
tbc_user_infoUser profile (JSON string: id, username, email, avatar, displayName)

Auth Guard

The RootLayoutNav component in app/_layout.tsx acts as a route guard:

  • Unauthenticated users can only access: /login, /register, /forgot-password, /webview (with noAuth param)
  • Authenticated users are redirected away from auth screens
  • Profile-incomplete users are redirected to /profile-complete
  • The /register route stays accessible even after auth (for the avatar/bio completion step)

5. API Client

All HTTP requests go through a centralized client at services/api/client.ts.

Request Pipeline

  1. Build URL with query params from the base URL
  2. Attach Bearer token from getAuthToken()
  3. Set Content-Type (skip for FormData uploads)
  4. 20-second timeout via AbortController
  5. Pre-flight connectivity check (NetInfo)
  6. Send request, parse JSON response

Error Handling

  • 401/403 with JWT codes → silent refresh + retry (one attempt, _isRetry flag prevents loops)
  • Multiple error shapes normalized — handles { message: string | string[] }, { code, message, data }, plain strings
  • Validation errors extracted from nested objects into flat field-keyed map
  • Network errors caught and mapped to readable messages

Response Header Interceptor

Every authenticated response is scanned for custom headers that update app-wide state:

X-TBC-Unread-MessagesMessage badge countUnreadCountsContext

Exported Methods

});

6. Data Fetching

The app uses a stale-while-revalidate (SWR) pattern via hooks/useCachedData.ts. Cached data renders instantly; fresh data loads in the background.

How It Works

Mount: 1. Read AsyncStorage instantly (no loading spinner if cache exists) 2. Fetch fresh data in background 3. Update state + persist to AsyncStorage App Focus: 1. Check if data is older than staleTime 2. If stale: fetch fresh in background 3. If fresh: skip network call Pull-to-Refresh: 1. Always fetches fresh (ignores staleTime) 2. Shows refresh indicator Cross-Screen Invalidation: 1. Screen A mutates data 2. Emits cache event via invalidateOn key 3. Screen B (listening on same key) refetches silently

Usage

const { data, isLoading, isRefreshing, error, refresh, mutate } = useCachedData({
  cacheKey: 'tbc_space_members_general',
  fetcher: () => getSpaceMembers('general'),
  refreshOnFocus: true,
  staleTime: 30000,  // skip network if fetched within 30s
  invalidateOn: CACHE_EVENTS.SPACES,  // cross-screen refresh
});

Batch Fresh Skip

When the startup batch pre-populates a widget's cache key, useCachedData skips its initial network fetch. This prevents duplicate requests on app boot.

// In useStartupData: writes to AsyncStorage + marks as batch-fresh
await AsyncStorage.setItem('tbc_widget_enrolled_courses', JSON.stringify(courses));
markBatchFresh(['tbc_widget_enrolled_courses']);

// In the widget: useCachedData checks isBatchFresh() on mount
// If batch-fresh, skip initial fetch — cache is already current

7. Startup Batch

A single HTTP request at hooks/useStartupData.ts replaces ~12 individual calls on authenticated app start.

Batch Paths

#PathDataDistributed To
1/tbc-ca/v1/app-configServer config + visibility flagsThemeContext, AppConfigContext
2/fluent-community/v2/profile/{user}Profile refresh (avatar, name)AuthContext.updateUser()
3/fluent-community/v2/notifications/unreadUnread notification countUnreadCountsContext + OS badge
4/fluent-community/v2/chat/unread_threadsUnread message threadsUnreadCountsContext
5/fluent-community/v2/feeds/welcome-bannerWelcome widget dataAsyncStorage (widget cache)
6/fluent-community/v2/courses?type=enrolledEnrolled coursesAsyncStorage (widget cache)
7/tbc-ca/v1/cart/countCart badge countUnreadCountsContext
Modules don't use the batch. Module widgets self-fetch via useCachedData when they mount. This keeps the batch lean and avoids coupling modules to core startup.

Lifecycle

  • Fires once per authentication (via hasRun ref)
  • Resets on logout so it re-fires on next login
  • If the batch fails, widgets fall back to their own fetching — no user-visible error

8. Real-Time (Pusher)

The app uses Pusher WebSockets for real-time chat. Socket config is server-driven — read from /app-config at startup and cached in contexts/AppConfigContext.tsx. Supports Pusher Cloud, Fluent Socket, and custom Soketi — the buyer configures once in WordPress admin, the app auto-detects the provider.

Connection Flow

Auth confirmed → AppConfigContext provides socketConfig (cached or from /app-config) → PusherContext calls initializePusher(userId, socketConfig) → Detects provider: wsHost present = Soketi, absent = Pusher Cloud → Creates Pusher client with provider-specific options → Subscribe to private-chat_user_{userId} → Listen for 'message' + 'reaction' events Authorizer: → Fetches fresh JWT each time (not closure-captured) → POST /chat/broadcast/auth with socket_id + channel_name → Returns { auth: string } for private channel access

Events

EventPayloadHandler
message{ thread_id, message: ChatMessage }Updates chat UI, increments unread count
reaction{ message_id, thread_id, reactions }Updates emoji reactions on messages

React Hooks

// Subscribe to new messages in a specific thread
useNewMessageListener((data) => {
  if (data.thread_id === currentThreadId) {
    appendMessage(data.message);
  }
});

// Subscribe to reaction changes
useReactionListener((data) => {
  if (data.thread_id === currentThreadId) {
    updateReactions(data.message_id, data.reactions);
  }
});

Lifecycle

  • Connect: On auth confirmed (PusherContext effect)
  • Reconnect: On app foreground (preserves existing handlers)
  • Disconnect: On logout (clears all handlers)
  • Ticker polling: Keeps xprofile.last_activity fresh so server sends real-time events (server skips broadcasts for users inactive >5 min)

10. Module System

Self-contained features that plug into the app without touching core code. Each module registers a manifest declaring what it provides.

Module Manifest

// modules/yourmodule/module.ts
import type { ModuleManifest } from '@/modules/_types';

export const YourModule: ModuleManifest = {
  id: 'yourmodule',
  name: 'Your Module',
  version: '1.0.0',

  // Optional registrations:
  tab: { ... },              // Bottom tab
  widgets: [{ ... }],        // Home screen widgets
  providers: [{ ... }],      // Context providers
  menuItems: [{ ... }],      // User menu items
  headerIcons: [{ ... }],    // Top header icons
  tabBarAddon: MiniPlayer,   // Tab bar addon (audio player, cart bar)
  routePrefixes: ['/yourmodule'], // For push/deep link routing
  companionPlugin: 'tbc-yourplugin', // WordPress plugin slug

  onInit: () => { ... },     // Called after auth confirmed
  onCleanup: () => { ... },  // Called on logout
  notificationHandler: (data, router) => { ... }, // Push handler
};

Enabling a Module

// modules/_registry.ts
import { YourModule } from './yourmodule/module';

export const MODULES: ModuleManifest[] = [
  YourModule,       // Add one line to enable
  // OtherModule,   // Comment out to disable
];

Route Stubs

If a module has a tab, it needs a one-line route stub in app/(tabs)/:

// app/(tabs)/yourmodule.tsx
export { default } from '@/modules/yourmodule/screens/YourModuleScreen';
Modules must be self-contained. Don't import from modules into core code. Core provides hooks and utilities that modules can use, not the other way around.

11. Theme System

Colors are synced from the WordPress site's Fluent Community color scheme. The app always matches the web portal's look.

Color Flow

Fluent Community admin (WordPress) ↓ saves color scheme tbc-community-app plugin (PHP) ↓ exposes via REST API GET /tbc-ca/v1/app-config (public, no auth) ↓ returns light + dark color tokens services/api/theme.ts → mapFluentToAppColors() ↓ maps Fluent keys to app tokens ThemeContext → caches in AsyncStorage ↓ provides via useTheme() hook Components use colors.primary, colors.surface, etc.

Token Categories

Server-synced tokens (from Fluent Community):

  • Surface/Background: surface, background, backgroundSecondary, activeBg, lightBg, deepBg, highlightBg
  • Text: text, textSecondary, textTertiary, textInverse
  • Accent: primary, primaryDark
  • Border: border, borderLight
  • Tab Bar: tabBar.background, tabBar.border, tabBar.active, tabBar.inactive

App-only tokens (local defaults, not from server):

  • Semantic: success/successLight, error/errorLight, warning/warningLight, info/infoLight
  • Special: overlay (modal backdrops)

Usage Rules

// Always use the hook
const { colors } = useTheme();

// GOOD
backgroundColor: colors.primary
color: colors.textInverse        // text on colored buttons
borderColor: colors.border

// BAD
backgroundColor: '#6366F1'      // hardcoded color
color: '#fff'                    // use colors.textInverse
borderColor: '#e5e7eb'           // use colors.border

// For transparent variants
import { withOpacity } from '@/constants/colors';
backgroundColor: withOpacity(colors.primary, 0.12)  // light brand tint

Full token mapping in Component Reference › Theme System.

Cross-Screen Invalidation

When one screen mutates data, other screens listening on the same cache event key automatically refresh:

// Screen A: create a post
await createFeed(data);
cacheEvents.emit(CACHE_EVENTS.FEEDS);  // tell other screens

// Screen B: useCachedData with invalidateOn
useCachedData({
  cacheKey: 'tbc_space_feeds_general',
  fetcher: () => getSpaceFeeds('general'),
  invalidateOn: CACHE_EVENTS.FEEDS,  // auto-refetch when event fires
});
No global store. The SWR pattern + cache events handle 95% of cross-screen data sync. Pusher handles the remaining 5% (real-time chat). This keeps the architecture simple and avoids the overhead of a state management library.

Setup Dashboard & Update System

The setup dashboard (npm run dashboard) is a zero-dependency local web app that handles the entire app lifecycle. It runs on Node built-ins only and serves at localhost:3456. Seven tabs: Config (visual editor for all buyer-editable values), Assets (branding checklist with thumbnails), Plugins & Firebase (plugin status and Firebase config), Modules (import and manage feature modules), Builds (EAS build management and store submission), Updates (license activation, one-click core updates with automatic backup/restore), and Status (live validation and connectivity checks).

Update Flow

  1. Buyer enters license key → dashboard validates against license server and auto-activates
  2. Dashboard checks for available updates (automatic on tab open, or manual "Check Now")
  3. Buyer clicks "Backup & Update" → dashboard backs up current state, downloads update, applies core files only
  4. Protected files (config, assets, modules, bundle IDs) are never overwritten
  5. If the dashboard itself was updated, it’s written as dashboard.next.js and swapped on next launch
  6. One-click restore from any backup if needed

Key Files

FilePurpose
setup/dashboard.jsThe dashboard itself — single-file Node.js web server with embedded HTML/CSS/JS. Config editor, asset management, build management, module import, update system, status checks. Zero npm dependencies.
manifest.jsonCore version, update URL, protected paths list, core plugins list. Read by the dashboard for update checks; updated automatically when an update is applied.
setup/.licenseBuyer’s license key (gitignored). Stored by the dashboard, sent to the license server with the buyer’s site URL for validation and activation.
