1. Overview

The module system lets you add features to the app — bottom tabs, home screen widgets, menu items, header icons, context providers, and persistent UI above the tab bar — without editing any core files. Each module is a self-contained folder under modules/ with a manifest that declares what it registers.

Zero core changes. Adding or removing a module requires exactly two edits: one import line and one array entry in modules/_registry.ts. If the module needs deep linking, also add its path prefix to moduleDeepLinkPaths in app.config.ts. Core code reads from the registry at build time — no runtime discovery, no dynamic imports.

The system is static by design. All modules are resolved at import time, producing a flat list of registrations that core code maps over. This keeps the bundle predictable, startup fast, and debugging simple.

2. Architecture

How it fits together

  _registry.ts       ← Active modules array + derived functions
  blog/

    screens/         ← Screen components
    widgets/         ← Home screen widgets
    services/        ← API calls
    types/           ← TypeScript types
    components/      ← UI components
  bookclub/
    module.ts

    ...

app/
  _layout.tsx        ← Injects module providers, lifecycle, notifications
  (tabs)/
    _layout.tsx      ← Renders module tabs + tab bar addons
    index.tsx        ← Renders module widgets on home screen
    calendar.tsx     ← Route stub (one-line re-export)
  blog/
    index.tsx        ← Route stub for /blog
    [id].tsx         ← Route stub for /blog/:id

components/
  navigation/
    TopHeader.tsx    ← Renders module header icons
    UserMenu.tsx     ← Renders module menu items
  home/
    widgetRegistry.ts ← Merges core + module widgets

Data flow

Module manifests are imported statically into _registry.ts. The registry exposes getter functions (getModuleTabs(), getModuleWidgets(), etc.) that core files call at render time. Modules never import core layouts — the dependency arrow is always core → registry → module manifest.

3. Module Manifest

Every module exports a ModuleManifest object from its module.ts file. This is the only file the registry imports.

// modules/_types.ts

export interface ModuleManifest {
  id: string;              // Unique module ID (never change once shipped)
  name: string;            // Human-readable name
  version: string;         // Semver version

  // Registration slots (all optional)
  tab?: TabRegistration;
  widgets?: WidgetRegistration[];
  providers?: ProviderRegistration[];
  menuItems?: MenuItemRegistration[];
  headerIcons?: HeaderIconRegistration[];
  tabBarAddon?: React.ComponentType;
  routePrefixes?: string[];

  // Metadata (informational — not consumed by runtime code)
  hideMenuKey?: string;       // Server visibility key
  companionPlugin?: string;   // WordPress plugin slug
  apiBase?: string;           // REST API base URL

  // Lifecycle hooks
  onInit?: () => void | Promise<void>;
  onCleanup?: () => void;

  // Push notification handler
  notificationHandler?: (data: Record<string, unknown>, router: Router) => boolean;
}

Manifest fields reference

FieldTypePurpose
id requiredstringUnique ID. Used in dev validation. Never change after shipping.
name requiredstringHuman-readable name (for documentation, not displayed in UI).
version requiredstringSemver version string.
tabTabRegistrationBottom tab — one per module maximum.
widgetsWidgetRegistration[]Home screen widgets. A module can register multiple.
providersProviderRegistration[]Context providers injected into the app tree.
menuItemsMenuItemRegistration[]Items in the avatar dropdown menu.
headerIconsHeaderIconRegistration[]Icons in the top header bar.
tabBarAddonComponentTypePersistent UI above the tab bar (e.g., mini player).

onCleanup() => voidCalled on logout. Clear caches, reset state.

4. Registration Types

TabRegistration

Adds a bottom tab to the tab bar.

interface TabRegistration {
  name: string;           // Must match filename in app/(tabs)/ (e.g. 'calendar')
  title: string;          // Label shown under the tab icon
  icon: string;           // Ionicon name for focused state
  iconOutline: string;    // Ionicon name for unfocused state
  order: number;          // Sort position (core: Home=10, Activity=20, Spaces=30)
  component: ComponentType;  // The tab screen component
  hideMenuKey?: string;   // Server visibility key
  interceptPress?: (router: any) => void;  // Override tab press behavior
  tabBarIcon?: (props: { focused: boolean; color: string }) => ReactNode;  // Custom icon
  tabColor?: ColorTokenKey;  // Override active/inactive color (theme token name)
}
Route stub required. If your module has a tab, you must create a one-line re-export file in app/(tabs)/. Expo Router requires a real file for each route.

WidgetRegistration

Adds a card to the home screen.

interface WidgetRegistration {
  id: string;             // Stable ID (AsyncStorage key — never change once shipped)
  title: string;          // Header text in HomeWidget wrapper
  icon?: string;          // Ionicon name for header
  seeAllRoute?: string;   // "See All" navigation target
  featureFlag?: string;   // If set and false, widget is excluded entirely
  defaultEnabled: boolean;   // Default visibility for new users
  canDisable: boolean;       // Whether user can toggle in edit mode
  externalWrapper: boolean;  // true = HomeWidget wraps it; false = widget handles own layout
  component: ComponentType<{ refreshKey: number }>;  // Widget component
}
refreshKey prop. The home screen passes a refreshKey number that increments on pull-to-refresh. Use it to trigger data refetch. Return null from your widget if there's no data — never render an empty card.

ProviderRegistration

Injects a React context provider into the app tree.

interface ProviderRegistration {
  id: string;       // Unique ID
  order: number;    // Lower = wraps outer in the tree
  component: ComponentType<{ children: ReactNode }>;
}

MenuItemRegistration

Adds an item to the avatar dropdown menu.

interface MenuItemRegistration {
  id: string;
  label: string;          // Display text
  icon: string;           // Ionicon name
  route: string | { pathname: string; params: Record<string, string> };
  order: number;          // Core items use 10-90 — place yours in the gaps
  hideMenuKey?: string;   // Server visibility key
  iconColor?: string;     // Theme token for icon color override
}

HeaderIconRegistration

Adds an icon button to the top header.

interface HeaderIconRegistration {
  id: string;
  icon: string;                // Ionicon name
  route: string | { pathname: string; params: Record<string, string> };
  order: number;               // Core: Messages=10, Notifications=20, Cart=30
  accessibilityLabel?: string;
  hideMenuKey?: string;
}

5. The Registry

The registry (modules/_registry.ts) is where modules are activated. It's the only file you edit when adding or removing a module.

// modules/_registry.ts

import { blogModule } from './blog/module';
import { bookclubModule } from './bookclub/module';

export const MODULES: ModuleManifest[] = [
  blogModule,
  bookclubModule,
  // Add new modules here — comment out to disable
];

Derived functions

The registry exports getter functions that core code calls. These sort by order, inherit visibility keys, and flatten nested arrays:

FunctionReturnsConsumed by
getModuleTabs()TabRegistration[] sorted by orderapp/(tabs)/_layout.tsx
getModuleWidgets()WidgetRegistration[] flat listcomponents/home/widgetRegistry.ts
getModuleProviders()ProviderRegistration[] sorted by orderapp/_layout.tsx
getModuleMenuItems()MenuItemRegistration[] sorted by ordercomponents/navigation/UserMenu.tsx
getModuleHeaderIcons()HeaderIconRegistration[] sorted by ordercomponents/navigation/TopHeader.tsx
getTabBarAddons()ComponentType[] in orderapp/(tabs)/_layout.tsx
getModuleRoutePrefixes()string[] flat listapp/_layout.tsx
getWidgetComponentMap()Record<string, ComponentType>app/(tabs)/index.tsx
initModules()Promise<void>app/_layout.tsx (after auth)
cleanupModules()voidCache registry (on logout)
handleModuleNotification()booleanapp/_layout.tsx (notification tap)

Dev-mode validation

In __DEV__ mode, the registry checks for:

  • Duplicate module IDs
  • Duplicate widget IDs
  • Multiple tabBarAddon registrations (warns they'll stack)

6. Integration Points

Here's exactly where each registration type gets rendered in the app:

Bottom tabs

app/(tabs)/_layout.tsx calls getModuleTabs() and renders a Tabs.Screen for each. Module tabs appear after core tabs (Home, Activity, Spaces), sorted by order.

// How core code renders module tabs (simplified)
{getModuleTabs().map((tab) => (
  <Tabs.Screen
    key={tab.name}
    name={tab.name}
    options={{
      title: tab.title,
      href: isHidden(tab.hideMenuKey) ? null : undefined,
      tabBarIcon: tab.tabBarIcon || (({ focused, color }) =>
        <Ionicons name={focused ? tab.icon : tab.iconOutline} color={color} />
      ),
    }}
    listeners={tab.interceptPress ? {
      tabPress: (e) => { e.preventDefault(); tab.interceptPress!(router); }
    } : undefined}
  />
))}

Home screen widgets

app/(tabs)/index.tsx merges core and module widgets into a single list. Users can reorder and toggle widgets in edit mode (long-press). Preferences persist in AsyncStorage.

Avatar dropdown menu

components/navigation/UserMenu.tsx renders module menu items after core items (Profile, Spaces, Bookmarks, etc.), sorted by order.

Top header icons

components/navigation/TopHeader.tsx renders module icons after core icons (Messages, Notifications, Cart) and before the avatar button.

Context providers

app/_layout.tsx wraps the app in module providers using reduceRight:

// ModuleProviders component in app/_layout.tsx
function ModuleProviders({ children }) {
  const providers = getModuleProviders();
  return providers.reduceRight(
    (acc, { component: Provider }) => <Provider>{acc}</Provider>,
    children
  );
}

// Position in provider tree:
<ThemeProvider>
  <AuthProvider>
    <AppConfigProvider>
      <PusherProvider>
        <ModuleProviders>       ← Module providers injected here
          <KeyboardProvider>
            <RootLayoutNav />
          </KeyboardProvider>
        </ModuleProviders>
      </PusherProvider>
    </AppConfigProvider>
  </AuthProvider>
</ThemeProvider>

Tab bar addons

app/(tabs)/_layout.tsx renders addon components in a measured container above the tab buttons. The height is tracked via BottomOffsetContext so content below knows where the tab bar starts.

Push notification routing

app/_layout.tsx calls handleModuleNotification(data, router) when a notification is tapped. Modules return true to claim the notification, false to let core handle it.

7. Visibility & Feature Flags

Three layers control what the user sees:

LayerMechanismControlled by
Build-time Module is in/out of MODULES array in _registry.ts Developer
Server-side hide_menu[] from /app-config API — matched against hideMenuKey WordPress admin (per-role visibility)
Feature flags featureFlag on widgets — checked against FEATURES.* in config.ts Developer (config.ts)

hideMenuKey inheritance

If a module sets hideMenuKey at the manifest level, it's inherited by the tab registration (unless the tab sets its own). Menu items and header icons set their own hideMenuKey individually.

// Module-level hideMenuKey inherited by tab
export const bookclubModule: ModuleManifest = {
  id: 'bookclub',
  hideMenuKey: 'bookclub',  // ← Tab inherits this
  tab: { ... },             // Tab doesn't need its own hideMenuKey
  menuItems: [{
    hideMenuKey: 'bookclub', // ← Menu items set their own
    ...
  }],
};

Widget user preferences

Users can reorder and toggle widgets on the home screen (long-press to enter edit mode). These preferences are stored in AsyncStorage and persist across sessions. The defaultEnabled and canDisable fields on WidgetRegistration control the initial state and whether toggling is allowed.

8. Lifecycle Hooks

onInit

Called once per app session after authentication is confirmed. Fire-and-forget — errors are logged but don't crash the app.

// In app/_layout.tsx
useEffect(() => {
  if (isAuthenticated) {
    initModules();  // calls onInit() on each module
  }
}, [isAuthenticated]);

onCleanup

Called automatically on logout via the cache registry system. Use it to clear module-specific caches, reset state, or tear down connections.

// _registry.ts self-registers with the cache system
registerCache({ clearMemory: cleanupModules });

9. Creating a Module (Step-by-Step)

Let's build a module called "events" that registers a home widget and a menu item.

Step 1: Create the folder structure

modules/events/
  module.ts           ← Manifest
  widgets/
    EventsWidget.tsx  ← Home widget component
  screens/
    EventsScreen.tsx  ← Main screen
  services/
    eventsApi.ts      ← API calls
  types/
    events.ts         ← TypeScript types

Step 2: Define your types

// modules/events/types/events.ts

export interface Event {
  id: number;
  title: string;
  date: string;
  location: string;
}

export interface EventsResponse {
  success: boolean;
  events: Event[];
}

Step 3: Create your API service

// modules/events/services/eventsApi.ts

import { SITE_URL } from '@/constants/config';
import { request } from '@/services/api/client';
import type { EventsResponse } from '../types/events';

const API_BASE = `${SITE_URL}/wp-json/my-events/v1`;

export async function getEvents() {
  return request<EventsResponse>('/events', { baseUrl: API_BASE });
}

export async function getUpcomingEvent() {
  return request<EventsResponse>('/events/upcoming', { baseUrl: API_BASE });
}
Use the centralized request() client from @/services/api/client for any endpoints that require authentication. It handles JWT injection, 401 retry, and silent token refresh automatically.

Step 4: Build your widget

// modules/events/widgets/EventsWidget.tsx

import React from 'react';
import { View, Text, StyleSheet } from 'react-native';
import { useTheme } from '@/contexts/ThemeContext';
import { useCachedData, WIDGET_STALE_TIME } from '@/hooks/useCachedData';
import { getUpcomingEvent } from '../services/eventsApi';

interface Props {
  refreshKey: number;
}

export function EventsWidget({ refreshKey }: Props) {
  const { colors } = useTheme();
  const { data } = useCachedData({
    cacheKey: 'tbc_widget_events',
    fetcher: getUpcomingEvent,
    refreshKey,
    refreshOnFocus: false,
    staleTime: WIDGET_STALE_TIME,
  });

  // Return null if no data — never render an empty card
  if (!data?.events?.length) return null;

  const event = data.events[0];
  return (
    <View style={[styles.card, { backgroundColor: colors.surface }]}>
      <Text style={{ color: colors.text }}>{event.title}</Text>
      <Text style={{ color: colors.textSecondary }}>{event.date}</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  card: { padding: 16, borderRadius: 12 },
});

Step 5: Build your screen

// modules/events/screens/EventsScreen.tsx

import { View } from 'react-native';

  id: 'events',

      icon: 'calendar-outline',

      canDisable: true,
      externalWrapper: true,
      component: EventsWidget,
    },
  ],

  menuItems: [
    {
      id: 'events',
      label: 'Events',
      icon: 'calendar-outline',
      route: '/events',
      order: 50,
      hideMenuKey: 'events',
    },
  ],

  routePrefixes: ['/events'],
  companionPlugin: 'my-events-plugin',
  apiBase: '/wp-json/my-events/v1',
};

Step 7: Create route stubs

// app/events/index.tsx
export { default } from '@/modules/events/screens/EventsScreen';
One-line re-exports only. Route stubs must be a single export { default } line. The actual screen component lives in your module folder. This keeps Expo Router happy while your module stays self-contained.

Step 8: Register in the registry

// modules/_registry.ts

import { eventsModule } from './events/module';   // Add import

export const MODULES: ModuleManifest[] = [
  // ...existing modules
  eventsModule,   // Add to array
];

Step 9: Add deep link path (if needed)

If your module handles deep links from your website (e.g. https://yoursite.com/events/my-event), add the path prefix to the moduleDeepLinkPaths array in app.config.ts:

// app.config.ts — YOUR CONFIG section
const moduleDeepLinkPaths: string[] = [
  '/blog/',
  '/bookclub/',
  '/events/',    // Add your module's path
];

Also set routePrefixes in your module manifest so push notification taps route correctly.

That's it. Your widget appears on the home screen, your menu item appears in the avatar dropdown, and /events routes to your screen.

10. Directory Structure

Recommended folder layout for a module. Only module.ts is required — everything else depends on what your module does.

modules/yourmodule/
  module.ts              ← Required: ModuleManifest export
  screens/               ← Screen components (one per route)
    YourScreen.tsx
    YourDetailScreen.tsx
  widgets/               ← Home screen widget components
    YourWidget.tsx
  components/            ← Internal UI components
    YourCard.tsx
    YourSheet.tsx
  services/              ← API service functions
    yourApi.ts
  contexts/              ← Context providers (if registering providers)
    YourContext.tsx
  hooks/                 ← Custom hooks
    useYourHook.ts
  types/                 ← TypeScript types
    your.ts
No barrel files. Never create index.ts re-export files. Always import directly from the source file. Metro doesn't tree-shake — barrel imports force loading every sibling even when you only need one.

11. API Service Patterns

Authenticated endpoints — use request()

For any endpoint that requires a logged-in user, use the centralized request() from @/services/api/client. It handles JWT injection, 401/403 retry, and silent token refresh.

// modules/bookclub/services/bookclubApi.ts

import { SITE_URL } from '@/constants/config';
import { request } from '@/services/api/client';

const API_BASE = `${SITE_URL}/wp-json/tbc-bc/v1`;

export async function getBooks() {
  return request<BooksListResponse>('/books', { baseUrl: API_BASE });
}

export async function createBookmark(bookId: number, timestamp: number) {
  return request<CreateBookmarkResponse>(`/books/${bookId}/bookmarks`, {
    baseUrl: API_BASE,
    method: 'POST',
    body: { timestamp },
  });
}

Public endpoints — fetch() is fine

For truly public endpoints that never need authentication (e.g., YouTube video proxying), you can use fetch() directly. This avoids unnecessary JWT overhead and 401 retry logic.

// modules/youtube/services/youtubeApi.ts

import { TBC_YT_URL } from '@/constants/config';

export async function getLatestVideos(limit: number = 1) {
  const response = await fetch(`${TBC_YT_URL}/latest?limit=${limit}`, {
    headers: { 'Accept': 'application/json' },
  });
  if (!response.ok) return null;
  const data = await response.json();
  return data.success ? data : null;
}

WP REST API (standard WordPress)

For standard WordPress endpoints (posts, comments, users), use request() with WP_REST_URL as the base. The blog module demonstrates this pattern with a thin wpRequest() wrapper:

// modules/blog/services/blogApi.ts

import { WP_REST_URL } from '@/constants/config';
import { request } from '@/services/api/client';

async function wpRequest<T>(endpoint: string, options = {}) {
  return request<T>(endpoint, {
    ...options,
    baseUrl: WP_REST_URL,
    includeHeaders: true,  // Needed for X-WP-Total pagination headers
  });
}

Logging in services

Use createLogger() from @/utils/logger. Follow the convention: log writes, skip reads. See Logging docs for full details.

import { createLogger } from '@/utils/logger';
const log = createLogger('EventsAPI');

export async function createEvent(data) {
  log('createEvent:', data.title);
  return request('/events', { baseUrl: API_BASE, method: 'POST', body: data });
}

12. Examples

Practical examples of things a client might actually want. Each shows just the manifest and any required files — copy, adapt, register.

Example A: Menu item that opens a WordPress page

The simplest possible module. No tab, no widget — just a link in the avatar dropdown that opens a WordPress page in the in-app browser.

// modules/resources/module.ts

import type { ModuleManifest } from '@/modules/_types';
import { SITE_URL } from '@/constants/config';

export const resourcesModule: ModuleManifest = {
  id: 'resources',
  name: 'Resources',
  version: '1.0.0',

  menuItems: [{
    id: 'resources',
    label: 'Resources',
    icon: 'document-text-outline',

      pathname: '/webview',

        url: `${SITE_URL}/resources/`,

    },

};

Files needed: Just modules/resources/module.ts and the registry entry. No route stubs, no screens, no components.

Example B: Home screen widget

A widget that shows a daily verse or announcement on the home screen. Fetches from a simple REST endpoint on your WordPress site.

// modules/daily-verse/module.ts

import type { ModuleManifest } from '@/modules/_types';
import { DailyVerseWidget } from './widgets/DailyVerseWidget';

export const dailyVerseModule: ModuleManifest = {
  id: 'daily-verse',
  name: 'Daily Verse',
  version: '1.0.0',

  widgets: [{
    id: 'daily-verse',
    title: 'Daily Verse',
    icon: 'book-outline',
    defaultEnabled: true,
    canDisable: true,
    externalWrapper: true,
    component: DailyVerseWidget,
  }],
};
// modules/daily-verse/widgets/DailyVerseWidget.tsx

import React from 'react';
import { View, Text, StyleSheet } from 'react-native';
import { useTheme } from '@/contexts/ThemeContext';
import { useCachedData, WIDGET_STALE_TIME } from '@/hooks/useCachedData';

async function fetchVerse() {
  return request<{ verse: string; reference: string }>('/daily-verse', {
    baseUrl: API_BASE,
  });
}

export function DailyVerseWidget({ refreshKey }: { refreshKey: number }) {
  const { colors } = useTheme();
  const { data } = useCachedData({
    cacheKey: 'tbc_widget_daily_verse',
    fetcher: fetchVerse,
    staleTime: WIDGET_STALE_TIME,
  });

  if (!data) return null;

  return (
    <View style={[styles.card, { backgroundColor: colors.surface }]}>
      <Text style={[styles.verse, { color: colors.text }]}>
        “{data.verse}”
      </Text>
      <Text style={[styles.ref, { color: colors.textSecondary }]}>
        {data.reference}
      </Text>
    </View>
  );
}

const styles = StyleSheet.create({
  card: { padding: 20, borderRadius: 12, alignItems: 'center' },
  verse: { fontSize: 16, fontStyle: 'italic', textAlign: 'center', lineHeight: 24 },
  ref: { fontSize: 13, marginTop: 8 },
});

Files needed: module.ts + widgets/DailyVerseWidget.tsx + registry entry.

Example D: Tab that opens a WebView on press (no screen)

A tab icon that doesn't navigate to a screen — tapping it opens a URL in the in-app browser. Useful for external links like giving pages, signup forms, or third-party tools.

// modules/give/module.ts

import type { ModuleManifest } from '@/modules/_types';
import { SITE_URL } from '@/constants/config';

export const giveModule: ModuleManifest = {
  id: 'give',
  name: 'Give',
  version: '1.0.0',

  tab: {
    name: 'give',
    title: 'Give',
    icon: 'heart',
    iconOutline: 'heart-outline',
    order: 100,
    component: () => null,   // Never renders — interceptPress handles it
    hideMenuKey: 'give',

13. Rules

Modules never import core layouts or navigation files. The dependency arrow is always core → registry → module. If you need to render something in a specific location, register it via the manifest.
  1. One manifest, one export. Your module.ts exports a single ModuleManifest object. The registry imports only this file.
  2. No barrel files. Import directly from source files. Never create index.ts re-exports.
  3. Import core via @/ aliases. Modules can import from @/components, @/contexts, @/services, @/hooks, @/constants, @/utils. Never import from other modules.
  4. No inter-module dependencies. Module A must never import from Module B. If two of your modules need shared utilities, create a modules/_shared/ folder for common code. The underscore prefix signals “infrastructure, not a module” (like _types.ts and _registry.ts). Core utilities (@/utils, @/hooks, etc.) are always available to all modules.
  5. IDs are permanent. Widget id and module id are used as AsyncStorage keys. Never change them after shipping — you'll lose user preferences.
  6. Route stubs are one-liners. Files in app/ for module routes must be a single re-export: export { default } from '@/modules/...'.
  7. Always use useTheme(). Never hardcode colors. Import useTheme from @/contexts/ThemeContext and use semantic tokens. See Theme System docs.
  8. Use createLogger() for logging. Tag pattern: {Module}API for services, {Module}{Name} for components. See Logging docs.
  9. Use useCachedData() for data fetching. The centralized hook handles caching, loading states, and refresh. Prefix cache keys with tbc_{module}_.
  10. Widgets return null when empty. Never render an empty card or placeholder. The home screen skips null widgets cleanly.
  11. Register companion plugin and apiBase. Even though they're informational, they make the manifest self-documenting for other developers.
  12. Don't create new dependencies without checking. The core already has centralized utilities for HTTP, caching, theming, logging, haptics, and date formatting. Check before adding.
