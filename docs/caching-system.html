Data Fetching & Caching

Stale-while-revalidate pattern with batch startup, cross-screen invalidation, and optimistic updates

1. How It Works

The app uses a stale-while-revalidate (SWR) pattern. When a screen mounts, it loads cached data from AsyncStorage immediately (no loading spinner if cache exists), then fetches fresh data from the network in the background. If the fresh data differs, the UI updates seamlessly.

Mount ↓ AsyncStorage.getItem(cacheKey) ↓ Cache hit? → Render immediately (isLoading = false) ↓ Batch-fresh? → Skip network fetch (startup batch already populated it) ↓ Fetch from API in background ↓ Update state + persist to AsyncStorage

Data stays in sync across screens through three mechanisms:

  1. Cache events — after a mutation (create post, bookmark, join space), the mutating screen emits a cache event. Other screens subscribed to that event silently re-fetch.
  2. App focus refresh — when the user returns to the app from the background, screens re-fetch (with a cooldown to prevent rapid-fire requests).
  3. Batch startup — on authenticated startup, a single batch API call pre-populates multiple cache keys. Screens that mount after the batch skip their initial network fetch.

2. useCachedData Hook

The single data-fetching hook used by every screen and widget in the app.

Import

import { useCachedData } from '@/hooks/useCachedData';

API

const {
  data,          // T | null — cached or fresh data
  isLoading,     // true only on first load with no cache
  isRefreshing,  // true during background/pull refresh
  error,         // Error | null
  refresh,       // () => Promise<void> — imperative pull-to-refresh
  mutate,        // (T | (prev => T | null)) => void — optimistic updates
} = useCachedData<T>({
  cacheKey,        // AsyncStorage key (required)
  fetcher,         // async () => T (required)
  refreshKey,      // number — changes trigger re-fetch (default: 0)
  refreshOnFocus,  // boolean — refetch on app resume (default: true)
  enabled,         // boolean — gate all fetching (default: true)
  invalidateOn,    // CacheEvent — auto-refresh on this event
  staleTime,       // ms — skip fetch if data is fresh (default: 0)
});

Options Reference

OptionTypeDefaultPurpose
cacheKey string — AsyncStorage key. Prefix with tbc_. Include dynamic segments for parameterized screens (e.g. `tbc_space_feeds_${slug}`).
fetcher () => Promise<T> — Async function that fetches and returns data. Should throw on failure.
refreshKey number 0 Parent-driven refresh. When the value changes, data re-fetches (respects staleTime). Used by widgets whose parent controls refresh timing.
refreshOnFocus boolean true Re-fetch when the app returns from background. Disable for widgets (parent drives refresh).
enabled boolean true Gate all fetching. Use when a required parameter isn't available yet (e.g. enabled: !!slug).
invalidateOn CacheEvent — Subscribes to a cache event. When the event fires, data silently re-fetches (no spinner).
staleTime number (ms) 0 Skip network fetch if data was fetched within this window. Screens use 0 (always fresh on focus, with a 5s cooldown). Widgets use WIDGET_STALE_TIME (120s).

Screen Example

// app/(tabs)/activity.tsx

const {
  data: feedsData,
  isLoading: loading,
  isRefreshing: refreshing,
  error: fetchError,
  refresh,
  mutate,
} = useCachedData<Feed[]>({
  cacheKey: 'tbc_activity_feeds',
  invalidateOn: CACHE_EVENTS.FEEDS,
  fetcher: async () => {
    const response = await feedsApi.getFeeds({ per_page: 20 });
    if (!response.success) {
      throw new Error(response.error?.message || 'Failed to load feeds');
    }
    return response.data.feeds?.data || [];
  },
});

const feeds = feedsData || [];
const error = fetchError?.message || null;
const setFeeds = useArrayMutate(mutate);

Widget Example

// components/home/CoursesWidget.tsx

import { useCachedData, WIDGET_STALE_TIME } from '@/hooks/useCachedData';

const { data: courses } = useCachedData<Course[]>({
  cacheKey: 'tbc_widget_enrolled_courses',
  staleTime: WIDGET_STALE_TIME,
  fetcher: async () => {
    const response = await coursesApi.getCourses({ type: 'enrolled', per_page: 5 });
    if (!response.success) throw new Error('Failed');
    return response.data.courses?.data || [];
  },
});

3. Screens vs Widgets

Screens and widgets use the same hook but with different defaults tuned to their lifecycle:

BehaviorScreensWidgets
staleTime 0 (always fetch on focus) WIDGET_STALE_TIME (120s)
refreshOnFocus true (default) false (parent drives refresh)
invalidateOn Set to relevant event (e.g. CACHE_EVENTS.FEEDS) Usually not set
Rapid focus protection 5s cooldown (FOCUS_COOLDOWN_MS) staleTime handles this
Batch startup N/A — screens aren't batch-populated Core widgets skip initial fetch if batch-fresh
Import WIDGET_STALE_TIME from @/hooks/useCachedData — never hardcode 120000 or 120_000. This keeps the value consistent across all widgets and easy to tune.

4. Cache Events (Cross-Screen Invalidation)

When a mutation happens on one screen (new post, bookmark toggle, profile edit), other screens need to know. Cache events are a lightweight pub/sub for this.

Source

utils/cacheEvents.ts

Available Events

ConstantValueEmitted When
CACHE_EVENTS.FEEDS'feeds'Post created, edited, deleted, commented on
CACHE_EVENTS.BOOKMARKS'bookmarks'Post bookmarked or unbookmarked
CACHE_EVENTS.SPACES'spaces'Space joined, left
CACHE_EVENTS.PROFILE'profile'Profile edited (avatar, name, bio, etc.)

Emitting an Event

import { cacheEvents, CACHE_EVENTS } from '@/utils/cacheEvents';

// After creating a post:
cacheEvents.emit(CACHE_EVENTS.FEEDS);

// After toggling a bookmark:
cacheEvents.emit(CACHE_EVENTS.BOOKMARKS);

Subscribing (Automatic via useCachedData)

// The Activity feed auto-refreshes when any screen emits FEEDS:
useCachedData<Feed[]>({
  cacheKey: 'tbc_activity_feeds',
  invalidateOn: CACHE_EVENTS.FEEDS,  // ← subscribes automatically
  fetcher: ...,
});

Subscribing (Manual — rare)

If you're not using useCachedData (e.g., manual state with side effects), subscribe directly:

useEffect(() => {
  return cacheEvents.subscribe(CACHE_EVENTS.FEEDS, () => {
    fetchFeeds();
  });
}, [fetchFeeds]);
Always use CACHE_EVENTS.FEEDS — never raw strings like 'feeds'. The CacheEvent type enforces this at compile time.

5. Batch Startup Loading

On authenticated startup, hooks/useStartupData.ts fires a single batch API call that replaces ~7 individual HTTP requests. Results are distributed to contexts (config, auth, badges) and written to AsyncStorage for widget caches.

What the Batch Fetches

#EndpointGoes To
1/tbc-ca/v1/app-configAppConfigContext
2/fluent-community/v2/profile/{user}AuthContext (avatar, name)
3/fluent-community/v2/notifications/unreadNotification badge count
4/fluent-community/v2/chat/unread_threadsMessage badge count
5/fluent-community/v2/feeds/welcome-bannerAsyncStorage → WelcomeBannerWidget
6/fluent-community/v2/courses?type=enrolled&per_page=5AsyncStorage → CoursesWidget
7/tbc-ca/v1/cart/countCart badge count

Batch-Fresh Coordination

utils/batchCache.ts prevents duplicate network requests. After the batch writes widget data to AsyncStorage, it calls markBatchFresh(keys). When those widgets mount, useCachedData checks isBatchFresh(cacheKey) — if true, it skips the initial network fetch since the cache was just populated.

// In useStartupData.ts — after writing to AsyncStorage:
await Promise.all(cacheWrites);
markBatchFresh(freshKeys);

// In useCachedData — on initial mount:
if (hasCachedData && isBatchFresh(cacheKey)) {
  log(cacheKey, 'skipping initial fetch — batch-fresh');
  return; // Use cache, don't hit network
}

Batch-fresh status auto-expires after 30 seconds, so subsequent focus/refresh cycles still work normally.

Modules don't use the batch. Module widgets (calendar, bookclub, YouTube, blog) self-fetch via useCachedData when they mount. Only core widgets participate in the batch to keep it lean and avoid coupling modules to core startup.

6. Optimistic Updates

For mutations where the user expects instant feedback (pin/unpin, bookmark toggle, reactions), the app updates the UI immediately and reverts if the API call fails.

Source

utils/optimisticUpdate.ts

Pattern

import { optimisticUpdate } from '@/utils/optimisticUpdate';

const response = await optimisticUpdate(
  setFeeds,                                           // setState function
  prev => prev.map(f =>                              // optimistic updater
    f.id === feed.id ? { ...f, is_sticky: true } : f
  ),
  () => feedsApi.toggleSticky(feed.id, true),         // API call
);

if (!response.success) {
  Alert.alert('Error', response.error?.message || 'Failed');
}

How It Works

  1. Captures a snapshot of current state inside the updater
  2. Applies the optimistic update immediately (UI updates)
  3. Awaits the API call
  4. On failure ({ success: false } or thrown error): reverts to snapshot
  5. Returns the ApiResponse so the caller can handle success (show alerts, emit cache events)
The setState parameter must be a function updater (not a direct value), because optimisticUpdate captures the snapshot inside the updater callback.

7. useArrayMutate Adapter

Feed screens need React.Dispatch<SetStateAction<Feed[]>> for the shared useFeedActions and useFeedReactions hooks. But useCachedData’s mutate accepts T | null. The adapter bridges the two:

import { useCachedData, useArrayMutate } from '@/hooks/useCachedData';

const { data: feedsData, mutate, ...rest } = useCachedData<Feed[]>({ ... });
const feeds = feedsData || [];
const setFeeds = useArrayMutate(mutate);

// Now setFeeds works like a normal setState for arrays:
const { handleDelete, handleEdit, ... } = useFeedActions({ setFeeds, refresh });
const handleReact = useFeedReactions(feeds, setFeeds);

Internally, useArrayMutate wraps the mutate call and treats null as an empty array, so downstream hooks never see null.

8. Cache Registry (Logout Cleanup)

On logout, all user-specific caches must be cleared. Instead of manually listing every key, services/cacheRegistry.ts provides a blanket clear.

How It Works

  1. AsyncStorage — removes ALL keys starting with tbc_ except an explicit keep-list
  2. In-memory caches — calls all registered clear functions (e.g., batchCache clears its Map)
// AuthContext logout calls:
import { clearAllUserCaches } from '@/services/cacheRegistry';
await clearAllUserCaches();

Keys That Survive Logout

These are site-level (not user-specific) and persist across login/logout:

KeyPurpose
tbc_app_config_cacheSite maintenance mode, update config, branding
tbc_app_visibility_cacheSite-level menu visibility (hide_menu[])
tbc_socket_config_cacheSocket provider config (Pusher/Soketi)

Registering In-Memory Caches

If you have an in-memory cache (Map, object, variable), register its clear function so it gets wiped on logout:

import { registerCache } from '@/services/cacheRegistry';

const myCache = new Map<string, Data>();

// Self-registers at import time — no manual wiring needed
registerCache({ clearMemory: () => myCache.clear() });

9. Cache Key Reference

All cache keys use the tbc_ prefix. This table shows the current keys, who writes them, and who reads them.

Cache KeyWriterReader
tbc_activity_feedsActivity screen fetcherActivity screen
tbc_bookmarksBookmarks screen fetcherBookmarks screen
tbc_spaces_listSpaces tab fetcherSpaces tab
tbc_space_feeds_${slug}Space page fetcherSpace page
tbc_profile_${username}Profile screen fetcherProfile screen
tbc_feed_detail_${id}Feed detail fetcherFeed detail screen
tbc_welcome_bannerStartup batch + widget fetcherWelcomeBannerWidget
tbc_widget_enrolled_coursesStartup batch + widget fetcherCoursesWidget
tbc_notificationsNotifications screen fetcherNotifications screen
tbc_messagesMessages screen fetcherMessages screen
Dynamic cache keys (e.g. tbc_space_feeds_${slug}) create a new AsyncStorage entry per slug. These are cleaned up on logout by the cache registry's blanket tbc_* clear, but they do accumulate during a session. This is fine for normal usage but worth noting if a user visits hundreds of unique spaces.

10. Adding Caching to a New Screen

Step 1: Choose your cache key

Prefix with tbc_. If the screen is parameterized, include the param:

const cacheKey = 'tbc_my_feature';           // static screen
const cacheKey = `tbc_my_feature_${id}`;     // parameterized screen
const cacheKey = 'tbc_widget_my_widget';     // widget

Step 2: Write the fetcher

The fetcher should return the data directly. Throw on failure — useCachedData catches it and sets error.

const fetcher = async () => {
  const response = await myApi.getData();
  if (!response.success) {
    throw new Error(response.error?.message || 'Failed to load data');
  }
  return response.data.items;
};

Step 3: Use the hook

const { data, isLoading, isRefreshing, error, refresh, mutate } = useCachedData({
  cacheKey: 'tbc_my_feature',
  fetcher,
  invalidateOn: CACHE_EVENTS.FEEDS,  // if relevant — omit if not
});

Step 4: If your screen has a list with shared feed actions

const setItems = useArrayMutate(mutate);
const { handleDelete, handleEdit, ... } = useFeedActions({ setFeeds: setItems, refresh });

Step 5: If your screen mutates data other screens care about

// After the mutation succeeds:
cacheEvents.emit(CACHE_EVENTS.FEEDS);

Step 6: For widgets, use WIDGET_STALE_TIME

import { useCachedData, WIDGET_STALE_TIME } from '@/hooks/useCachedData';

const { data } = useCachedData({
  cacheKey: 'tbc_widget_my_widget',
  staleTime: WIDGET_STALE_TIME,
  fetcher,
});

11. Rules

  1. Use useCachedData for all data fetching. Don’t write manual useEffect + useState fetch cycles. The hook handles caching, loading states, error handling, focus refresh, and cache invalidation.
  2. Prefix all cache keys with tbc_. The cache registry clears all tbc_* keys on logout. Non-prefixed keys won't get cleaned up.
  3. Use CACHE_EVENTS.X, not raw strings. The CacheEvent type prevents typos and makes event usage searchable.
  4. Use WIDGET_STALE_TIME, not 120000. Import from @/hooks/useCachedData. Keeps the value consistent and easy to change.
  5. Emit cache events after mutations. If your screen creates/edits/deletes data that appears on other screens, emit the relevant event so they refresh.
  6. Use optimisticUpdate for instant-feedback mutations. Pin, bookmark, react — anything the user expects to happen instantly. It handles snapshot + rollback.
  7. Fetchers should throw on failure. Return data on success, throw on failure. useCachedData catches errors and sets error state. Don’t return null to indicate failure.
  8. Don’t add modules to the startup batch. Module widgets self-fetch via useCachedData. Only core widgets participate in the batch.
  9. Register in-memory caches for logout cleanup. If you create a Map, cache object, or any in-memory store, call registerCache() so it gets cleared on logout.
  10. Use useArrayMutate for feed list screens. Don’t write manual mutate adapters. The hook bridges useCachedData’s mutate to the SetStateAction<T[]> that useFeedActions and useFeedReactions expect.
