1. Overview

The app uses a centralized tagged logger that automatically silences all output in production builds. In __DEV__ mode, every log line is prefixed with a tag so you can quickly filter and trace issues by subsystem.

Zero production cost. In production, createLogger returns no-op functions — no string formatting, no console calls, no performance impact.

How it works

// utils/logger.ts — the entire implementation

type LogFn = (...args: any[]) => void;
type Logger = LogFn & { warn: LogFn; error: LogFn };

const noop: LogFn = () => {};

export function createLogger(tag: string): Logger {
  if (__DEV__) {
    return Object.assign(
      (...args: any[]) => console.log(`[${tag}]`, ...args),
      {
        warn: (...args: any[]) => console.warn(`[${tag}]`, ...args),
        error: (...args: any[]) => console.error(`[${tag}]`, ...args),
      },
    );
  }
  return Object.assign(noop, { warn: noop, error: noop });
}

2. Basic Usage

import { createLogger } from '@/utils/logger';

const log = createLogger('MyFeature');

// General info
log('initialized with', items.length, 'items');

// Warnings (non-fatal issues)
log.warn('retrying after timeout');

// Errors (something broke)
log.error('failed to load data:', error);
Always declare the logger at module level (top of file), not inside functions or components. This ensures the __DEV__ check runs once at import time.

3. Methods

MethodMaps toUse for
log(...args) console.log General info — state changes, request details, lifecycle events
log.warn(...args) console.warn Non-fatal issues — retries, fallbacks, unexpected but handled states
log.error(...args) console.error Failures — API errors, caught exceptions, broken invariants

4. When to Log

DO log

  • Write operations — creating, updating, deleting (posts, messages, profiles, threads)
  • Auth events — login, logout, token refresh, credential storage
  • Errors and failures — API errors, batch failures, unexpected states
  • Connection events — Pusher connect/disconnect, push notification registration
  • Batch operations — what's being batched, which sub-requests failed
  • Cache events — cache hits, misses, clears (when debugging data freshness)

DO NOT log

  • Every read/GET request — too noisy, the API client already logs at request level
  • Render cycles — never log inside render functions or on every re-render
  • Sensitive data — never log tokens, passwords, or full user credentials
  • UI state changes — tab switches, scroll positions, animation frames

Pattern: log on mutations, not on reads

// GOOD — log the write operation
export async function createThread(data: CreateThreadRequest) {
  log('createThread intent:', data.intent_type, data.intent_id);
  return post<CreateThreadResponse>(ENDPOINTS.CHAT_THREADS, data);
}

// GOOD — log errors and failures
if (!response.success) {
  log.error('batch failed:', response.error?.message);
  throw new Error(response.error?.message || 'Batch request failed');
}

// BAD — don't log every read
export async function getThreads() {
  log('fetching threads');  // <-- unnecessary, too noisy
  return get<ThreadsResponse>(ENDPOINTS.CHAT_THREADS);
}

7. Adding Logging to a New File

Two lines at the top of your file — that's it:

import { createLogger } from '@/utils/logger';

const log = createLogger('YourTag');

Then use log(), log.warn(), and log.error() throughout the file.

Step-by-step for an API service

// services/api/widgets.ts

import { ENDPOINTS } from '@/constants/config';
import { get, post } from './client';
import { createLogger } from '@/utils/logger';

const log = createLogger('WidgetsAPI');

export async function getWidgets() {
  // No logging needed — it's a read
  return get('/widgets');
}

export async function createWidget(data: CreateWidgetData) {
  // Log the write operation
  log('createWidget:', data.name);
  return post('/widgets', data);
}

export async function deleteWidget(id: number) {
  // Log destructive operations
  log('deleteWidget:', id);
  return post(`/widgets/${id}/delete`, {});
}

Step-by-step for a module

// modules/mymodule/services/myApi.ts

import { createLogger } from '@/utils/logger';

const log = createLogger('MyModuleAPI');

// Follow the same pattern — log writes, skip reads

9. Rules

Never use raw console.log / console.warn / console.error.
Always use createLogger. The only exception is the module registry's __DEV__-wrapped validation warnings and the ErrorBoundary component.
  1. Import from @/utils/logger — the only logging utility in the project.
  2. Declare at module level — const log = createLogger('Tag') at the top of the file, not inside functions.
  3. Log writes, not reads — mutations, creates, deletes, state transitions. Skip GET requests.
  4. Never log sensitive data — no tokens, passwords, full credentials, or personal data.
  5. Keep tags short and unique — check the tag list before picking a new one.
  6. Use the right level — log() for info, log.warn() for handled issues, log.error() for failures.
  7. No per-tag toggle config — all loggers are on in dev, all off in production. Use terminal search to filter.
  8. Don't create a new logging utility — createLogger is the only one. Don't add dependencies like winston, pino, etc.
