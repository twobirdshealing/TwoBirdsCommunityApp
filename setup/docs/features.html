Setup Dashboard — Deep Dive

The setup dashboard (npm run dashboard) is a zero-dependency local web app that handles the entire app lifecycle — from initial setup to ongoing maintenance. It runs on Node built-ins only (no npm install required) and launches at localhost:3456.

This is a major differentiator. No competitor offers a local dashboard for app configuration, builds, and updates. Buyers get a polished GUI instead of editing JSON files by hand or learning git workflows. Non-technical team members can manage updates and configuration without touching the terminal.

Update System

The Updates tab provides a complete update lifecycle that requires zero git knowledge:

  1. License activation — buyer enters their license key (received on purchase). The dashboard validates it against the license server and activates it, linking it to their site URL. License key stored locally in setup/.license.
  2. Update checking — automatically checks for new versions when the tab opens. Shows current version vs. latest, with full changelog. Manual "Check Now" button.
  3. One-click update — "Backup & Update" downloads the update package, creates a full backup of the current state, and applies only core files. Buyer customizations (config, assets, bundle IDs, modules, Firebase configs) are never overwritten. A confirmation dialog lists exactly which files will be updated and which are protected.
  4. Dependency merging — if the update includes new npm dependencies, they're merged into the buyer's package.json (preserving their name, version, and scripts). Dashboard prompts to run npm install after.
  5. Dashboard self-update — if the update includes a new dashboard version, it's written as dashboard.next.js. On next launch, the dashboard swaps itself and restarts automatically.
  6. Manual upload — buyers can also apply updates from a .tar.gz or .zip file (for air-gapped environments or manual delivery).
  7. Backup & restore — automatic backup before every update in setup/.backups/. Dashboard lists all backups with dates and sizes. One-click restore to roll back. Auto-prunes to keep only the last 3 backups.

Technical Details

  • Zero dependencies — built entirely on Node.js built-ins (http, fs, path, https, zlib). No npm install, no framework, no bundler. Single file: setup/dashboard.js.
  • Custom tar parser — reads .tar.gz packages using built-in zlib and a hand-written tar header parser. No third-party archive libraries.
  • Protected paths — defined in manifest.json. The update system checks this list before writing any file. Includes: config.ts, app.json, eas.json, app.config.ts, package.json, assets/images/, Firebase configs, modules/, and the license/backup files.
  • Manifest-driven — manifest.json contains the product version, update server URL, protected paths list, and core plugin list. Updated automatically when an update is applied.

22. Module Ecosystem

The app's module system is an architecture-level exclusive — Fluent Community has no equivalent. Modules are self-contained feature packages that register any combination of: bottom tabs, home widgets, menu items, header icons, context providers, tab bar addons, and UI slots.

Each module gets its own dedicated documentation page. This section provides the overview.
Module Type Tab Widget Menu Description
Multi-Reactions Core ✕ ✕ ✕ Multi-emoji reaction system via slot injection (picker, breakdown modal, swap). Replaces default like button. Companion plugin: tbc-multi-reactions
Add-on modules ADD-ON Varies Varies Varies Additional modules available as add-ons or custom builds. Each can register tabs, widgets, menu items, and more. See module docs for details.

Module Architecture Highlights

  • Zero-touch core: Modules never modify core files. Register once in _registry.ts and the system wires everything up.
  • Server-driven visibility: The server's hide_menu array can hide module tabs, menu items, and header icons without an app update.
  • Feature-flag-gated widgets: Widgets can declare a featureFlag key (e.g. courses, messaging) and are excluded entirely when that flag is disabled in wp-admin.
  • Forward-compatible widgets: When a new module is enabled, its widget auto-appears on the home screen in the next launch.
  • Push notification routing: Modules can register push handlers to intercept and route their own notification types.
  • Context providers: Modules can inject React context providers (e.g., an audio player state) into the app's provider tree.
  • Tab bar addons: Modules can inject persistent UI above the tab bar (e.g., a mini player).
  • Slot injection: Modules can replace core UI components via named slots (e.g., replacing the default like button with a custom reaction picker).
