1. Overview

The app has no centralized form library — all form state lives as local component state (useState) in the screen that owns the form. A shared DynamicFormField component renders field types from server-driven configs, while validation runs inline at step transitions or before submission.

Two primary form screens consume this infrastructure:

ScreenFilePurpose
Registration Wizardapp/register.tsxMulti-step signup with dynamic fields, email verify, module-injectable steps
Profile Editapp/profile/edit.tsxEdit name, bio, social links, custom fields

Both screens follow the same pattern:

const [formData, setFormData] = useState<Record<string, any>>({});
const [fieldErrors, setFieldErrors] = useState<Record<string, string>>({});

2. DynamicFormField Component

components/common/DynamicFormField.tsx is the universal field renderer. It takes a FormFieldConfig and renders the appropriate input type with themed styling, error display, and accessibility.

4. Server-Driven Field Config

Registration fields come from the server, not hardcoded in the app. On mount, register.tsx calls getRegistrationFields() which hits the tbc-ca/v1/auth/register/fields endpoint.

FieldsResponse

export interface FieldsResponse {
  registration_enabled: boolean;
  email_verification_required: boolean;
  fields: Record<string, RegistrationField>;
  message?: string;
  /** Module-specific fields from server plugins */
  [key: string]: any;
}

RegistrationField

export interface RegistrationField {
  label: string;
  placeholder?: string;
  instructions?: string;
  type: 'text' | 'email' | 'password' | 'phone' | 'date'
      | 'number' | 'select' | 'radio' | 'gender'
      | 'textarea' | 'url' | 'inline_checkbox'
      | 'checkbox' | 'multiselect';
  input_type?: string;
  required: boolean;
  options?: string[];
  inline_label?: string;
  step: number;  // 1 = basic, 2 = custom fields, etc.
}

Fields are grouped by their step property. Step 1 fields render on the first screen, step 2 on the second, etc. The wizard dynamically adjusts its step count based on what the server returns and which verification features are enabled.

Profile Custom Fields (FC Native)

Profile edit gets its custom field configs from the profile object via profile.custom_field_groups, which uses FC’s native NativeCustomField format:

export interface NativeCustomField {
  slug: string;
  label: string;
  type: 'text' | 'textarea' | 'number' | 'date'
      | 'select' | 'radio' | 'url' | 'multiselect';
  placeholder: string;
  options: string[];
  is_required: boolean;
  is_enabled: boolean;
  privacy: 'public' | 'logged_in' | 'private' | 'internal';
  group: string;
  value?: string | string[];
}

export interface CustomFieldGroup {
  slug: string;
  label: string;
  edit_description?: string;
  is_system?: boolean;
  fields: NativeCustomField[];
}

In profile edit, the custom_field_groups array is flattened into a slug-to-field map. Only enabled fields (is_enabled: true) are rendered. The radio type is converted to select for mobile rendering (radio renders as a SelectModal dropdown instead of inline buttons).

5. Validation Logic

Validation is not real-time — it runs at specific trigger points:

  • Step transition — validateStep(stepNumber) in register.tsx, called before advancing to next step
  • Form submission — inline validation in handleSave() / handleSubmitRegistration() before the API call
  • Server-side — API returns result.errors: Record<string, string> which are merged into fieldErrors

Registration Validation Rules

FieldRuleError Message
All requiredNon-empty after trim{label} is required.
email/^[^\s@]+@[^\s@]+\.[^\s@]+$/Please enter a valid email address.
usernameMin 4 charsUsername must be at least 4 characters.
username/^[a-z0-9_]+$/Username can only contain lowercase letters, numbers, and underscores.
conf_passwordMust match passwordPasswords do not match.

Profile Edit Validation Rules

FieldRuleError Message
first_nameNon-empty after trimFirst name is required
Custom fieldsfield.is_required && field.is_enabled && !value{label} is required
Clear on change: When a user modifies a field, its error is immediately cleared from fieldErrors. This prevents stale error messages from persisting after the user corrects their input.

6. Error Display Pattern

Error display is handled at two levels:

Field-Level Errors

DynamicFormField shows errors when the error prop is set:

  • Red border on the input container (borderColor: themeColors.error)
  • Red error text below the field (styles.fieldError)
{error && (
  <Text style={[styles.fieldError, { color: themeColors.error }]}>
    {error}
  </Text>
)}

Server-Side Errors

When the API returns validation errors, they arrive as Record<string, string> keyed by field name. These are spread into fieldErrors state so each field shows its server-returned message. In registration, server errors also trigger navigation back to the step containing the errored field.

// After API call returns errors
if (result.errors) {
  setFieldErrors(result.errors);
  // Navigate back to the step with the error
  const errorKeys = Object.keys(result.errors);
  const step1Fields = getFieldsForStep(1).map(([k]) => k);
  const hasStep1Error = errorKeys.some((k) => step1Fields.includes(k));
  if (hasStep1Error) {
    setStep(1);
  } else {
    setStep(2);
  }
}

7. Registration Form (Multi-Step)

app/register.tsx implements a wizard with conditional steps. The step count adapts based on server config — steps are skipped if their feature is disabled. A dot-style step indicator shows progress.

Step 1: Basic Account Info

Always shown. Fields from step 1 of the server config (typically full_name, email, username, password, conf_password). If there are no custom fields (step 2), the step 2 fields (e.g., terms checkbox) are merged into step 1.

Step 2: Custom Profile Fields

Dynamic fields from the server with step: 2. Only shown if there are custom fields beyond just the terms checkbox. The terms agreement checkbox (inline_checkbox) typically appears here.

Step 3: Email Verification conditional

Shown only if email_verification_required: true in the fields response, or when the server returns email_verification_required with a verification_token after form submission. Uses EmailVerifyStep component — a 6-digit code input that the user receives by email.

Steps 4+: Module-Injected Steps conditional

Add-on modules can inject additional steps into the registration wizard via the registrationSteps manifest field. Each step has a shouldActivate() gate that checks the submit response and registration config to determine if the step should be shown. Module steps receive formData, submitResponse, submitting, onResubmit, onComplete, and onBack props. See Module System for details.

Username Sanitization

Before submitting, the registration form sanitizes the username: payload.username.toLowerCase().replace(/[^a-z0-9_]/g, ''). This ensures only lowercase letters, numbers, and underscores are sent to the server.

Verification extras accumulation: When a registration requires multiple verification steps (email + module steps), each step’s extras are accumulated in a verificationExtrasRef and included in every subsequent resubmit. This ensures prior verifications aren’t lost when a later step resubmits the form.

Basic info fields are rendered with raw TextInput components (not DynamicFormField), as they have hardcoded layout and behavior.

Section 2: Social Links

Dynamic providers fetched via useSocialProviders() hook. Each provider has a key, title, placeholder, and domain prefix. Rendered using the SocialLinksForm component, which shows an icon + label row above each input, with the domain prefix displayed inline.

Section 3: Custom Fields

Rendered from profile.custom_field_groups using DynamicFormField. Only enabled fields are shown. The NativeCustomField format is adapted to FormFieldConfig before passing to DynamicFormField:

const config = {
  label: field.label,
  type: field.type === 'radio' ? 'select' : field.type,
  placeholder: field.placeholder || '',
  instructions: '',
  required: field.is_required,
  options: field.options,
};

Note that radio type fields are converted to select so they render as dropdown modals on mobile instead of inline radio buttons.

Photo Uploads

The ProfilePhotoPicker component renders at the top with a cover photo and overlapping avatar. Both are tappable and show camera overlays with loading indicators during upload. Uploads are handled immediately via showAvatarPicker/showCoverPicker utilities and patchProfileMedia API, not as part of the form save.

Save Behavior

The save payload preserves admin-managed fields (is_verified, badge_slugs, status) to prevent the server from resetting them for moderator accounts.

10. Module-Injected Verification

Add-on modules can inject verification steps into the registration wizard and login flow. Each module provides its own UI component, hook, and API service. The module system's registrationSteps manifest field controls when steps appear.

How It Works

  • Registration steps — modules register steps with a shouldActivate({ submitResponse, registrationConfig }) gate that checks server response and registration config to decide if the step appears
  • Post-login gates — modules use responseHeaders mapping to detect server headers and redirect users (e.g., to a completion screen)
  • Each module owns its own endpoints, hooks, and UI — the core registration system just provides the injection points
  • Pre-creation phase — module steps with phase: 'pre-creation' are checked when registration returns success: false with module-specific flags

See Module System and Registration & Onboarding for full details on the injection mechanism.

11. Select Modal

components/common/SelectModal.tsx provides the dropdown picker for select and gender field types (and radio fields in profile edit, which are converted to select).

Props

PropTypePurpose
visiblebooleanControls modal visibility
titlestringModal header text
optionsstring[]List of selectable options
selectedValuestring | undefinedCurrently selected option (highlighted with checkmark)
onSelect(value: string) => voidCalled when an option is picked
onClose() => voidCalled to dismiss the modal

When DynamicFormField receives a press on a select-type field, it calls onSelectPress(fieldKey). The parent screen opens SelectModal with the field’s options array and handles the selection callback. The modal uses AnimatedPressable for option rows, colors.overlay for the backdrop, and hapticSelection on pick. The selected option is highlighted with a primary-tinted background and checkmark icon.

This keeps the modal state at the screen level (not inside DynamicFormField), avoiding nested modal issues on Android.

Rules

  1. Use DynamicFormField for all server-driven fields — don’t hand-roll inputs for field types it already supports. It handles theming, errors, and accessibility.
  2. Validate on step transition, not on every keystroke — real-time validation causes flickering errors. Run validateStep() when advancing and clear individual errors on change.
  3. Clear errors on field change — when onChange fires, remove that field’s key from fieldErrors immediately so the user sees the error disappear as they correct it.
  4. Merge server errors into fieldErrors — API validation errors arrive as Record<string, string>. Spread them into state so each field shows its server message.
  5. Use cf_ prefix for custom field error keys in profile edit — this prevents collisions with basic field keys like email or first_name.
  6. Checkbox/multiselect values are JSON strings — always JSON.parse() before reading and JSON.stringify() before saving. The server expects this format.
  7. PasswordStrengthMeter is advisory only — never block submission based on score. The server enforces minimum requirements separately.
  8. Keep password scoring in sync with web — the mobile PasswordStrengthMeter.tsx and web registration must use the same algorithm. Update both when changing scoring.
  9. SelectModal lives at screen level, not inside DynamicFormField — nested modals break on Android. The field emits onSelectPress and the screen manages the modal.
  10. Profile edit converts radio to select — FC native custom fields can have radio type, but the edit screen adapts them to select for mobile-friendly dropdown rendering.
