1. How It Works

Thread List (/messages) ├─ Fetches all threads + unread IDs in parallel ├─ Client-side search (name, username, preview) ├─ Swipe-to-delete threads ├─ Pusher updates thread previews in real-time └─ Tap thread → Chat screen Chat Screen (/messages/user/[userId]) ├─ Thread resolution: find existing thread or show "new conversation" ├─ Cursor-based message pagination (load older by scrolling up) ├─ Send text + image attachments + reply quotes ├─ Real-time: Pusher appends new messages instantly ├─ Reactions: emoji picker, optimistic toggle, Pusher sync └─ Block/unblock thread participants

The messaging system is built on Fluent Community’s Chat API (v2.2.0). Threads represent conversations between users. Messages are loaded via cursor-based pagination and updated in real-time via Pusher.

Navigation Into Chat

The chat screen is always navigated to by user ID, not thread ID. This supports both:

  • Existing conversations — the hook resolves the user ID to an existing thread
  • New conversations — when tapping “Message” on a profile, no thread exists yet. The hook shows the user’s info and creates a thread on first message send.
// Navigate to chat with a user
router.push(`/messages/user/${userId}`);

// From thread list (also uses userId from thread info)
const userId = getThreadUserId(thread);
router.push(`/messages/user/${userId}`);

3. Thread Resolution

The useChatMessages hook handles finding the right thread for a given user ID. This is the most complex part of the messaging system.

useChatMessages({ targetUserId, knownThreadId }) ↓ knownThreadId provided? ├─ Yes: getMessages(threadId) directly └─ No: getThreadsForUser(userId) ↓ selected_thread returned? (existing conversation) ├─ Yes: load messages from that thread └─ No: intended_object returned (user info, no thread yet) ↓ Show user info + empty chat Thread created on first message send

Response Fields

FieldMeaning
selected_threadExisting thread with this user — load its messages
intended_objectTarget user info (id, title, photo) — no thread exists yet

5. Cursor-Based Pagination

Messages use cursor-based pagination (not page numbers). The cursor is the id of the oldest loaded message.

// First load — newest messages
const response = await messagesApi.getMessages(threadId);
// response.data.messages = [newest, ..., oldest]
// response.data.has_more = true/false

// Load older messages
const oldestId = messages[0].id;
const older = await messagesApi.getMessages(threadId, oldestId);
// Prepend older.data.messages to existing array

Message Order

The API returns messages in descending order (newest first). The hook reverses them to ascending (oldest first) for chronological display in a bottom-anchored FlashList.

6. Sending Messages

Existing Thread

await messagesApi.sendMessage(threadId, text, attachments, replyData);

New Conversation (No Thread Yet)

// Creates thread + sends first message in a single API call
const response = await messagesApi.startChatWithUser(userId, text);
// response.data.thread — the new thread
// response.data.message — the sent message

Attachments

Image attachments are uploaded separately first (via media upload), then their URLs are passed in the mediaItems array:

await messagesApi.sendMessage(threadId, text, [
  { url: 'https://...uploaded-image.jpg', type: 'image' },
]);

Reply Quotes

To reply to a specific message, pass reply_to (message ID) and reply_text (quoted text):

await messagesApi.sendMessage(threadId, text, undefined, {
  reply_to: originalMessageId,
  reply_text: 'The text being replied to',
});

7. Real-Time (Pusher)

New messages and reactions arrive via Pusher on the user’s private channel (private-chat_user_{id}). See Pusher doc for connection details.

Hooks

HookSourcePurpose
useNewMessageListener(handler)contexts/PusherContext.tsxSubscribe to incoming messages
useReactionListener(handler)contexts/PusherContext.tsxSubscribe to reaction events

Message Events

// In useChatMessages.ts
useNewMessageListener((pusherMessage: PusherMessage) => {
  // Append new message to the list if it's for our thread
  if (pusherMessage.thread_id === thread?.id) {
    setMessages(prev => [...prev, newMessage]);
    // Auto-scroll to bottom
  }
});

useReactionListener((reaction: PusherReaction) => {
  // Update reactions on the target message
  setMessages(prev => prev.map(m =>
    m.id === reaction.message_id
      ? { ...m, meta: { ...m.meta, reactions: reaction.reactions } }
      : m
  ));
});

Thread List Updates

The thread list screen also subscribes to useNewMessageListener. When a new message arrives, it updates the thread’s preview message and moves it to the top of the list.

Badge Counts

Unread message counts are tracked in UnreadCountsContext. The count updates via:

  • Startup batch — initial count
  • Response headers — X-TBC-Unread-Messages on every API call
  • Pusher — bumps count on incoming message event

8. Message Reactions

hooks/useChatReactions.tsx

Chat message reactions use the same emoji set as feed reactions but with a different API endpoint. Reactions are stored in the message’s meta.reactions field.

Data Shape

// meta.reactions is a map of emoji → array of user IDs
{
  "reactions": {
    "👍": [1, 42],     // user IDs who reacted with thumbs up
    "❤️": [1],         // user IDs who reacted with heart
  }
}

Optimistic Updates

When a user taps a reaction, the UI updates immediately (optimistic). If the API call fails, the change is reverted. Pusher also delivers the update, which reconciles the server state.

API

// Toggle reaction (add if not present, remove if present)
await messagesApi.toggleReaction(messageId, '👍');

9. Block & Unblock

Users can block conversations to prevent receiving messages from a specific user.

// Block
await messagesApi.blockThread(threadId);

// Unblock
await messagesApi.unblockThread(threadId);

When a thread is blocked:

  • The chat input is disabled
  • A “You blocked messages” indicator is shown
  • The user can unblock from the chat menu or the blocked state UI

Block state is read from threadDetails.blocked_thread in the messages response.

10. Data Types

types/message.ts

ChatThread

interface ChatThread {
  id: number;
  title: string;
  message_count: string | number;
  status: 'active' | 'inactive' | 'disabled';
  created_at: string;
  updated_at: string;
  info?: ThreadInfo;     // Other participant (v2.2.0)
  type?: 'user' | 'community';
  messages: ChatMessage[]; // Recent messages (preview)
  space_id?: number | null;
}

ChatMessage

interface ChatMessage {
  id: number;
  thread_id: string | number;
  user_id: string | number;
  text: string;              // HTML content
  created_at: string;
  xprofile?: XProfile;       // Sender (on individual messages, not previews)
  meta?: {
    reactions?: Record<string, number[]>;  // emoji → user IDs
    reply_to?: number;                      // ID of quoted message
    reply_text?: string;                    // Text of quoted message
  } | null;
  attachments?: ChatAttachment[];
}

ThreadInfo (v2.2.0)

interface ThreadInfo {
  id: string | number;
  title: string;       // Display name
  photo: string;       // Avatar URL
  username: string;
  is_verified: number | boolean;
  badge: string | null;
}
Message text is HTML. The text field contains HTML like <div class="chat_text"><p>Hello</p></div>. Use getMessageText() or getMessagePreview() from types/message.ts to extract plain text.

12. Rules

  1. Navigate to chat by user ID, not thread ID. The useChatMessages hook handles thread resolution. This supports both existing and new conversations.
  2. Use messagesApi for all chat API calls. Don’t call endpoints directly. The service handles request building and type safety.
  3. Use the helper functions for thread data. Don’t access thread.info.title directly — use getThreadDisplayName(thread). The helpers handle null checks and fallbacks.
  4. Strip HTML from message text before display. Use getMessageText() for full text or getMessagePreview() for truncated previews. Never render raw message.text as plain text.
  5. Use useNewMessageListener and useReactionListener for real-time updates. Don’t subscribe to Pusher channels directly for messaging.
  6. Mark threads read on screen mount. Call messagesApi.markThreadsRead([threadId]) when the chat screen loads so badge counts stay accurate.
  7. Check FEATURES.MESSAGING before showing chat UI. Sites without Fluent Messaging installed should hide all messaging UI.
  8. Reactions use the same meta.reactions shape as feeds — a map of emoji string to array of user IDs. Toggle (not add/remove) is the API contract.
