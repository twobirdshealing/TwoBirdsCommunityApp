1. Overview

Courses are structured learning content managed through Fluent Community's LMS. The app provides:

  • Course browsing — all courses and enrolled courses tabs with search
  • Category filtering — horizontal chip bar to filter by course topic
  • Course detail — sections with collapsible lesson lists, progress tracking
  • Lesson viewer — HTML content, YouTube videos, documents, comments
  • Lock screens — custom lock screens for private/restricted courses with join-request support and enroll fallback
  • Quizzes — full quiz lesson type with single and multiple choice, scoring, pass/fail, retake
  • Media players — YouTube, Vimeo, Wistia, and other oEmbed providers
  • Progress — per-lesson completion with optimistic updates
  • Completion celebration — confetti modal when all lessons are completed
  • Resource links — clickable external links configured in course settings
  • Free preview — non-enrolled users can preview select lessons
  • Drip content — scheduled unlock dates shown on locked sections/lessons
  • Home widget — carousel of enrolled courses on the home screen

Navigation Flow

Courses List (/courses)
  → Course Detail (/courses/[slug])
    → Lesson View (/courses/[slug]/lesson/[lessonSlug])
      → Previous / Next lesson navigation

4. Course List Screen

app/courses/index.tsx — route: /courses

  • Tabs: "All Courses" and "My Courses" (enrolled only) via segmented control
  • Search: Text input with 400ms debounce, clear button when non-empty
  • Category filtering: Horizontal chip bar fetched with with_categories: 1 on the first page. Tapping a chip re-fetches with topic_slug. Tapping the active chip again clears it. "All" chip clears the filter.
  • Search optimization: Uses a ref-based approach to prevent duplicate API calls — fetchCourses reads search from a ref rather than closing over the state, so useCallback identity doesn't change on every keystroke.
  • List: FlashList with infinite scroll pagination (per_page: 15, sort: alphabetical)
  • Pull-to-refresh: Resets pagination and refetches
  • Cards: CourseCard with cover photo, title, description, stats, and progress bar for enrolled courses

5. Course Detail Screen

app/courses/[slug].tsx — route: /courses/[slug]

Layout

  1. Hero section — cover photo (or placeholder book icon with emoji) with gradient overlay showing title and stats (sections, lessons). Stats are computed from the sections array.
  2. Instructor card — uses unified Avatar + UserDisplayName components with verification badge support via course.creator. Tappable to profile. Shows short_description bio (2 lines), total courses, and optionally total students. Conditionally hidden via hide_instructor_view.
  3. Enrollment CTA / Lock Screen — Three states: (a) enrolled: progress bar + "Continue Learning" (or "Review Course" at 100%), (b) not enrolled with lockscreen_config: renders CourseLockScreen, (c) not enrolled without lock config: "Enroll in Course" button.
  4. Course details — "About This Course" section rendering course_details_rendered HTML (stripped to text)
  5. Resource links — "Resources" section with clickable links from course.settings.links. Filters out disabled links (enabled !== 'no'). Opens via Linking.openURL(). Shows link icon + title + open-in-new icon.
  6. Sections list — "Course Content" heading, then collapsible sections with lesson rows. Sections default to expanded when there are 3 or fewer sections.

Continue Learning

The "Continue Learning" button navigates to the first incomplete, viewable, unlocked lesson. If all lessons are complete, it navigates to the first lesson in the course.

6. Lesson View Screen

app/courses/[slug]/lesson/[lessonSlug].tsx — route: /courses/[slug]/lesson/[lessonSlug]

Data Fetching

The lesson screen fetches the full course via getCourseBySlug(slug, lessonSlug) with the intended lesson slug. This returns the course, all sections, and the track. The target lesson is found by matching lessonSlug across all sections. If the lesson's content is empty (lazy-loaded), it fetches individually via getLessonBySlug().

Layout

  1. Nav bar title — shows "Lesson N of M" (e.g., "Lesson 3 of 16")
  2. Hero image — extracted from the first <figure> containing an <img> (or standalone <img>) in the lesson HTML, removed from body to avoid duplication. Rendered at natural aspect ratio. Includes gradient overlay with title and action badges (comments count, documents count).
  3. YouTube video — if lesson.meta.media.url contains a YouTube URL, renders YouTubeEmbed via extractYouTubeId()
  4. oEmbed video — if media URL is not YouTube, falls back to OEmbedPlayer WebView using lesson.meta.media.html
  5. Lesson title — shown only when there is no hero image (hero has title overlay)
  6. Inline CSS — lesson's inline_css prepended as <style> tag to content HTML
  7. Lesson body — remaining HTML rendered via HtmlContent component
  8. Quiz — if content_type === 'quiz' and meta.quiz_questions exists, renders QuizView
  9. Documents — downloadable files from lesson.meta.document_lists with file-type icons (PDF, image, video, audio, archive). Tapping opens via Linking.openURL()
  10. Comments button — if enabled, shows comment count and navigates to /comments/[postId]
  11. Sticky bottom bar — state-specific UI (see below)

Bottom Bar States

StateUI

Enrolled, not complete"Mark as Complete" button

Completion

When a lesson is marked complete and is_completed: true is returned in the response (meaning the entire course is now complete), the app triggers the CompletionCelebration modal after a 1400ms delay (to let the success animation play first).

Lesson Navigation

Previous/Next buttons navigate sequentially through lessons across section boundaries. Sections are flattened into a linear list. Locked or non-viewable lessons cannot be navigated to (buttons disabled). Navigation uses router.replace() to avoid deep back stacks.

Locked Lessons

If a lesson is locked (is_locked: true or can_view: false), the screen shows a lock icon with the lesson's access_message HTML (via HtmlContent) or a fallback "Please enroll in this course to access this lesson" message and a "Back to Course" button.

7. Enrollment

Enrollment Flow

  1. User taps "Enroll in Course" on the course detail screen
  2. API call: enrollInCourse(courseId)
  3. Server returns CourseEnrollResponse with updated CourseTrack
  4. Optimistic update via mutate(): set isEnrolled: true, progress: 0, merge server track
  5. Refetch course detail via refresh() to unlock lesson content
  6. UI switches to progress bar + "Continue Learning" button

Lock Screens

Non-enrolled users of private/restricted courses see a lock screen instead of the course content. This is handled by the CourseLockScreen component (components/course/CourseLockScreen.tsx):

  • Custom HTML lock screen: When config.showCustom is true and config.lockScreen exists, renders the custom HTML via HtmlContent. Shows "Request Access" button if canSendRequest, otherwise "Enroll in Course" button.
  • Default lock card: Lock icon + "This course is private" title. Shows "Request Access" if canSendRequest, otherwise "Enroll in Course" button with context-appropriate description text.
  • Pending state: After a join request is submitted, shows a clock icon + "Request Submitted" + description message.

8. Progress Tracking

Completion Flow

  1. User taps "Mark as Complete" on a lesson
  2. Optimistic update: add lesson ID (as string) to completed_lessons array
  3. API call: toggleLessonCompletion(courseId, lessonId, 'completed')
  4. Server recalculates progress percentage
  5. Response includes updated CourseTrack — replaces local state entirely
  6. Animated success badge ("Completed!") shows for ~800ms, then transitions to nav row with spring animation
  7. If is_completed: true, triggers CompletionCelebration modal after 1400ms delay
Optimistic updates revert on failure. If the API call fails, the lesson is removed from completed_lessons and progress reverts. The server's CourseTrack is always authoritative.

Progress Bar

components/course/ProgressBar.tsx renders a horizontal progress bar:

  • Color: primary until 99%, then success (green) at 100%
  • Track background: withOpacity(primary, 0.15)
  • Clamped to 0–100 range
  • Optional percentage label via showLabel prop (default false)
  • Configurable height (default 6px)

9. Lesson Content Rendering

HTML Content

Lesson content is stored as HTML in the content field and rendered via the HtmlContent component (see Components).

Hero Image Extraction

The first <figure> containing an <img> (or standalone <img>) is extracted from the HTML and rendered as a full-width hero image above the body. The image is rendered at its natural aspect ratio (resolved via RNImage.getSize(), falling back to ~16:9). This prevents the image from appearing twice.

Lazy Loading

The course detail API may return lessons with empty content fields when lazy_load is true. In this case, the lesson view fetches individual lesson content via getLessonBySlug().

Intended lesson slug. When navigating to a lesson via deep link, the course is fetched with intendedLessonSlug query parameter. The server pre-populates that specific lesson's content, avoiding the lazy-load second fetch.

Video & Documents

  • YouTube: If lesson.meta.media.url contains a YouTube URL (detected via extractYouTubeId()), a YouTubeEmbed component renders below the hero
  • oEmbed: If the media URL is not YouTube but lesson.meta.media.html exists, the OEmbedPlayer WebView renders the oEmbed HTML (supports Vimeo, Wistia, and any provider WordPress returns)
  • Documents: lesson.meta.document_lists array rendered as downloadable file cards with file-type icons (PDF, image, video, audio, archive). Tapping opens via Linking.openURL()

Comments

Lesson comments are enabled when lesson.meta.enable_comments !== 'no' AND course.settings.disable_comments !== 'yes'. The comments button navigates to /comments/[postId] using the lesson's ID as the post ID.

10. Quiz System

components/course/QuizView.tsx handles the full quiz-taking experience for lessons with content_type === 'quiz'.

Flow

  1. Mount — fetches existing quiz result via getQuizResult() (for retakes or resuming)
  2. Show form or results — if a previous result exists, shows the results view; otherwise shows the question form
  3. Answer questions — user selects answers for each question
  4. Submit — POST answers to server via submitQuiz(), receives score + pass/fail
  5. Results — displays score circle, pass/fail status, attempt count, and per-question breakdown

Question Types

  • single_choice — radio-style selection, only one answer allowed
  • multiple_choice — checkbox-style selection, toggle for multi-select

Only questions with question.enabled === true are shown to the user. Questions may have label_rendered (HTML) which is rendered via HtmlContent, falling back to plain-text label.

Answer Selection

  • Haptic feedback (light) on each selection
  • Toggle behavior for multi-select questions
  • Validation: all questions must be answered before submit is enabled
  • Haptic feedback (medium) on submit

Results View

  • Score circle showing percentage with color coding (success for pass, error for fail)
  • Pass/fail status indicator
  • Correct answers count (e.g., "5 of 10 correct")
  • Passing score shown when enablePassingScore is true
  • Attempt count from result.meta.attempts (shown when > 1)
  • Per-question breakdown showing correct/incorrect with user's answer (unless hideResult is true)

Retake

The retake button clears all answers and the existing result, allowing the user to re-submit the quiz. Available when the quiz was failed, or when enforcePassingScore is off (retake even after passing).

Server Enforcement

Quiz enforcement is server-side. The Fluent Quiz Pro module's QuizModule.php hooks into is_allowed_to_complete_lesson. If enforce_passing_score is on and the quiz hasn't been passed, the completion API rejects the request. No client-side enforcement is needed — the server is authoritative.

11. Media Players

Lessons can include video from multiple providers. The detection logic determines which player to render:

  1. Check for YouTube ID — if the media URL contains a YouTube video ID (extracted via extractYouTubeId() from utils/youtube.ts), render YouTubeEmbed
  2. Check for oEmbed HTML — if no YouTube ID but lesson.meta.media.html exists, render OEmbedPlayer
  3. No video — if neither condition is met, no video player is rendered

YouTubeEmbed

Existing YouTube player component. Receives a video ID and renders an embedded YouTube player.

OEmbedPlayer

components/media/OEmbedPlayer.tsx — a WebView wrapper that renders lesson.meta.media.html in a responsive container. Provider-agnostic: works with any oEmbed HTML WordPress returns, including Vimeo, Wistia, and other oEmbed providers.

Never render both players. The detection is sequential: YouTube ID first, then oEmbed fallback. Only one player renders per lesson.

13. Category Filtering

The course list screen includes a horizontal chip bar for filtering courses by topic category. Categories are fetched alongside courses by passing with_categories: 1 to the API on the first page only.

  • Tapping a category chip re-fetches courses with the topic_slug parameter
  • Tapping the already-active category chip clears the filter (toggle behavior)
  • The "All" chip clears the filter and shows all courses
  • Chips are displayed in a horizontally scrollable row above the course list
  • Active chip uses primary background with textInverse text; inactive uses backgroundSecondary

14. Free Preview & Drip Content

Free Preview

Lessons with meta.free_preview_lesson === 'yes' are viewable by non-enrolled users. The LessonRow component shows a "Preview" badge with an eye icon for these lessons. Free preview lessons bypass the lock check — even if is_locked is true, they remain tappable.

Drip Content

Sections and lessons can have scheduled unlock dates. When locked with a future date:

  • Locked sections show the unlock date in the section meta text via formatMediumDate()
  • Locked lessons show an "Unlocks {date}" badge with a clock icon
  • Uses section.unlock_date for sections and lesson.unclock_date for lessons (API typo preserved)

15. Resource Links

Courses can include external resource links configured in course.settings.links. These appear on the course detail screen in a "Resources" section between the course description and the sections list.

  • Disabled links (enabled === 'no') are filtered out
  • Each link shows a link icon + title + open-in-new icon
  • Tapping opens the link's permalink via Linking.openURL()
  • Link text and icon use primary color

19. Rules

  1. Use course slug for fetching, course ID for mutations. getCourseBySlug(slug) and getLessonBySlug() use slugs. enrollInCourse(id) and toggleLessonCompletion(id, lessonId) use numeric IDs. Don't mix them up.
  2. Pass intendedLessonSlug for deep link navigation. When navigating to a specific lesson, include this parameter in the course detail fetch. The server pre-populates that lesson's content, avoiding a second fetch.
  3. Handle lazy-loaded lesson content. Course detail may return lessons with empty content fields when lazy_load is true. Always check for empty content and fetch individually via getLessonBySlug() when needed.
  4. Optimistic updates are authoritative until server responds. Add lesson ID to completed_lessons immediately. On API success, replace the entire track with the server response. On failure, revert the optimistic change.
  5. Flatten sections for prev/next navigation. Lessons span multiple sections but prev/next navigation is linear. Flatten all sections' lesson arrays into a single list.
  6. Check both lesson and course settings for comments. Comments are enabled only when lesson.meta.enable_comments !== 'no' AND course.settings.disable_comments !== 'yes'. Either flag can disable them.
  7. Hero images are extracted from HTML, not duplicated. The first <figure> containing an <img> (or standalone <img>) is removed from the HTML body after being rendered as a hero. Don't render the full HTML unmodified or the image appears twice.
  8. Progress bar turns green at 100%. Use primary color for 0–99% and success for 100%. This is the ProgressBar component's built-in behavior — don't override it.
  9. Cache course detail by slug. Use cache key tbc_course_{slug} with useAppQuery. The home widget uses its own cache key (tbc_widget_enrolled_courses) with WIDGET_STALE_TIME.
  10. Lesson IDs in completed_lessons are strings (or numbers). The server returns lesson IDs as strings in the array. Compare with String(lesson.id) when checking completion status.
  11. Quiz enforcement is server-side. The Fluent Quiz Pro module hooks is_allowed_to_complete_lesson. If enforce_passing_score is on and the quiz hasn't been passed, the completion API rejects. Don't add client-side enforcement — let the server be authoritative.
  12. Use permalink for course links, not url. The API sends permalink as the URL field. Filter out disabled links (enabled === 'no').
  13. Use unified Avatar + UserDisplayName for instructor. Don't use raw Image + Text for the instructor row — use the shared Avatar and UserDisplayName components to get verification badges and consistent styling. The instructor data is at course.creator, not course.instructor.
  14. oEmbed fallback for non-YouTube video. Check for YouTube ID first (via extractYouTubeId()), then fall back to OEmbedPlayer if media.html exists. Never render both.
  15. Use router.replace() for lesson-to-lesson navigation. Previous/Next navigation replaces the current route to avoid building up a deep back stack of lesson screens.
