1. Overview

The app has no centralized form library — all form state lives as local component state (useState) in the screen that owns the form. A shared DynamicFormField component renders 13 field types from server-driven configs, while validation runs inline at step transitions or before submission.

Two primary form screens consume this infrastructure:

ScreenFilePurpose
Registration Wizardapp/register.tsxMulti-step signup with dynamic fields, email verify, module-injectable steps
Profile Editapp/profile/edit.tsxEdit name, bio, social links, custom fields with visibility

Both screens follow the same pattern:

const [formData, setFormData] = useState<Record<string, any>>({});
const [fieldErrors, setFieldErrors] = useState<Record<string, string>>({});

2. DynamicFormField Component

components/common/DynamicFormField.tsx is the universal field renderer. It takes a FormFieldConfig and renders the appropriate input type with themed styling, error display, and accessibility.

4. Server-Driven Field Config

Registration fields come from the server, not hardcoded in the app. On mount, register.tsx calls getRegistrationFields() which hits the tbc-ca/v1/auth/register/fields endpoint.

FieldsResponse

export interface FieldsResponse {
  registration_enabled: boolean;
  email_verification_required: boolean;
  [key: string]: unknown;  // Extensible via add-on module hooks
  fields: Record<string, RegistrationField>;
  message?: string;
}

RegistrationField

export interface RegistrationField {
  label: string;
  placeholder?: string;
  instructions?: string;
  type: 'text' | 'email' | 'password' | 'phone' | 'date'
      | 'number' | 'select' | 'radio' | 'gender'
      | 'textarea' | 'url' | 'inline_checkbox'
      | 'checkbox' | 'multiselect';
  input_type?: string;
  required: boolean;
  options?: string[];
  inline_label?: string;
  step: number;  // 1 = basic, 2 = custom fields, etc.
}

Fields are grouped by their step property. Step 1 fields render on the first screen, step 2 on the second, etc. The wizard dynamically adjusts its step count based on what the server returns and which verification features are enabled.

Profile Custom Fields

Profile edit gets its custom field configs from the profile object itself:

export interface CustomFieldConfig {
  label: string;
  type: string;
  placeholder: string;
  instructions: string;
  required: boolean;
  options: string[];
  visibility: string;              // Default visibility level
  allow_user_override: boolean;    // Can user change visibility?

These are available at profile.custom_field_configs and rendered with the same DynamicFormField component.

5. Validation Logic

Validation is not real-time — it runs at specific trigger points:

  • Step transition — validateStep(stepNumber) in register.tsx, called before advancing to next step
  • Form submission — inline validation in handleSave() / handleSubmit() before the API call
  • Server-side — API returns result.errors: Record<string, string> which are merged into fieldErrors

Registration Validation Rules

FieldRuleError Message
All requiredNon-empty after trim{label} is required.
email/^[^\s@]+@[^\s@]+\.[^\s@]+$/Please enter a valid email address.
usernameMin 4 charsUsername must be at least 4 characters.
username/^[a-z0-9_]+$/Username can only contain lowercase letters, numbers, and underscores.

Profile Edit Validation Rules

FieldRuleError Message
first_nameNon-empty after trimFirst name is required

Clear on change: When a user modifies a field, its error is immediately cleared from fieldErrors. This prevents stale error messages from persisting after the user corrects their input.

6. Error Display Pattern

Error display is handled at two levels:

Field-Level Errors

DynamicFormField shows errors when the error prop is set:

  • Red border on the input container (borderColor: themeColors.error)
  • Red error text below the field (styles.fieldError)
{error && (
  <Text style={[styles.fieldError, { color: themeColors.error }]}>
    {error}
  </Text>
)}

Server-Side Errors

When the API returns validation errors, they arrive as Record<string, string> keyed by field name. These are spread into fieldErrors state so each field shows its server-returned message.

// After API call returns errors
if (result.errors) {
  setFieldErrors(result.errors);
}

Error Key Convention

ContextKey PatternExample
Registration fieldsField name directlyemail, username, phone
Profile custom fieldscf_{key} prefixcf_denomination, cf_birthday

7. Registration Form (Multi-Step)

app/register.tsx implements a wizard with up to 5 conditional steps. The step count adapts based on server config — steps are skipped if their feature is disabled.

Step 1: Basic Account Info

Always shown. Fields: full_name, email, username, password, conf_password. If there are no custom fields, the terms checkbox also appears here.

Step 2: Custom Profile Fields

Dynamic fields from the server with step: 2. The terms agreement checkbox (inline_checkbox) is always on step 2 when custom fields exist.

Step 3: Email Verification conditional

Shown only if email_verification_required: true in the fields response. Uses EmailVerifyStep component — a 6-digit code input that the user receives by email.

Steps 4+: Module-Injected Steps conditional

Add-on modules can inject additional steps into the registration wizard via the registrationSteps manifest field. Each step has a shouldActivate() gate that checks the server response to determine if the step should be shown. See Module System for details.

Auth guard note: The register route stays accessible even after authentication so users can complete the profile step (avatar upload) after their account is created. See Registration & Onboarding for the full flow.

10. Phone Input & Country Picker

The phone field type uses the react-native-international-phone-number library.

How It Works

  • Default country: US (can be overridden)
  • Country detection: On mount, calls getCountryByPhoneNumber(currentValue) to detect country from existing value
  • Stored value: Full E.164 format string (e.g., +12025551234)
  • Validation: Server-side only — the app passes the raw value to the API
  • Theme: Follows app dark/light mode via theme={isDark ? 'dark' : 'light'}

Country state is managed locally within DynamicFormField as [phoneCountry, setPhoneCountry].

11. Module-Injected Verification

Add-on modules can inject verification steps into the registration wizard and login flow. Each module provides its own UI component, hook, and API service. The module system's registrationSteps manifest field controls when steps appear.

How It Works

  • Registration steps — modules register steps with a shouldActivate(fieldsResponse) gate that checks server config to decide if the step appears
  • Post-login gates — modules use responseHeaders mapping to detect server headers and redirect users (e.g., to a completion screen)
  • Each module owns its own endpoints, hooks, and UI — the core registration system just provides the injection points

See Module System and Registration & Onboarding for full details on the injection mechanism.

12. Select Modal

components/common/SelectModal.tsx provides the dropdown picker for select, gender, and radio field types.

When DynamicFormField receives a press on a select-type field, it calls onSelectPress(fieldKey, field). The parent screen opens SelectModal with the field’s options array and handles the selection callback.

This keeps the modal state at the screen level (not inside DynamicFormField), avoiding nested modal issues on Android.

Rules

  1. Use DynamicFormField for all server-driven fields — don’t hand-roll inputs for field types it already supports. It handles theming, errors, and accessibility.
  2. Validate on step transition, not on every keystroke — real-time validation causes flickering errors. Run validateStep() when advancing and clear individual errors on change.
  3. Clear errors on field change — when onChange fires, remove that field’s key from fieldErrors immediately so the user sees the error disappear as they correct it.
  4. Merge server errors into fieldErrors — API validation errors arrive as Record<string, string>. Spread them into state so each field shows its server message.
  5. Use cf_ prefix for custom field error keys in profile edit — this prevents collisions with basic field keys like email or first_name.
  6. Checkbox/multiselect values are JSON strings — always JSON.parse() before reading and JSON.stringify() before saving. The server expects this format.
  7. PasswordStrengthMeter is advisory only — never block submission based on score. The server enforces minimum requirements separately.
  8. Keep password scoring in sync with web — the mobile PasswordStrengthMeter.tsx and web registration.js must use the same algorithm. Update both when changing scoring.
  9. SelectModal lives at screen level, not inside DynamicFormField — nested modals break on Android. The field emits onSelectPress and the screen manages the modal.
  10. Phone validation is server-side only — the app sends the raw E.164 string and lets the server (Twilio) validate it. Don’t add client-side phone format validation.
