2. The 5-Step Wizard

app/register.tsx renders a multi-step form. Steps 3–5 are conditional based on server config.

StepComponentPurposeConditional?
1Inline (renderStep1)Name, email, username, passwordAlways shown
2Inline (renderStep2)Custom profile fields + terms checkboxSkipped if no custom fields
3EmailVerifyStep6-digit email verification codeOnly if email_verification_required: true
4PhoneOtpStepSMS/voice OTP codeOnly if otp_required: true
5ProfileCompletionStepsBio, social links, avatar, cover photoOnly if profile_completion.enabled: true

Step 1: Basic Information

FieldValidationNotes
full_nameRequired, non-emptyFrom Fluent Community config
emailRequired, regex: /^[^\s@]+@[^\s@]+\.[^\s@]+$/Server also checks uniqueness
usernameRequired, 4+ chars, /^[a-z0-9_]+$/Lowercase + underscores only
passwordRequired, shown with PasswordStrengthMeter
conf_passwordMust match password (if required by FC)Optional based on FC config

Step 2: Custom Fields + Terms

Renders any custom fields where show_on_signup = true in the WordPress admin. All fields are rendered via the DynamicFormField component. A terms inline checkbox is always present (required).

Step Transitions

After step 1 (or step 2 if custom fields exist), handleSubmitRegistration() sends a POST /register request. The server response determines what happens next:

  • email_verification_required: true → advance to step 3
  • otp_required: true → advance to step 4
  • success: true with tokens → auto-login, then step 5 or main app

3. Registration API

All endpoints in services/api/registration.ts. Registration endpoints are public (no auth required). Profile completion endpoints require JWT auth.

Public Endpoints

FunctionMethodEndpointPurpose
getRegistrationFields()GETtbc-fp/v1/register/fieldsField definitions + server config (OTP, email verify, profile completion)
submitRegistration(data)POSTtbc-fp/v1/registerSubmit registration form. Returns tokens OR verification requirement.
forgotPassword(login)POSTtbc-ca/v1/password/forgotInitiate password reset (sends OTP or email)
resetPassword(token, login, pwd)POSTtbc-ca/v1/password/resetSet new password with reset token

Authenticated Endpoints

FunctionMethodEndpointPurpose
checkProfileComplete()GETtbc-fp/v1/register/statusCheck if profile is complete, return missing fields + existing data
completeRegistration()POSTtbc-fp/v1/register/completeMark profile as complete (sets _tbc_registration_complete = '1')

FieldsResponse

interface FieldsResponse {
  registration_enabled: boolean;
  otp_required: boolean;
  voice_fallback: boolean;
  email_verification_required: boolean;
  profile_completion?: {
    enabled: boolean;
    require_bio: boolean;
    require_avatar: boolean;
  };
  fields: Record<string, RegistrationField>;
}

RegisterResponse

The POST /register response has three possible shapes:

// ✓ Success — user created, tokens returned
{
  success: true,
  access_token: "eyJ...",
  refresh_token: "eyJ...",
  user: { id: 42, username: "jane", display_name: "Jane", email: "jane@..." }
}

// → Email verification needed (re-submit with code + token)
{
  success: false,
  email_verification_required: true,
  verification_token: "signed-jwt-token",
  message: "Please check your email for a verification code."
}

// → Phone OTP needed (verify, then re-submit with session key)
{
  success: false,
  otp_required: true,
  session_key: "abc123",
  phone_masked: "+1 (555) 1**-****",
  message: "..."
}

// ✗ Validation errors
{
  success: false,
  message: "Registration failed.",
  errors: {
    email: "This email is already registered.",
    username: "This username is already taken."
  }
}
Re-submission pattern: After email verification or OTP, the app re-calls submitRegistration() with the original form data plus the verification data (__two_fa_code / __two_fa_signed_token for email, or tbc_fp_session_key for OTP). The server validates the verification and creates the user.

4. OTP Verification (Twilio)

Phone verification via Twilio SMS or voice call. The universal OTP service in services/api/otp.ts handles all OTP contexts (registration, password recovery, profile).

OTP Endpoints

FunctionMethodEndpointPurpose
verifyOtp(sessionKey, code)POSTtbc-fp/v1/otp/verifyVerify the code entered by the user
resendOtp(sessionKey)POSTtbc-fp/v1/otp/resendResend SMS (60-second cooldown)
requestVoiceCall(sessionKey)POSTtbc-fp/v1/otp/voiceRequest voice call instead of SMS (if enabled)

OTP Flow

  1. User submits registration form with phone number
  2. Server initiates Twilio verification → returns session_key + phone_masked
  3. App advances to step 4 (PhoneOtpStep), starts 60-second resend timer
  4. User enters the 4–8 digit code from SMS
  5. App calls verifyOtp(sessionKey, code)
  6. On success → app re-submits registration with tbc_fp_session_key
  7. Server sees verified session → creates user and returns tokens

useOtpVerification Hook

hooks/useOtpVerification.ts — shared state management for OTP across registration and password recovery.

const otp = useOtpVerification({
  onVerified: (sessionKey) => handleSubmitRegistration(sessionKey),
});

// Initialize when server returns OTP data
otp.start({ sessionKey: 'abc123', phoneMasked: '+1 (555) 1**-****', voiceFallback: true });

// State
otp.code           // Current input value
otp.setCode        // Update input
otp.sessionKey     // Server session identifier
otp.phoneMasked    // Masked phone for display
otp.resendTimer    // Seconds until resend is allowed (60 → 0)
otp.voiceFallback  // Whether voice call option is available
otp.error          // Error message to display
otp.verifying      // Loading state during verification

// Actions
otp.handleVerify()    // POST /otp/verify
otp.handleResend()    // POST /otp/resend (blocked if resendTimer > 0)
otp.handleVoiceCall() // POST /otp/voice
otp.reset()           // Clear all state
Session lifecycle: OTP sessions are stored in the WordPress wp_options table and expire after 5 minutes. If the session expires, the user must go back and re-submit the form to get a new session.

5. Email Verification

Email verification uses Fluent Community's built-in 2FA system. When enabled on the site, registration requires a 6-digit email code before proceeding.

Flow

  1. User submits registration form (step 1 + 2)
  2. Server returns email_verification_required: true + verification_token (signed JWT)
  3. App advances to step 3 (EmailVerifyStep)
  4. User checks email, enters 6-digit code
  5. App re-submits registration with __two_fa_code + __two_fa_signed_token
  6. Server validates code → proceeds to OTP or user creation

EmailVerifyStep Component

components/register/EmailVerifyStep.tsx

  • 6-digit code input with number-pad keyboard
  • 60-second resend countdown timer
  • Token passed as prop from parent (stored in verificationToken state)
  • On expired token → shows error, back button returns to previous step

6. Profile Completion

After registration (or on next login if skipped), users may be required to complete their profile with bio, social links, and avatar.

Step 5 During Registration

When profile_completion.enabled: true, the registration wizard shows ProfileCompletionSteps after auto-login.

Login-Time Gate

app/profile-complete.tsx — standalone screen shown to returning users whose profile is incomplete.

  • Fetches existing profile data via checkProfileComplete()
  • Pre-populates form with existing bio, avatar, social links
  • Shows which fields are missing (status.missing array)

ProfileCompletionSteps Component

components/profile/ProfileCompletionSteps.tsx — used by both register.tsx and profile-complete.tsx.

Internal Step 1: Bio + Social

  • bio — textarea, max 500 chars. Required if profile_completion.require_bio: true.
  • website — URL field, optional
  • Social links — configurable providers (Instagram, Twitter, LinkedIn, etc.)

Internal Step 2: Avatar + Cover

  • avatar — image picker (library/camera) + upload. Required if profile_completion.require_avatar: true.
  • cover_photo — optional cover image upload
  • Shows upload progress indicator

Completion API

// Check profile status (authenticated)
const status = await checkProfileComplete();
// → { profile_complete: false, missing: ['avatar', 'bio'], existing: { bio: '', avatar: '', ... } }

// After user finishes the steps:
await completeRegistration();
// → POST /register/complete sets _tbc_registration_complete = '1'
markProfileComplete(); // Updates AuthContext
router.replace('/(tabs)');

X-TBC-Profile-Incomplete Header

Every authenticated API response includes this header (set by class-response-headers.php). The API client intercepts it and updates AuthContext:

// In services/api/client.ts:
const hProfile = response.headers.get('X-TBC-Profile-Incomplete');
if (hProfile !== null) {
  onResponseHeaders({ profileIncomplete: hProfile === '1' });
}

// In AuthContext — the callback:
// profileIncomplete → markProfileIncomplete() → needsProfileCompletion = true
// !profileIncomplete → markProfileComplete()  → needsProfileCompletion = false

7. Password Recovery

Password recovery uses the same OTP infrastructure as registration. Endpoints are on the tbc-ca/v1 prefix (tbc-community-app plugin).

Flow

  1. User enters username or email → forgotPassword(login)
  2. Server returns either otp_sent: true (SMS) or email_sent: true (email link)
  3. If OTP → same PhoneOtpStep UI, same useOtpVerification hook
  4. OTP verified → server returns reset_token + login
  5. User enters new password → resetPassword(token, login, newPassword)

Password Recovery API

FunctionMethodEndpointPurpose
forgotPassword(login)POSTtbc-ca/v1/password/forgotSend OTP or email reset link
verifyPasswordOtp(session, code)POSTtbc-fp/v1/otp/verifyVerify OTP (returns reset_token)
resetPassword(token, login, pwd)POSTtbc-ca/v1/password/resetSet new password
OTP vs Email: The server decides which method to use based on whether the user has a phone number on file. If they do, OTP is sent via SMS. Otherwise, a password reset email is sent.

8. Auth Guard & Route Protection

app/_layout.tsx controls which routes are accessible based on auth state.

Auth Group Routes

const inAuthGroup =
  currentSegment === 'login' ||
  currentSegment === 'register' ||
  currentSegment === 'forgot-password' ||
  currentSegment === 'webview' ||
  currentSegment === 'profile-complete';

Navigation Logic

  1. Unauthenticated + NOT in auth group → redirect to /login
  2. Authenticated + profile incomplete + NOT on gate → redirect to /profile-complete
  3. Authenticated + profile complete + on login page → redirect to /(tabs)
Register stays accessible after auth. The /register route is in the auth group, so it remains accessible even after registerAndLogin() completes. This is intentional — the avatar upload step (step 5) happens after the user is already authenticated.

Auto-Login After Registration

// In register.tsx, after successful registration:
const response = await submitRegistration(formData);

if (response.success && response.access_token) {
  // Store tokens + set isAuthenticated = true
  await registerAndLogin(response.access_token, response.refresh_token!, response.user!);

  // Then advance to step 5 (profile completion) if enabled
  if (fieldsConfig?.profile_completion?.enabled) {
    setStep(5);
  } else {
    router.replace('/(tabs)');
  }
}

9. Custom Profile Fields

Custom fields are defined in the WordPress admin (tbc-fluent-profiles settings) and returned dynamically by GET /register/fields.

RegistrationField Type

interface RegistrationField {
  label: string;
  placeholder?: string;
  instructions?: string;
  type: 'text' | 'email' | 'password' | 'phone' | 'date' | 'number'
      | 'select' | 'radio' | 'gender' | 'textarea' | 'url'
      | 'inline_checkbox' | 'checkbox' | 'multiselect';
  input_type?: string;
  required: boolean;
  options?: string[];      // For select/radio/checkbox/multiselect
  inline_label?: string;   // For inline_checkbox (e.g., terms text)
  step: 1 | 2;            // Which wizard step to show this field on
}

Field Rendering

All custom fields are rendered via DynamicFormField (components/common/DynamicFormField.tsx). This component maps the type to the appropriate input: text input, date picker, select modal, radio group, checkbox, textarea, etc.

Field Storage (WordPress)

  • Definitions stored in wp_options key tbc_fp_fields
  • User values stored in wp_usermeta with key prefix _tbc_fp_{field_key}

13. Rules

  1. All registration behavior is server-driven. The app reads GET /register/fields on mount and adapts the wizard steps. Don't hardcode which steps to show — let the server config decide.
  2. Use useOtpVerification for all OTP flows. Don't implement OTP state inline. The hook handles countdown timers, deduplication, and error clearing.
  3. Re-submit the full form after verification. Email verification and OTP verification are not standalone — the app re-calls submitRegistration() with the original form data plus verification tokens. The server creates the user on this second call.
  4. Keep the register route in the auth group. The avatar upload step (step 5) happens after registerAndLogin() — the user is already authenticated but still on the register screen.
  5. Default checkProfileComplete to true on error. If the profile status check fails, assume the profile is complete. Never block users from the app because of a transient API error.
  6. ProfileCompletionSteps is shared. The same component is used by both the registration wizard (step 5) and the login-time gate screen (profile-complete.tsx). Don't duplicate this UI.
  7. Registration endpoints are public. The tbcPublicRequest helper uses raw fetch(), not the authenticated API client. Don't add auth headers to registration calls.
  8. OTP sessions expire after 5 minutes. If the session expires during verification, the user must go back and re-submit the form to get a new session. Don't cache or reuse expired sessions.
  9. Field errors are keyed by field name. Server validation errors come back as errors: { fieldKey: "message" }. Display them inline next to the corresponding field, not as a generic banner.
  10. Custom fields use DynamicFormField. All field types (text, select, radio, date, checkbox, textarea, etc.) are rendered by the same component. Don't create separate field components for registration.
