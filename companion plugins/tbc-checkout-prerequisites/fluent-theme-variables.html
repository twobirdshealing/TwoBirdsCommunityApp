Fluent Community Theming Guide

How to make any plugin match FluentCommunity's light/dark theme. Covers PHP setup, CSS variables, and patterns.

Step 1: Get the Variables on Your Page

The --fcom-* variables are injected as inline CSS by Fluent Community's fluent_community/enqueue_global_assets action. This loads global.css, theme-default.css, the color scheme CSS (light + dark), and the dark mode detection script.

Key insight: You don't need to hardcode dark mode colors. Fire this one action and Fluent handles light/dark switching automatically via the customizer. If the user picks a blue dark theme, your plugin matches. No extra CSS needed.

PHP: Enqueue in your plugin

// In your wp_enqueue_scripts callback, before your own styles: public function maybe_enqueue_assets() { // Load Fluent Community global styles + color variables (light/dark) if (did_action('fluent_community/enqueue_global_assets') === 0) { do_action('fluent_community/enqueue_global_assets', true); } // Now enqueue your plugin CSS - variables are available wp_enqueue_style('my-plugin', MY_PLUGIN_URL . 'css/my-plugin.css', [], MY_VERSION); }

What this does

  1. Enqueues global.css and theme-default.css from Fluent Community
  2. Calls Utility::getColorCssVariables() which generates inline CSS setting all --fcom-* variables on body (light) and html.dark body (dark)
  3. Enqueues portal_general.js which handles the dark mode html.dark class toggle
Guard with did_action() - If the page already uses the Fluent Community template (like portal pages), the action has already fired. The guard prevents double-loading.

Step 2: Understand How Dark Mode Works

The flow:

1. Fluent's customizer lets users pick light + dark color schemes

2. Utility::getColorCssVariables() generates CSS for both:
   body { --fcom-primary-bg: #FFFFFF; --fcom-primary-text: #19283a; ... }
   html.dark body { --fcom-primary-bg: #1d1e1f; --fcom-primary-text: #E5EAF3; ... }

3. portal_general.js checks localStorage('fcom_global_storage') and adds/removes html.dark

4. Your CSS uses var(--fcom-primary-bg, #fff) — the variable handles both modes automatically

What you DON'T need to do

Anti-patternWhy it's wrong
Writing html.dark .my-thing { background: #1d1e1f; } Hardcoded dark values won't sync if user picks a different dark scheme in Fluent customizer
Duplicating every rule with html.dark overrides The variables already flip automatically — one rule handles both modes
Using @media (prefers-color-scheme: dark) Fluent uses html.dark class, not the OS media query

What you DO need for dark mode (minimal)

/* 1. Native form elements need color-scheme for browser rendering */ html.dark .my-wrapper select, html.dark .my-wrapper input { color-scheme: dark; } /* 2. Semantic colors with hardcoded dark text (e.g. #856404) may need lightening since they sit on dark backgrounds */ html.dark .my-warning-banner { color: #ffc107; /* lighten the dark-amber #856404 */ }

Step 3: Use the Variables

Use var(--fcom-*, fallback) everywhere instead of hardcoded colors. The fallback only applies if Fluent Community isn't active at all.

Complete Example: Themed Plugin Component

PHP (enqueue)

public function maybe_enqueue_assets() { global $post; if (!$post || !has_shortcode($post->post_content, 'my_plugin_shortcode')) { return; } // Get Fluent's theme variables onto the page if (did_action('fluent_community/enqueue_global_assets') === 0) { do_action('fluent_community/enqueue_global_assets', true); } wp_enqueue_style('my-plugin', MY_PLUGIN_URL . 'css/my-plugin.css', [], MY_VERSION); wp_enqueue_script('my-plugin', MY_PLUGIN_URL . 'js/my-plugin.js', ['jquery'], MY_VERSION, true); }

CSS (my-plugin.css)

/* Everything uses variables - ONE set of rules for light AND dark */ .my-wrap { color: var(--fcom-primary-text, #19283a); font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif; } .my-card { background: var(--fcom-primary-bg, #fff); border: 1px solid var(--fcom-primary-border, #e3e8ee); border-radius: 8px; padding: 20px; } .my-card:hover { background: var(--fcom-active-bg, #f0f3f5); } .my-heading { color: var(--fcom-primary-text, #19283a); } .my-meta { color: var(--fcom-secondary-text, #525866); } .my-link { color: var(--fcom-text-link, #2271b1); } .my-btn { background: var(--fcom-primary-button, #2B2E33); color: var(--fcom-primary-button-text, #fff); border: 1px solid var(--fcom-primary-button, #2B2E33); border-radius: 6px; padding: 8px 16px; cursor: pointer; } .my-input { background-color: var(--fcom-primary-bg, #fff); border: 1px solid var(--fcom-primary-border, #e3e8ee); border-radius: 6px; color: inherit; padding: 6px 10px; } .my-table { background: var(--fcom-primary-bg, #fff); border: 1px solid var(--fcom-primary-border, #e3e8ee); } .my-table thead { background: var(--fcom-secondary-bg, #f0f2f5); } .my-table th { color: var(--fcom-secondary-text, #525866); border-bottom: 2px solid var(--fcom-primary-border, #e3e8ee); } .my-table td { border-bottom: 1px solid var(--fcom-primary-border, #e3e8ee); } /* Status badges: use rgba so they work on light AND dark */ .my-badge-success { background: rgba(40, 167, 69, 0.1); color: #28a745; border: 1px solid rgba(40, 167, 69, 0.3); } .my-badge-danger { background: rgba(220, 53, 69, 0.1); color: #dc3545; border: 1px solid rgba(220, 53, 69, 0.3); } .my-badge-warning { background: rgba(255, 193, 7, 0.1); color: #856404; border: 1px solid rgba(255, 193, 7, 0.3); } /* ONLY dark-mode-specific overrides needed: */ /* 1. Native form elements */ html.dark .my-wrap select, html.dark .my-wrap input { color-scheme: dark; } /* 2. Hardcoded dark text on semantic banners */ html.dark .my-badge-warning { color: #ffc107; }

Source: Where Variables Come From

FileWhat it does
app/Functions/Utility.php getColorCssVariables() — reads customizer config, generates inline CSS for both light (body) and dark (html.dark body) selectors
app/Functions/Utility.php getColorSchemas() — returns all available light/dark skin presets with their variable values
app/Hooks/Handlers/PortalHandler.php Registers the fluent_community/enqueue_global_assets action that loads CSS + injects variables
Modules/Theming/TemplateLoader.php Fires fluent_community/enqueue_global_assets when page uses Fluent template. Also injects dark mode detection script from localStorage
assets/global.css Core layout styles that consume --fcom-* variables
assets/theme-default.css Default theme skin styles

How the CSS gets generated

// Utility::getColorCssVariables() output example: // Light: body { --fcom-primary-bg: #FFFFFF; --fcom-primary-text: #19283a; --fcom-secondary-text: #525866; --fcom-primary-border: #e3e8ee; --fcom-primary-button: #2B2E33; --fcom-text-link: #2271b1; /* ... etc */ } // Dark (auto-generated from customizer dark scheme): html.dark body { --fcom-primary-bg: #1d1e1f; --fcom-primary-text: #E5EAF3; --fcom-secondary-text: #A3A6AD; --fcom-primary-border: #414243; --fcom-primary-button: #3a3d42; --fcom-text-link: #6db3f2; /* ... etc - values depend on chosen dark scheme */ }

Theming Checklist for New Plugins

#TaskDone?
1Add do_action('fluent_community/enqueue_global_assets', true) in enqueue callback (with did_action guard)
2Replace all hardcoded background colors with --fcom-primary-bg / --fcom-secondary-bg / --fcom-active-bg
3Replace all hardcoded text colors with --fcom-primary-text / --fcom-secondary-text / --fcom-text-off / --fcom-text-link
4Replace all hardcoded border colors with --fcom-primary-border / --fcom-secondary-border
5Replace button colors with --fcom-primary-button / --fcom-primary-button-text
6Use rgba() for semantic status indicators (success/danger/warning) so they work on any background
7Add html.dark .my-wrap select, input { color-scheme: dark; } for native form elements
8Check any hardcoded dark text colors in banners/badges — lighten under html.dark if needed
9Use color: inherit on inputs/selects instead of explicit text color (inherits from parent)
10Test in Fluent customizer with different light + dark schemes to confirm sync
