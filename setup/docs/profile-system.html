1. Data Types

All profile types in types/user.ts.

XProfile (Lightweight)

Embedded in feeds, comments, messages — anywhere a user reference appears.

interface XProfile {
  user_id: number;
  username: string;
  display_name: string;
  avatar: string | null;
  short_description: string | null;
  total_points: number;
  is_verified: number;           // 0 or 1
  status: 'active' | 'suspended' | 'banned';
  created_at: string;
  meta?: {
    website?: string;
    cover_photo?: string;
    bio?: string;
    social_links?: Record<string, string>;
    badge_slug?: string[];       // Badge slugs for this user
  };
  badge?: Badge | null;          // Primary badge if assigned
  is_follower?: boolean;         // Mutual follow flag (FC Pro 2.3.0+)
}

2. Profile API

All endpoints in services/api/profiles.ts. Base path: /fluent-community/v2/profile/{username}.

patchProfileMedia(username, data)PUT/profile/{username}Upload avatar and/or cover photo (wraps in { data } envelope)
getUserFeeds(userId, page, perPage)GET/feeds?user_id={id}User's posts (paginated)

Both update and media endpoints wrap data in an envelope. updateProfile sends { data: { first_name, ... } } and patchProfileMedia sends { data: { avatar, cover_photo } } — the fields are nested inside a data key, matching the Fluent Community API format.

Profile View Screen

The profile view screen handles both own and other users' profiles:

  • Own profile: Shows settings gear icon. 3-dot menu opens ProfileMenu with Edit Profile, Deactivate, Delete Account options.
  • Other profile: Shows Follow/Unfollow, Notify toggle, Chat button, and a separate dropdown with Block option.
  • Restricted profile: Only About tab visible, shows "Profile is Private" lock icon message on About tab.
  • Tabs: About (always), Posts, Spaces, Comments (configurable via feature flags + server navs)
  • Tab data is lazy-loaded on first visit and paginated on scroll
  • Profile data uses useAppQuery with CACHE_EVENTS.PROFILE invalidation

5. Badge System

Badges are assigned by admins and displayed next to user names throughout the app.

Badge Definitions API

services/api/badges.ts — public endpoint, no auth required.

// GET tbc-ca/v1/badge-definitions
// Returns: { success: true, badges: { [slug]: { title, color, background_color, show_label, config } } }

Two-Layer Cache

  1. MMKV — persists across app restarts, loaded instantly on launch
  2. In-memory — module-level variable, background-refreshed from API

Self-registers with cacheRegistry so logout clears the in-memory cache.

Badge Hooks

// Load definitions + get resolver function
const { resolveBadges } = useBadgeDefinitions();
const badges: Badge[] = resolveBadges(['moderator', 'founding_member']);

// Convenience: resolve directly from slugs
const badges = useProfileBadges(xprofile.meta?.badge_slug);

Both hooks share the same cache. First call triggers MMKV load + API refresh. All subsequent calls use the cached result. Listeners are notified when the cache updates.

Badge Display

  • Pill mode (show_label='yes' or default): Colored background + icon/emoji + label text
  • Icon-only mode (show_label='no'): No background, larger icon, no label
  • Supports custom logo images or emoji fallback

Verified Badge

Separate from profile badges. is_verified: 1 shows a checkmark icon inline with the display name. Set by admins, not editable by users.

6. Connections & Following

Follow Actions

// Follow a user (level 0 → 1)
await profilesApi.followUser('janedoe');

// Unfollow (level 1/2 → 0)
await profilesApi.unfollowUser('janedoe');

// Toggle email notifications (level 1 ↔ 2)
await profilesApi.toggleFollowNotification('janedoe');

useFollowToggle Hook

hooks/useFollowToggle.ts — shared across directory, members list, and connections screen.

const {
  followMap,             // Record<userId, followLevel>
  setFollowMap,          // Initialize from server data
  followLoadingMap,      // Record<userId, boolean> — raw loading state map
  handleFollowPress,     // Toggle follow with optimistic update (takes MemberCardData)
  handleNotifyPress,     // Toggle notification level 1 ↔ 2 (takes MemberCardData)
  isFollowing,           // (userId) → boolean
  isNotifyOn,            // (userId) → boolean (level === 2)
  isFollowLoading,       // (userId) → boolean
} = useFollowToggle();
Note: handleFollowPress and handleNotifyPress accept a MemberCardData object (not a username string). They extract user_id and username from member.xprofile or the member object directly.

Connections Screen

app/profile/[username]/connections.tsx — two tabs (Followers / Following).

  • Search bar — debounced (400ms) server-side filtering by name or username
  • Sort toggle — switch between Recent (default) and A-Z (alphabetical by display name)
  • "Follows you" badge — shown on member cards when is_follower is true (mutual follow indicator)
  • Infinite scroll with pagination
  • Each item rendered as MemberCard with follow/notify/message actions
  • Pull-to-refresh on both tabs
  • Lazy-loads each tab on first visit — search/sort state is shared across tabs
  • Followers response nests profile in data[].follower, following in data[].followed

Save Flow

// Single POST saves all field changes
await updateProfile(username, {
  first_name: 'Jane',
  last_name: 'Doe',
  short_description: 'Community member',
  website: 'https://example.com',
  social_links: { twitter: 'janedoe' },
  custom_fields: { phone: '+15551234567' },
  // Conditional fields
  email: formData.email,        // only if can_change_email
  username: formData.username,  // only if can_change_username
  // Preserve admin-managed fields
  is_verified: profile.is_verified,
  badge_slugs: profile.badge_slugs,
  status: profile.status,
});
Always preserve admin fields on save. Include is_verified, badge_slugs, and status in the update payload with their current values. Otherwise, the Fluent Community API may clear them.

Post-Save

After a successful save, the edit screen updates the AuthContext user via updateUser({ displayName }) and emits CACHE_EVENTS.PROFILE to invalidate cached profile data across the app.

8. Custom Profile Fields

Custom fields use Fluent Community's native custom_field_groups format (FC 2.3.0+). The profile response includes grouped fields with their current values.

NativeCustomField

interface NativeCustomField {
  slug: string;
  label: string;
  type: 'text' | 'textarea' | 'number' | 'date' | 'select' | 'radio' | 'url' | 'multiselect';
  placeholder: string;
  options: string[];            // For select/radio/multiselect
  is_required: boolean;
  is_enabled: boolean;
  privacy: 'public' | 'logged_in' | 'private' | 'internal';
  group: string;
  value?: string | string[];    // Current value (string[] for multiselect)
}

CustomFieldGroup

interface CustomFieldGroup {
  slug: string;
  label: string;
  edit_description?: string;
  is_system?: boolean;
  fields: NativeCustomField[];
}

Legacy Types (Backwards Compatibility)

The legacy CustomFieldValue and CustomFieldConfig types still exist in types/user.ts for backwards compatibility with older API responses, but the active UI code uses the native format above.

Display Rendering (AboutTab)

The AboutTab component iterates through custom_field_groups, flattening all enabled fields with non-empty values. Rendering is type-specific:

  • URL: Clickable links (auto-prepends https:// if missing), displayed without protocol prefix
  • Date: Formatted to locale string ("Month Day, Year")
  • Multiselect: Rendered as tag pills in a wrapping row with a block layout (label above tags)
  • Phone-like: Detected by slug containing "phone" — rendered as tappable tel: links
  • Other: Plain text

Edit Form Rendering

The edit screen flattens custom_field_groups into a slug-keyed map and renders each enabled field via DynamicFormField. The NativeCustomField is adapted to DynamicFormField's config shape (e.g., radio type maps to select for mobile). Select/radio fields use SelectModal for option picking.

9. Profile Tabs

Profile tabs are controlled by both server-controlled feature flags and server-side navigation config.

Tab Visibility Logic

// A tab is shown if BOTH conditions are met:
// 1. Feature flag is enabled: features.profile_tabs.posts = true (via useFeatures())
// 2. Server nav includes the slug: profile_navs contains 'user_profile_feeds'

// If server returns no profile_navs, feature flag alone determines visibility
TabFeature FlagServer Nav SlugDefault
About(always shown)—Always
Postsfeatures.profile_tabs.postsuser_profile_feedsfalse
Spacesfeatures.profile_tabs.spacesuser_spacesfalse
Commentsfeatures.profile_tabs.commentsuser_commentsfalse

When is_restricted: true, only the About tab is shown regardless of flags.

Tab Data Loading

  • Tabs are lazy-loaded — data fetched only when the tab is first selected
  • Posts and Comments support pagination (load more on scroll)
  • Spaces are loaded all at once (no pagination)
  • All tabs support pull-to-refresh
  • Posts tab reuses FeedList with full feed actions (react, bookmark, edit, delete)
  • Comments tab uses FlashList with CommentCard (supports delete for own comments)
  • Spaces tab uses FlashList with SpaceCard

11. Privacy & Blocking

Profile Privacy

is_restricted: true (Fluent Community 2.2.01+) limits what non-connected users can see:

  • Only the About tab is visible
  • Posts, Spaces, and Comments tabs are hidden
  • A "Profile is Private" message is shown with a lock icon on the About tab

Blocking

// Block a user
await profilesApi.blockUser('spammer');
// → Their posts are hidden from your feed, DMs are rejected

// Unblock
await profilesApi.unblockUser('spammer');
  • Block/Unblock option appears in a separate dropdown menu on other users' profiles (gear icon)
  • Confirmation alert before blocking with descriptive message
  • is_blocked_by_you: true on the profile response
  • When blocked, the Follow/Notify/Chat buttons are replaced with a single "Blocked" button
  • Cannot block yourself (own profile shows ProfileMenu instead)

Account Management

From the own-profile menu (ProfileMenu), accessible via the gear icon:

  • Edit Profile: Opens the profile edit screen
  • Deactivate Account: Soft delete with alert confirmation, reversible via web admin. Calls POST /tbc-ca/v1/account/deactivate. On success, logs out and redirects to login.
  • Delete Account: Two-step confirmation — first an alert, then a modal where user must type "DELETE" in a text input before the button becomes active. Calls DELETE /tbc-ca/v1/account/delete with { confirm: 'DELETE' }. Permanent. On success, logs out and redirects to login.

13. Rules

  1. Always use UserDisplayName for rendering user names. It handles the verified badge and profile badges automatically. Don't manually render name + badge combinations.
  2. Always use Avatar for user images. It handles null avatars (fallback initials), sizing, caching, and online indicators. Don't render avatar images manually.
  3. Preserve admin fields on profile update. Include is_verified, badge_slugs, and status in the payload with their current values. The FC API may clear them otherwise.
  4. Use useFollowToggle for follow actions in list contexts. It handles optimistic updates, loading state, and notification toggling. The profile view screen manages follow state locally via useAppQuery + mutate instead.
  5. Use useProfileBadges or useBadgeDefinitions for badge resolution. Don't fetch badge definitions yourself — the hooks share a two-layer cache (MMKV + in-memory).
  6. Check both feature flag AND server navs for profile tabs. A tab should only be shown if features.profile_tabs.x is true (via useFeatures()) AND the server's profile_navs includes the corresponding slug (or profile_navs is not provided).
  7. Handle is_restricted profiles. When true, only show the About tab. Don't fetch posts/comments/spaces for restricted profiles.
  8. Social links location differs between XProfile and Profile. On XProfile, social links are in meta.social_links. On the full Profile endpoint, they're at the top level: profile.social_links. Same for badge_slugs vs meta.badge_slug.
  9. Navigate to profiles by username. Always use router.push('/profile/{username}') — never by user ID. The API uses username as the path parameter.
  10. Custom fields are server-defined. Don't hardcode field keys or types. Render all fields dynamically based on custom_field_groups. Use DynamicFormField for edit forms.
  11. Emit CACHE_EVENTS.PROFILE after profile mutations. Both the edit screen and avatar/cover uploads emit this event so the profile view screen refreshes its cached data.
